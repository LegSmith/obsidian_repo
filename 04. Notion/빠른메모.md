### 🎙️ ASR / RNN-T / End-to-End 모델 용어 정리

## 📌 ASR (Automatic Speech Recognition) — 자동 음성 인식

- **음성(오디오) → 텍스트**로 변환하는 기술 전체를 가리키는 포괄적 용어.
- STT(Speech-to-Text)와 동의어로 혼용되며, 학술/산업 양쪽에서 모두 사용됨.
- 전통적 ASR은 크게 **음향 모델(AM) + 언어 모델(LM) + 발음 사전** 3요소로 구성.

---

## 📌 전통적 ASR 파이프라인 vs End-to-End

|구분|전통 파이프라인|End-to-End(E2E)|
|---|---|---|
|구조|AM + 발음사전 + LM 별도 학습|단일 모델이 오디오 → 텍스트 직결|
|예시|HMM-GMM, HMM-DNN|RNN-T, CTC, Whisper, Conformer|
|장점|각 모듈 교체/튜닝 용이|학습 단순, 레이턴시 낮음, 스트리밍 가능|
|단점|구성 복잡, 오류 전파|데이터 대량 필요, 해석 어려움|

---

## 📌 End-to-End (E2E) 모델

- 전처리(특징 추출)부터 최종 텍스트 출력까지 **하나의 신경망**으로 처리.
- 중간에 HMM, 발음사전 같은 별도 모듈 없이 학습 데이터만으로 훈련 가능.
- 대표 아키텍처: **CTC, Attention Encoder-Decoder, RNN-T**

---

## 📌 RNN-T (Recurrent Neural Network Transducer)

**핵심 한 줄 요약**: 오디오를 실시간으로 처리하면서(스트리밍) 텍스트를 출력할 수 있는 End-to-End ASR 아키텍처

### 구성 요소 3가지

1. **Encoder** (= 음향 인코더)
    - 입력: 음성 프레임 시퀀스 (mel-spectrogram 등)
    - 역할: 음성의 고수준 음향 표현 추출
    - 구현체: RNN, LSTM, Conformer, Transformer 등
2. **Prediction Network** (= 언어 디코더, LM 역할)
    - 입력: 이전에 출력한 토큰(텍스트) 시퀀스
    - 역할: 다음에 나올 글자/단어를 예측 (언어 모델 역할)
    - 구현체: RNN, LSTM, Embedding Layer
3. **Joint Network**
    - Encoder 출력 + Prediction Network 출력을 **결합**하여 최종 토큰 확률 분포 계산
    - 이 네트워크가 '언제 어떤 글자를 출력할지' 결정

### 왜 RNN-T가 스트리밍에 유리한가?

- CTC나 Attention Encoder-Decoder는 **전체 오디오를 다 본 후** 디코딩 → 레이턴시 높음
- RNN-T는 오디오 프레임이 들어올 때마다 **즉시 부분 출력 가능** → 실시간 자막, 전화 STT에 적합
- Amazon Transcribe, Google STT 등 상용 스트리밍 ASR 대부분이 RNN-T 기반

---

## 📌 CTC (Connectionist Temporal Classification)

- E2E ASR의 또 다른 대표 방식. Encoder 출력에서 직접 CTC 손실로 학습.
- **blank 토큰**을 이용해 입력-출력 길이 불일치 문제를 해결.
- RNN-T보다 단순하지만 스트리밍 품질은 떨어짐. Whisper의 학습에 CTC 변형 사용.

---

## 📌 Whisper (OpenAI) — Faster Whisper와의 관계

- **Whisper**: OpenAI가 공개한 E2E ASR 모델. Transformer Encoder-Decoder 기반. **Offline(비스트리밍)** 방식.
- **Faster Whisper**: Whisper를 `CTranslate2`로 최적화한 추론 라이브러리. 속도 2~4배↑, 메모리↓
- 내부 아키텍처는 Whisper와 동일하고 **가중치 호환** — 추론 엔진만 교체한 것.

---

## 📌 주요 용어 한눈에 보기

|용어|한 줄 설명|
|---|---|
|ASR|음성 → 텍스트 변환 기술 전체 (= STT)|

|RNN-T|스트리밍 가능한 E2E ASR 대표 아키텍처|
|CTC|blank 토큰 기반의 E2E 학습 목적함수/구조|

### n8n Docker 컨테이너 실행 시 권한 문제 정리

## 1. 발생한 문제점

1. **포트 매핑 오류**
    - n8n은 컨테이너 내부 포트를 항상 **5678**에서 서비스하지만, `p 15677:5677` 과 같이 잘못 매핑하면 브라우저에서 접속이 불가능함.
    - 올바른 방식은 `p [외부포트]:5678`.
2. **볼륨 권한 문제**
    - 커스텀 노드 저장소(`/home/node/.n8n/nodes`)를 마운트한 후 로그에서 다음과 같은 에러 발생:

        ```
        Error: EACCES: permission denied, open '/home/node/.n8n/nodes/package.json'
        ```

    - 이는 Docker 볼륨(`n8n_custom_nodes_jy`)이 root 권한으로 생성되어, 컨테이너 내부의 `node` 사용자(UID 1000)가 파일을 쓸 수 없어서 생긴 문제.
3. **설정 파일 권한 경고**
    - `Permissions 0644 for n8n settings file /home/node/.n8n/config are too wide` 메시지 발생.
    - 이는 n8n이 설정 파일의 권한을 엄격하게 관리하려고 하기 때문이며, 환경 변수로 제어 가능.

---

## 2. 해결 방법

### (1) 포트 매핑 수정

컨테이너 내부 포트는 반드시 5678로 맞춰야 함.

```bash
-p 15677:5678
```

---

### (2) 컨테이너를 `node` 사용자로 실행

권한 충돌을 피하기 위해 `--user 1000:1000` 옵션 추가.

```bash
--user 1000:1000
```

### (3) Docker 볼륨 권한 수정

이미 생성된 볼륨은 root 권한으로 잡혀 있으므로, 호스트에서 소유자를 변경해야 함.

볼륨 경로 확인:

```bash
docker volume inspect n8n_custom_nodes_jy
```

출력 예:

```
"Mountpoint": "/var/lib/docker/volumes/n8n_custom_nodes_jy/_data"
```

권한 수정:

```bash
sudo chown -R 1000:1000 /var/lib/docker/volumes/n8n_custom_nodes_jy/_data
```

정상 확인:

```bash
ls -ld /var/lib/docker/volumes/n8n_custom_nodes_jy/_data
```

→ 결과에 `1000 1000` 소유자가 보이면 성공.

### (4) 컨테이너 실행 전체 명령어

```bash
docker run -d \                                        # 컨테이너를 백그라운드(detached) 모드로 실행
  --name n8n_jy \                                      # 컨테이너 이름을 'n8n_jy'로 지정
  -p 15677:5678 \                                      # 호스트 포트 15677 → 컨테이너 내부 포트 5678 (n8n 기본 포트)
  --user 1000:1000 \                                   # 컨테이너를 UID 1000 (node 유저) 권한으로 실행하여 권한 충돌 방지
  -e GENERIC_TIMEZONE=Asia/Seoul \                     # 타임존을 한국 표준시(KST)로 설정
  -e N8N_SECURE_COOKIE=false \                         # HTTPS 보안 쿠키 사용 여부 (테스트 환경에서는 false로)
  -e N8N_ENFORCE_SETTINGS_FILE_PERMISSIONS=true \      # 설정 파일 권한을 강제로 안전하게 적용
  -e DB_SQLITE_POOL_SIZE=2 \                           # SQLite 연결 풀 크기 설정 (동시 접속 제어)
  -e N8N_RUNNERS_ENABLED=true \                        # n8n Task Runners 기능 활성화 (향후 기본값이 될 예정)
  -e N8N_EDITOR_BASE_URL=http://192.168.50.80:15677 \  # 브라우저에서 접속할 n8n 에디터 주소
  -e WEBHOOK_URL=http://192.168.50.80:15677/ \         # 외부 서비스에서 호출할 n8n 웹훅 주소
  -v n8n_data_jy:/home/node/.n8n \                     # n8n 데이터(워크플로우, 자격증명 등) 저장 볼륨
  -v n8n_custom_nodes_jy:/home/node/.n8n/nodes \       # 커스텀 노드 저장소 볼륨
  --restart unless-stopped \                           # 컨테이너가 꺼지면 자동 재시작, 단 수동으로 stop하면 재시작 안 함
  n8nio/n8n:latest                                     # 실행할 n8n 공식 이미지 (최신 버전)
```
