`SpringBootWebSecurityConfiguration` 에서 자동 설정에 의한 기본 보안 설정 클래스가 생성된다.

- 스프링 ==_**서버가 가동되면 스프링 시큐리티의 초기화 작업 및 보안 설정이 이루어진다.**_==
- ==별도의 설정이나 코드를 작성하지 않아도 기본적인 웹 보안 기능이 현재 시스템에 연동되어 작동==한다. → `Spring Boot` 의 기능
    1. 기본적으로 ==**모든 요청에 대하여 인증여부를 검증**==하고 _**인증이 승인되어야 자원에 접근이 가능**_
    2. 인증 방식은 `Form 로그인 방식`과 `HttpBasic 로그인 방식`을 제공한다.
    3. 인증을 시도할 수 있는 로그인 페이지가 자동적으로 생성되어 랜더링 된다.
    4. 인증 승인이 이루어질 수 있도록 한개의 계정이 기본적으로 제공
        - `SecurityProperties` 설정 클래스에서 생성 → username(user) , password(랜덤 문자열) 제공
- `SpringBootWebSecurityConfiguration` 예제 코드
    
    ```java
    @Bean
    @Order(SecurityProperties.BASIC_AUTO_ORDER)
    SecurityFilterChain defaultSecurityFilterChain(HttpSecurity http){
    		http.authorizeRequests().anyRequest().authenticated(); // 모든요청(anyRequest)에 인증(authenticated)를 받아야 한다.
    		http.formLogin(); // 폼 로그인 인증방식 제공
    		http.httpBasic(); // httpBasic 인증방식 제공
    		
    		return http.build();
    }
    ```
    
- 위 코드의 문제점 !!
    - ==_**현재 시큐리티가 제공하는 한개의 계정으로 인증이 가능하지만 현재는 다른 계정들을 추가할 수 있는 방법이 없다 !**_==
    - _==**계정의 사용자들의 권한(관리자 or 매니저 or 사용자 등등)을 체크할 수 있는 기능이 없다 !!**==_

## 기본 보안 설정 클래스들

- 기본계정을 생성하는 클래스는 `SecurityProperties` 이다. 클래스 코드 내부를 보면 안에 `User` 라는 클래스가 있다.
    
    ```java
    @ConfigurationProperties(prefix = "spring.security")
    public class SecurityProperties {
    		// 생략 ...
    		
    		public static class User {
    
    		/**
    		 * Default user name.
    		 */
    		private String name = "user";
    
    		/**
    		 * Password for the default user name.
    		 */
    		private String password = UUID.randomUUID().toString();
    
    		/**
    		 * Granted roles for the default user name.
    		 */
    		private List<String> roles = new ArrayList<>();
    
    		private boolean passwordGenerated = true;
    
    		public String getName() {
    			return this.name;
    		}
    
    		public void setName(String name) {
    			this.name = name;
    		}
    
    		public String getPassword() {
    			return this.password;
    		}
    
    		public void setPassword(String password) {
    			if (!StringUtils.hasLength(password)) {
    				return;
    			}
    			this.passwordGenerated = false;
    			this.password = password;
    		}
    	
    		// 생략 ..
    }
    ```
    
- 그리고 `SpringBootWebSecurityConfiguration` 클래스에 의해 기본적인 보안 설정들이 이루어진다.
- 아래 `SecurityFilterChainConfiguration` 클래스는 `@ConditionalOnDefaultWebSecurity` 어노테이션에 의해 특정 조건을 만족해야 실행된다.
    
    ```java
    @Configuration(proxyBeanMethods = false)
    @ConditionalOnWebApplication(type = Type.SERVLET)
    class SpringBootWebSecurityConfiguration {
    		
    		
    		@Configuration(proxyBeanMethods = false)
    		@ConditionalOnDefaultWebSecurity
    		static class SecurityFilterChainConfiguration {
    
    			@Bean
    			@Order(SecurityProperties.BASIC_AUTH_ORDER)
    			SecurityFilterChain defaultSecurityFilterChain(HttpSecurity http) throws Exception {
    				http.authorizeHttpRequests((requests) -> requests.anyRequest().authenticated());
    				http.formLogin(withDefaults());
    				http.httpBasic(withDefaults());
    				return http.build();
    			}
    }
    ```
    
- `@ConditionalOnDefaultWebSecurity` 어노테이션 내부로 들어가면 아래와 같이 나온다.
    
    ```java
    @Target({ ElementType.TYPE, ElementType.METHOD })
    @Retention(RetentionPolicy.RUNTIME)
    @Documented
    @Conditional(DefaultWebSecurityCondition.class)
    public @interface ConditionalOnDefaultWebSecurity {
    
    }
    ```
    
- `DefaultWebSecurityCondition` 클래스로 이동하면 조건 2개가 보인다.
    
    - `@ConditionalOnClass` 어노테이션에 의해 `SecurityFilterChain` 클래스와 `HttpSecurity` 클래스가 클래스 패스에 존재하는지 체크한다.
    - `@ConditionalOnMissingBean` 어노테이션에 의해 `SecurityFilterChain` 클래스의 `Bean` 이 생성되어 있지 않아야 참이다.
    - 위 두 조건이 참이여야 기본 보안 기능이 설정된다 !
    
    ```java
    class DefaultWebSecurityCondition extends AllNestedConditions {
    
    	DefaultWebSecurityCondition() {
    		super(ConfigurationPhase.REGISTER_BEAN);
    	}
    
    	@ConditionalOnClass({ SecurityFilterChain.class, HttpSecurity.class })
    	static class Classes {
    
    	}
    
    	@ConditionalOnMissingBean({ SecurityFilterChain.class })
    	static class Beans {
    
    	}
    
    }
    ```
    

  

## HttpBasic vs FormLogin

웹 애플리케이션에서 사용자 인증 방법으로 자주 사용되는 'Form 로그인 방식'과 'httpBasic 로그인 방식'은 다음과 같은 주요 차이점이 있습니다.

### 1. Form 로그인 방식

Form 로그인은 사용자가 웹 페이지의 로그인 양식을 통해 자신의 아이디와 비밀번호를 입력하고, 이 정보를 서버에 전송하여 인증을 받는 방식입니다.

**특징:**

- **사용자 인터페이스 제공:** 사용자는 HTML 양식을 통해 자신의 정보를 입력합니다. 이는 사용자에게 친숙하고 직관적인 인터페이스를 제공합니다.
- **맞춤형 인증:** 서버는 사용자의 로그인 정보를 받아 복잡한 로직을 처리할 수 있습니다. 예를 들어, 로그인 실패 시 재시도 횟수 제한, 캡차 도입, 이중 인증 등 추가 보안 조치를 적용할 수 있습니다.
- **세션 관리:** 일반적으로 세션 쿠키를 사용하여 사용자의 로그인 상태를 유지합니다. 이는 사용자가 사이트 내에서 페이지를 이동할 때마다 로그인을 유지할 수 있게 합니다.

**예시:**  
웹사이트에 로그인 페이지가 있고, 사용자가 이메일과 비밀번호를 입력한 후 '로그인' 버튼을 클릭합니다. 서버는 이 정보를 받아 데이터베이스에서 확인하고, 일치하면 사용자에게 세션 쿠키를 발급하여 로그인 상태를 유지합니다.

### 2. HTTP Basic Authentication (httpBasic 로그인 방식)

HTTP Basic Authentication은 HTTP 헤더를 통해 사용자 아이디와 비밀번호를 인코딩된 형태로 서버에 전송하여 인증을 받는 방식입니다. 이 정보는 매 요청마다 전송되어야 합니다.

**특징:**

- **간단한 인증 메커니즘:** 사용자 인터페이스 없이 HTTP 헤더를 통해 인증 정보를 전송합니다. 이는 API나 서버 간 통신에 적합합니다.
- **상태 비저장:** HTTP Basic은 상태를 유지하지 않는(stateless) 프로토콜입니다. 따라서 매 요청마다 사용자 인증 정보를 전송해야 합니다.
- **보안 취약점:** 인증 정보가 Base64로 인코딩되어 전송되기 때문에, HTTPS를 사용하지 않으면 네트워크 스니핑을 통해 정보가 노출될 위험이 있습니다.

**예시:**  
클라이언트가 서버에 요청을 보낼 때, HTTP 헤더에 `Authorization: Basic <인코딩된 사용자명과 비밀번호>`를 포함하여 보냅니다. 서버는 이 정보를 디코딩하여 유효성을 검사하고, 유효하다면 요청에 응답합니다.

### 차이점 요약

- **사용자 경험:** Form 로그인은 사용자 친화적인 웹 인터페이스를 제공하는 반면, HTTP Basic은 API 또는 서버 간 통신에 주로 사용되며 사용자 인터페이스를 제공하지 않습니다.
- **보안:** Form 로그인은 HTTPS 사용 시 보안이 강화됩니다. HTTP Basic은 항상 HTTPS를 사용해야 하며, 그렇지 않으면 정보가 쉽게 노출될 수 있습니다.
- **상태 관리:** Form 로그인은 세션을 통해 사용자 상태를 관리하지만, HTTP Basic은 상태 비저장 방식으로 매 요청마다 인증 정보를 전송해야 합니다.

이러한 차이점을 이해하고 각 상황에 맞는 인증 방식을 선택하는 것이 중요합니다.
