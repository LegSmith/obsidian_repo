## API URI 설계

---

설계를 할 때 ==**리소스를 식별하면서 URI 계층 구조를 활용**==하면 아래와 같이 설계할 수 있다.

- 회원 목록 조회 : `/members`
- 회원 조회 : `/member/{id}`
- 회원 등록 : `/member/{id}`
- 회원 수정 : `/member/{id}`
- 회원 삭제 : `/member/{id}`

    → 조회,등록,수정,삭제 URI 는 같지만 구분할 수 있는게 필요하다


### 리소스와 행위를 분리

- **==URI 는 리소스만 식별 !==**
- 행위는 조회, 등록, 삭제, 변경이다. → **==HTTP 메서드로 구분한다==**

## HTTP 메서드

---

주요 메서드  
`GET` : 리소스 ==**조회**==  
`POST` : 요청 데이터 처리, 주로 ==**등록**==에 사용  
`PUT` : 리소스를 ==**대체**==, 해당 리소스가 없으면 생성  
`PATCH` : 리소스 부분 ==**변경**==  
`DELETE`==**삭제|**==  

**그 외 메서드**  
`HEAD` : 과 동일하지만 상태 줄과 헤더만 반환  
`OPTIONS` : 대상 리소스에 대한 통신 가능 옵을 설명 → CORS 에서 주로사용  
`CONNECT` : 대상 자원으로 식별되는 서버에 대한 터널을 설정  
`TRACE` : 대상 리소스에 대한 경로를 따라 메시지 루프백 테스트를 수행

### GET

- **리소스 조회**
- 서버에 전달하고 싶은 데이터를 `Query(쿼리 파라미터, 쿼리 스트링)`를 통해서 전달
- **==보통 메시지 바디에 데이터를 넣지 않지만 바디에 데이터를 넣을 수 있다.==**

### POST

- **요청 데이터 처리**
- ==**메시지 바디를 통해 서버로 요청 데이터 전달**==
- 서버는 ==_메시지 바디를 통해 들어온 데이터를 처리_==하는 모든 기능을 수행
- 주로 신규 데이터 등록에 많이 쓰인다.
- 클라이언트가 메시지바디에 데이터를 담아 POST 요청을 하면 **==서버는 신규 리소스 식별자를 생성하고 HTTP 201 코드와 Location(자원이 생성된경로)를 같이 보낸다.==**

    ![[Untitled 37.png|Untitled 37.png]]

- POST 는 요청 데이터를 다양하게 처리한다.
    1. 새 리소스 생성(등록)
    2. ==**요청 데이터 처리**==
        1. 주문 서비스에서 **==결제완료 → 배달시작 → 배달완료 같은 프로세스 처리==**
        2. POST의 결과로 새로운 리소스가 생성되지 않을 수 있다.
        3. POST : `/orders/{orderId}/start-delivery` → ==**컨트롤 URI**==
    3. 다른 메서드로 처리하기 애매한 경우
        1. JSON 으로 조회 데이터를 넘겨야 하는데 GET 메서드 사용하기 어려운 경우

### PUT

- ==**리소스를 대체**==
    - 리소스가 있으면 대체
    - 리소스가 없으면 생성
    - **==즉, 리소스를 덮어버린다고 생각하면 된다.==**
- ==**클라이언트가 리소스를 식별**==
    - PUT : `/members/100` → 클라이언트가 요청을 할 때 리소스를 식별하고 있다.

### PATCH

- PUT 의 단점을 보완할 수 있게 ==**리소스를 부분 변경**==한다.
- 서버에서 PATCH 를 지원하지 않은 경우가 드물게 있다 → POST 를 사용!!

### DELETE

- **리소스 제거**
- **==HTTP 바디가 없다 !!==**

### 안전 (Safe)

- ==**호출해도 리소스를 변경하지 않는다**.== → `GET` 메서드는 안전
- 즉, 리소스의 변경이 있으면 안전하지 않다.

### 멱등 (Idempotent)

- ==**한 번 호출하든 두 번 호출하든 100번 호출하든 결과과 똑같다.**== → `f(f(x)) = f(x)`
- 멱등 메서드
    - `GET` : 여러 번 조회화든 같은 결과과 조회
    - `PUT` : 결과를 대체하기 떄문에 여러번 호출해도 똑같이 대체되서 결과가 같다.
    - `DELETE` : 리소스를 여러번 삭제해도 결과과 같다.
    - `POST` : 주문시스템이라면 **==여러변 결제하면 그만큼 나가는 금액이 달라지므로 멱등이 아니다.==**
- 멱등 메서드 활용
    - 자동 복구 메커니즘

        → `DELETE` 를 호출했는데 서버에 응답이없을 때 다시 시도해도 괜찮다 !

- **==멱등은 외부 요인으로 중간에 리소스가 변경되는 것 까지는 고려하지 않는다.==**

### 캐시가능 (Casheable)

- 응답 결과 리소스를 캐시해서 사용해도 되는가 ?
- `GET` , `HEAD` , `POST` , `PATCH` 메서드는 캐시가능 → 실제로는 `GET` , `HEAD` 정도만 사용
    - `POST` , `PATCH` 는 구현이 쉽지 않다.
