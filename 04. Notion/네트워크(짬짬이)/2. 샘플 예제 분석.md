## 1. Ethernet 헤더 분석 및 LSO 이슈

![[image 24.png|image 24.png]]

- 위 그림은 Ethernet 헤더이다. 실제 L2 Frame 의 헤더라 생각하면 된다.
- ==**L2 Frame 헤더의 기본적인 크기는 14Byte**==이다. 추가로 _Trailer(Frame Check Sequence, CRC) 데이터가 붙으면 그 데이터만큼 늘어나긴한다._
- 하지만 보통의 경우는 (`Destination MAC ADdress` , `Source MAC address`, `Type` , `Data`) 까지만 L2 Frame 이라 한다
    - 이 때, ==**Data 의 길이는 가변**==이다.
- 그래서 ==**Data 를 포함하면 보통 1514Byte이다. 그중 14Byte에 해당되는데 Ethernet 헤더이다. 1500Byte가 남는데 보통 Packet 의 크기**==이다.
- **Destination MAC address(목적지 주소)는 6Byte(48bit)를 차지**한다.
    - 그림을 봤을 떄 0 부터 31이라 총 32Bit(4Byte)이고 그 아래에 16Bit(2Byte) 만큼 차지한다.
- **Source MAC address(출발지 주소)는 총 6Byte(48bit)를 차지**한다.
- **Type 은 상위 계층 프로토콜(예: IPv4, ARP, IPv6 등)을 식별하는 필드로 2Byte(16Bit)를 차지**한다.. 쉽게 말해서 Payload 가 어떤 데이터인지 판별할 수 있다.
    
    - Type 은 국제 표준에 따라 정해진다.
    - 만약 Type 값이 `0x0800` 이면 Payload 는 IPv4 값이다.
    - 그리고 `0x0806` 은 ARP(Address Resolution Protocol)은 LAN 에서 자주보인다.
    
    ![[image 1 14.png|image 1 14.png]]
    
- 보통 `MAC address` 라 하면 ==**Destination MAC address, Source MAC address, Type**== 까지를 말한다.
- 그리고 Data 는 가변길이이고 Payload 라 한다.
- 즉, 이 ==**Payload 를 실어나르기 위한 Ethernet 헤더**==이다.

  

### MAC address

- MAC address 는 보통 NIC(Network Interface Card)를 식별하는 식별자이다.
- 그런데 NIC 을 구현하는 Driver에도 MAC address 가 있다.
- 그렇게 되면 우선순위는 Driver 가 NIC 보다 우선이다.
    - MAC address 는 H/W 주소이기 때문에 변경불가능하다 생각하지만 Driver에서 주소를 바꿔버리면 된다.
    - 즉, ==**보안성이 없다 !**==

### 바이트 순서(Byte Order)방법

- 데이터를 저장하거나 전송할 때 다중 바이트(16bit, 32bit, 64bit)로 표현된 값을 _==**메모리나 네트워크 상에 어떻게 배열할지 결정하는 규칙**==_이다.

1. **Network Order**
    - 네트워크에서 데이터를 전송할 때 사용되는 **표준 바이트 순서**, `빅 엔디안(Big Endian)` 방식으로 데이터를 배열
    - 예제) 16진수 `0x1234` 를 Network Order 로 전송하면 메모리에 [12] [34] 로 저장된다.
2. **Host Order**
    
    - 시스템(호스트)의 CPU나 운영체제가 사용하는 기본 바이트 순서.
    - CPU 아키텍쳐에 따라 빅엔디안(Big Endian) , 리틀 앤디안(Little Endian) 방식을 쓴다.
    - 리틀 엔디안(Little Endian) 은 덜 중요한 바이트가 먼저 배열한다.
    - 예제) 16진수 `0x1234` 를 Host Order 로 전송하면 메모리에 [34][12] 로 저장된다.
    
    보통 ==**Intel 프로세서를 사용하면 Host Order 방식을 채택**==하고 있다 !!  
    그래서 _Type 값을 볼 때 국제표준으로 0800 이 IPv4 인데 Intel 프로세서를 탑재한 PC에서 볼때는 0008 로 봐야한다 !!_
    

  

### Ethernet 헤더 C 코드로 정의

```c
\#pragma pack(push, 1)  // 구조체 메모리 정렬을 1바이트로 설정
typedef struct EtherHeader {
    unsigned char dstMac[6];  // 목적지 MAC 주소
    unsigned char srcMac[6];  // 출발지 MAC 주소
    unsigned short type;      // 이더넷 타입 (예: IP, ARP 등)
} EtherHeader;
\#pragma pack(pop)  // 구조체 메모리 정렬을 원래대로 복구
```

- 위 `#prgma pack(push,1)` 명령어로 아래 EtherHeader 구조체를 메모리에 연속으로 배열되게 선언해주는 코드이다.
    
    _==**참고로 C언어에서는 6Byte 자료형이 없기 때문에 unsigned char(1Byte)자료형을 6개의 배열(1Byte * 6)으로 구현했다.**==_
    
- 위와같이 설정안하면 C컴파일러가 알아서 최적화 하기 때문에 패킷분석 시 오류가 날 수 있기 때문에 위와 같이 설정했다.
    - 컴파일러 설정을 바꾸는 방법도 있다 !
- 즉, 어떤 프로토콜을 구조체로 선언하여 정의할려고하면 `#pragma pack` 으로 구조체 앞 뒤를 묶어주면 오류를 피할 수 있다.
- 전체 코드로 보면 나머지는 동일하고 `packet_handler()` 이 함수에서 캡쳐한 패킷들을 어떻게 할지 정하는 로직이 들어간다.
    
    ```c
    \#include <pcap.h>
    \#pragma comment(lib, "wpcap") // wpcap 라이브러리를 썻다는 것을 정의.
    
    \#include <stdio.h>
    \#include <tchar.h>
    
    
    // Ethernet 구조체 정의
    \#pragma pack(push, 1)
    typedef struct EtherHeader {
    	unsigned char dstMac[6]; // 목적지 주서
    	unsigned char srcMac[6]; // 출발지 주소
    	unsigned short type;     // Payload 타입
    } EtherHeader;
    \#pragma pack(pop)
    
    
    
    BOOL LoadNpcapDlls()
    {
    	_TCHAR npcap_dir[512];
    	UINT len;
    	len = GetSystemDirectory(npcap_dir, 480);
    	if (!len) {
    		fprintf(stderr, "Error in GetSystemDirectory: %x", GetLastError());
    		return FALSE;
    	}
    	_tcscat_s(npcap_dir, 512, _T("\\Npcap"));
    	if (SetDllDirectory(npcap_dir) == 0) {
    		fprintf(stderr, "Error in SetDllDirectory: %x", GetLastError());
    		return FALSE;
    	}
    	return TRUE;
    }
    
    /* prototype of the packet handler */
    void packet_handler(u_char* param, const struct pcap_pkthdr* header, const u_char* pkt_data);
    
    int main()
    {
    	pcap_if_t* alldevs;
    	pcap_if_t* d;
    	int inum;
    	int i = 0;
    	pcap_t* adhandle;
    	char errbuf[PCAP_ERRBUF_SIZE];
    
    	/* Load Npcap and its functions. */
    	if (!LoadNpcapDlls())
    	{
    		fprintf(stderr, "Couldn't load Npcap\n");
    		exit(1);
    	}
    
    	/* Retrieve the device list */
    	if (pcap_findalldevs(&alldevs, errbuf) == -1)
    	{
    		fprintf(stderr, "Error in pcap_findalldevs: %s\n", errbuf);
    		exit(1);
    	}
    
    	/* Print the list */
    	for (d = alldevs; d; d = d->next)
    	{
    		printf("%d. %s", ++i, d->name);
    		if (d->description)
    			printf(" (%s)\n", d->description);
    		else
    			printf(" (No description available)\n");
    	}
    
    	if (i == 0)
    	{
    		printf("\nNo interfaces found! Make sure Npcap is installed.\n");
    		return -1;
    	}
    
    	printf("Enter the interface number (1-%d):", i);
    	scanf_s("%d", &inum);
    
    	if (inum < 1 || inum > i)
    	{
    		printf("\nInterface number out of range.\n");
    		/* Free the device list */
    		pcap_freealldevs(alldevs);
    		return -1;
    	}
    
    	/* Jump to the selected adapter */
    	for (d = alldevs, i = 0; i < inum - 1; d = d->next, i++);
    
    	/* Open the device */
    	/* Open the adapter */
    	if ((adhandle = pcap_open_live(d->name,	// name of the device
    		65536,			// portion of the packet to capture. 
    		// 65536 grants that the whole packet will be captured on all the MACs.
    		1,				// promiscuous mode (nonzero means promiscuous)
    		1000,			// read timeout
    		errbuf			// error buffer
    	)) == NULL)
    	{
    		fprintf(stderr, "\nUnable to open the adapter. %s is not supported by Npcap\n", d->name);
    		/* Free the device list */
    		pcap_freealldevs(alldevs);
    		return -1;
    	}
    
    	printf("\nlistening on %s...\n", d->description);
    
    	/* At this point, we don't need any more the device list. Free it */
    	pcap_freealldevs(alldevs);
    
    	/* start the capture */
    	pcap_loop(adhandle, 0, packet_handler, NULL);
    	pcap_close(adhandle);
    	return 0;
    }
    
    
    /* Callback function invoked by libpcap for every incoming packet */
    void packet_handler(u_char* param, 
    					const struct pcap_pkthdr* header,
    					const u_char* pkt_data) // pcap_loop()가 돌면서 패킷을 감지할때 그 때 읽어들인 감청 데이터는 pkt_data 에 들어간다.
    {
    	EtherHeader* pEther = (EtherHeader*)pkt_data; // 감청 데이터를 EtherHeader 로 형변환
    
    	printf("Destination MAC address : %02X-%02X-%02X-%02X-%02X-%02X\n",
    		pEther->dstMac[0], pEther->dstMac[1], pEther->dstMac[2], pEther->dstMac[3], pEther->dstMac[4], pEther->dstMac[5]);
    }
    ```

### LSO 이슈

- 위 예제에서 패킷의 길이까지 출력하다 보면 간혹 1514 가 넘는 경우가 있다.
    
    - 이 경우 1514 Byte를 넘는다는 뜻.
    
    ![[image 4 10.png|image 4 10.png]]
    
- 위 같이 나오는 경우는 NIC 의 옵션 때문이다.
- 사용하는 PC가 좋은 NIC 를 사용하는 경우, NIC 에 있는 NPU(Network Process Unit) 가 CPU 가 해야할 일을 일부 대신 한다.
- 그래서 CPU 를 덜 쓰게 되고 성능이 좋아지게 된다. 이런 경우 LSO(Large Send Offload) 기능이 들어가있다.
    - TCP Segmentation offload 도 LSO와 같은 기능이다.
- LSO 가 하는 역할은 스트림 데이터를 TCP Segment로 분할 시 NIC 가 수행하도록 설정하는 기능이다.
- 1Gbps 급 NIC에서는 큰 문제가 없지만 10Gbps 급 고속 환경에서 장애 이슈가 있다.

### 핵심 함수 분석

- 이 예제는 실제 L2를 넘나드는 Ethernet 이 아닌 디스크에 저장된 오프라인 Pcap 파일을 캡쳐해서 분석하고 출력하는 예제이다.
- 우선 실제 디스크에 있는 PCAP 파일을 여는 코드는 main() 함수 안에 있으며 다음와 같다.
    
    ```c
    	if ((fp = pcap_open_offline(
    		"C:\\SampleTraces\\ip-fragments.pcap",
    		errbuf)) == NULL)
    	{
    		fprintf(stderr, "\nUnable to open the file %s.\n",
    			"C:\\SampleTraces\\ip-fragments.pcap");
    		return -1;
    	}
    ```
    
    - `pcap_open_offline(pcap_dir, errBuf)` : 실제 디스크에 있는 PCAP파일을 여는 함수이다.
        - `pcap_dir` : 열려고 하는 pcap 파일 경로
        - `errBuf` : 오류 메시지를 저장하는 버퍼
        - 반환값 : 성공 시 파일 핸들, 실패 시 NULL
    
      
    
- `main()` 함수에서 `pcap_loop()` 함수가 핵심이다. pcap_loop() 는 pcap 라이브러리에서 제공하는 함수로 기본 형태는 다음과 같다.
    
    ```c
    pcap_loop(pcap_t *p, int cnt, pcap_handler callback, u_char *user)
    ```
    
    - `pcap_t *p` : pcap_open_offline() 함수나 pcap_open_live
    - `int cnt` : 캡쳐할 패킷의 갯수를 정한다. 만약 cnt 값이 0 이면 파일의 끝까지 패킷을 처리하겠다는 뜻.
    - `pcap_handler callback` : 패킷이 캡쳐될 때 호출되는 콜백 함수
    - `u_char *user` : 사용자 데이터로 콜백 함수로 전달된다.
- 아래 코드는 디스크에 있는 오프라인 패킷 데이터를 캡쳐하여 처리하고 출력하는 함수이다.
    
    ```c
    void dispatcher_handler(u_char* temp1, 
    	const struct pcap_pkthdr* header, 
    	const u_char* pkt_data)            
    {
    	u_int i = 0;
    
    	/* print pkt timestamp and pkt len */
    	printf("%ld:%ld (%ld)\n", header->ts.tv_sec, header->ts.tv_usec, header->len);
    
    	EtherHeader* pEther = (EtherHeader*)pkt_data;
    
    	printf(
    		"SRC: %02X-%02X-%02X-%02X-%02X-%02X -> "
    		"DST: %02X-%02X-%02X-%02X-%02X-%02X, type:%04X\n",
    		pEther->srcMac[0], pEther->srcMac[1], pEther->srcMac[2],
    		pEther->srcMac[3], pEther->srcMac[4], pEther->srcMac[5],
    		pEther->dstMac[0], pEther->dstMac[1], pEther->dstMac[2],
    		pEther->dstMac[3], pEther->dstMac[4], pEther->dstMac[5],
    		htons(pEther->type));
    
    	/* Print the packet */
    	for (i = 1; (i < header->caplen + 1); i++)
    	{
    		printf("%.2x ", pkt_data[i - 1]);
    		if ((i % LINE_LEN) == 0) printf("\n");
    	}
    
    	printf("\n\n");
    }
    ```
    
    - `u_char * temp1` : 사용자 데이터를 전달받는 포인터(현재 예제에서는 NULL 을 넘기기 때문에 null)
    - `pcap_pkthdr* header` : 캡쳐된 패킷의 헤더(메타 데이터)를 담고 있는 구조체
        - `pcap_pkthdr` 은 구조체이며 코드는 다음과 같다.
            
            ```c
            	struct pcap_pkthdr {
            		struct timeval ts;	// 패킷 캡쳐 시간
            		bpf_u_int32 caplen; // 캡처된 패킷 데이터의 길이
            		bpf_u_int32 len;	// 실제 패킷의 전체 길이(네트워크에서 전달된 원본 패킷 크기)
            	};
            ```
            
    - `u_char* pkt_data` : 캡쳐된 패킷의 실제 내용(바이너리 데이터)
- `dispatcher_handler()` 함수의 전체 동작을 간략히 설명하면 다음과 같다.
    
    1. 캡쳐된 패킷의 헤더를 이용하여 패킷의 캡쳐시간(sec, msec)와 패킷의 전체 길이를 출력한다
    2. 패킷을 EtherHeader 로 형변환 후 출발지주소, 목적지주소, 데이터타입을 출력한다
    3. 패킷 데이터를 16진수로 출력
