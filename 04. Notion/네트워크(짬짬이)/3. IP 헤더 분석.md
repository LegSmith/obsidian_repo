## 1. IP 헤더 형식

기존 `Ethernet` 은 L2 에 있엇지만, `IP` 는 L3에 위치한다.

- 기존의 Ethernet 을 분석할 때는 Frame 단위의 데이터를 분석했지만, IP 즉 L3 에서는 Packet 단위의 데이터를 분석하게 된다.

### IP 헤더 구조

![[image 25.png|image 25.png]]

- 좌상단의 0, 우상단의 31을 보면 알 수 있듯이 한 줄의 32bit구조임을 알 수 있다.
- `Vrseion` : 해당 헤더가 IPv4 인지, IPv6 인지 구분한다.
- `IHL(Internet Header Length)` : IP 헤더의 길이이고 4bit 필드이다.
    - 만약 IHL 값이 5면 한줄의 32bit(4Byte) 이고 5줄이니까 총 헤더의 길이는 20Byte임을 알 수 있다. 하지만 Options 에 따라 IHL 의 길이는 바뀌지만, 보통 Options 필드가 없는 기본 IPv4 헤더는 20Byte이다.
- `Tos(Type of Service)` : 패킷의 대한 서비스 품질(QoS)를 지정하고자 만든 8bit 필드이다.
- `Total length` : 패킷의 길이이고 16bit 필드이다. 이 필드가 **==IP 패킷을 분석할 때 활용되는 필드==**이다.
- `Identification` : 패킷 단편화 시 동일한 ID를 가진 조각들을 재조립할 때 사용하는 필드이다.
- `Flags`(3bit) + `Fragment Offset`(13bit)
    - **Flags** : `0` 예약 , `DF(Don't Fragment)`는 단편화금지, `MF(More Fragments)`는 뒤에 더 많은 조각이 있음을 의미한다.
    - **Fragment Offset** : 원본 데이터의 **==어느 위치부터 이 패킷이 시작되는지 표시==**하는 필드이다.
- `TTL(Time to Live)` : 필드는 **IP 패킷이 네트워크를 통해 전달될 수 있는 최대 시간**을 설정하는 8비트(1바이트) 크기의 필드
- `Protocol` : Nest Protocal 이라 하는 필드이다.
    - Ethernet 에서 Type 필드에 따라 Ethernet 값을 뭘로 해석할지 정하게 되는데, IP 패킷에서 Protocol 필드는 INNA 에서 정한 값을 기준으로 상위(전송) 계층 프로토콜을 식별하는 필드이다.
- `Checksum` : IP 헤더 자체의 무결성을 검증하기 위한 필드이다.
    - Payload 는 포함하지 않고 IP 헤더만 대상이다.
- `Source address` : 출발지 주소
- `Destination address` : 목적지 주소
- `Options` : 기록라우트(Record Route), 타임스탬프, 보안 등 특별한 기능을 위해 확장된 필드이다.
    - 0~40 Byte 를 가지며, 잘 사용되지는 않는다.
- `Payload` : 실제 상위 계층으로 전송되는 데이터이다.


### IP 패킷구조 및 유의할 점

- IP 패킷의 구조는 크게보면 Header 와 Payload 로 이루어져 있다.

    |Header|Payload|
    |---|---|

- 패킷의 최대 크기를 `MTU(Maximum Transmission Unit)` 라 하고, 이 때 MTU 가 **==Total length 필드의 값==**이다.
- MTU 값의 최대크기는 ==**보통 1500byte(1.4 kbyte)**== 이다. 이유는 다음과 같다.
    - Internet 이란 거대한 Router 의 집합체라고 볼 수 있다.(항상은 아님;;)
    - 근데 Router 들이 대부분 MTU가 1500이다.
    - **==MTU 가 1500 이 넘어가면 단편화가 일어나기 때문에 효율이 떨어진다!==**
- 그리고 Total length 를 계산할 때는 유의할 점이 있다.
    - Total length 는 16bit 즉, **==2Byte 기 때문에 Byte Order 방법을 신경써야 한다.==**
    - 패킷에 쓰여지는 모든 데이터들은 Network Order 방식을 따르지만, 로컬 PC에서 패킷을 뺴서 연산을 할 경우 Intel 기준으로 Host Order 방식이기 때문에 값을 변환해야 한다!

### IP 헤더 C 언어 구현

- 통신시 데이터를 보면 L2 Frame > L3 Packet > L4 Segment 로 감싸 져잇다.

    ![[image 1 15.png|image 1 15.png]]

    출처 : [https://crnetpackets.com/2016/01/27/the-relation-between-maximum-transmission-unit-mtu-and-the-maximum-segment-size-mss/](https://crnetpackets.com/2016/01/27/the-relation-between-maximum-transmission-unit-mtu-and-the-maximum-segment-size-mss/)

- 그래서 기존 Ethernet 패킷 분석할 때 EtherHeader 구조체로 형변환하여 패킷캡쳐가 가능했던 이유도 네트워크를 통해 송수신되는 패킷의 제일 앞단 헤더가 Ethernet 구조이기 때문이다.
- 가장 앞단의 L2 Header 의 type 필드를 통해 다음에 올 데이터(L3 IP Packet)를 유추할 수 있었다.

    - IPv4 의 type은 `0x0800` 이다.

    ```c
    typedef struct EtherHeader {
    	unsigned char dstMac[6];
    	unsigned char srcMac[6];
    	unsigned short type;
    } EtherHeader;
    ```


### IP 헤더 구조체 정의

- `verIhl` : C언어에서는 4bit 자료형이 없기 때문에 `Version(4bit)`와 `IHL(4bit)`를 묶었다.

    - Version만 잘라낼 때는 비트연산을 해서 잘라내면 된다.

    ```c
    \#pragma pack(push, 1)
    typedef struct IpHeader {
    	unsigned char verIhl; 
    	unsigned char tos;
    	unsigned short length;
    	unsigned short id;
    	unsigned short fragOffset;
    	unsigned char ttl;
    	unsigned char protocol;
    	unsigned short checksum;
    	unsigned char srcIp[4];
    	unsigned char dstIp[4];
    } IpHeader;
    \#pragma pack(pop)
    ```


## 2. Frame 데이터 분석

### Encapsulation

- Encapsulation(캡슐화)는 상위계층 헤더 및 데이터가 캡슐화 되는 것이다.
- 이를 코드(C언어)로 구현하면 포인터라고 보면된다.
- 반대로 하위계층에서 상위계층으로 끄집어내는 것은 Decapsulation(디캡슐?) 이라 한다.
    - L2 Frame 에서 L2 Header를 벗겨내고, L3 Packet 에서 L3 Header를 벗겨내고 L4 Segment(TCP)까지 오게 되면 이 떄 L4 Header 를 까게되면 Payload 는 하나의 Stream 이다.
    - 이런 Stream 이 여러개가 와서 하나의 Stream 으로 해석하게 된다. 이 때부터는 Application 수준이다.

### 수작업 패킷 분석

![[image 2 13.png|image 2 13.png]]

- 우선 L2 Frame 에서 헤더구조를 보면 `Dst(목적지주소, 6Byte)→Src(출발지주소, 6Byte)→Type(2Byte)→Data(대략 1500Byte)` 로 구성되어 있다.
- 이 때 **==L2 Frame 헤더에서 Type 필드 값을 통해 Data(Payload) 값을 유추==**할 수 있다.
- 우선 위 사진은 16진수로 이루어져 있다.
    - 예) `0x00` → `0000 0000` 이므로 8bit, 즉 ==**하나의 데이터가 1byte**==이다.
- 즉, 앞에서 총 6개의 데이터가 Dst, 그다음 6 개가 Src, 그다음 2개가 Type 이므로 13,14 번째 데이터를 통해 Payload 를 유추할 수 있다.
    - 0x08 과 0x00 이므로 0800, 즉 ==**IPv4 데이터를 캡슐화**== 하고 있다.
- 그리고 `0x45` 부터는 **IP 헤더**이다.

    ![[image 3 12.png|image 3 12.png]]

- `0x45` 에서 4는 **Version**, 5 는 **IHL** 이므로 **==Packet 은 IPv4 이고 헤더의 총 길이는 20, 즉 Options 필드가 없다.==**
- `0x00` 은 ToS 값이고 0이다.
- `0x00` `0x30` 은 **Total length** 필드값이다. 10진수로 변환 시 **==총 패킷의 길이가 48==**인것을 알 수 있다.
- 그다음 `0x01` `0xc0` 은 **Identification** 필드의 값이다.
- 그다음 `0x40` 에서 4는 위 그림에서 **Fragment offset** 필드 좌측에 4개의 구분자에 1bit 씩 들어간다고 생각하면 된다.
    - 16진수 4는 2진수 `0100` 이므로 DF 필드값이 1, 즉 ==**단편화 금지**== 이다.
- 다음 `0x80` 은 **TTL**, `0x06` 은 `Protocol` 이다.
    - INNA에서 정한값을 기준은 `0x06`은 **TCP** 이다.
    - `0x80` 은 10진수로 128이다. Windows 에 경우 초기값이 128, Linux 계열은 256이다.
- 다음 `0x4b` `0xc5` 는 **Checksum** 필드값이다.
- 다음 `0xc0` `0xa8` `0x01` `0xb6` 은 **Src(출발지주소)**, `0xa1` `0x3a` `0x49` `0xaa` 는 **Dst(목적지 주소)**이다.
- 보통 그다음은 Options 필드값인데 **==IHL 값이 5이므로 헤더의 길이는 20이여서 Options 필드가 없으므로 그다음부터는 Payload==** 이다.
- 그다음 `Payload` 는 **TCP 헤더**이다.
    - Protocol 이 `0x60` 즉, **TCP** 이기 때문이다.
