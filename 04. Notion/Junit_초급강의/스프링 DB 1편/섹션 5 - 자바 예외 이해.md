## 예외 계층

---

![[Untitled 65.png|Untitled 65.png]]

- `Object` : 예외도 객체이기 때문에 모든 계층의 최상위 부모인 `Object` 가 있다.
- `Throwable` : **최상위 예외**이다. 하위에는 `Exception` , `Error` 가 있다.
- `Error` : 메모리 부족이나 심각한 시스템 오류와 같이 **==애플리케이션에서 복구 불가능한 시스템 예외==**이다.
    - try-catch 문에서 **==Error 나 상위 예외인 Throwable 로 잡아서는 안된다**__._== 애플리케이션 로직에서는 `Exception` 으로 예외를 잡아야 한다.
    - 참고로 `Error` 도 **언체크 예외**이다.
- `Exception` : **체크 예외**
    - ==**애플리케이션 로직에서 사용할 수 있는 실질적인 최상위 예외**==이다
    - `Exception` 과 그 하위 예외는 모두 ==**컴파일러가 체크하는 체크 예외**==이다. ( 단, `RuntimeException` 은 예외로 한다 )
- `RuntimeException` : 언체크 예외 or 런타임 예외
    - **==컴파일러가 체크하지 않은 언체크 예외==**이다.
    - `RuntimeException` 과 ==**그 하위 예외는 모두 언체크 예외**==이다.

## 체크 예외 기본 이해

---

예외를 처리하는 방법은 `throw` 로 ==**호출한 쪽으로 예외를 던지거나**==, `try-catch` 로 ==**예외를 처리하여 정상흐름으로 동작**==하게 만들 수 있다.

### try-catch 로 예외처리

- `MyCheckedException` 은 ==**체크 예외인 Exception 을 상속받아 간단하게 메시지를 받는 생성자를 가진 클래스**==이다.
- `Service` 클래스에서 `callCatch()`는 Repository의 `call()` 메서드를 호출하여 `call()` 에서 ==**던진 예외를**== `callCatch()` ==**에서 처리**==하는 시나리오이다.

    ```java
    @Slf4j
    public class CheckedTest {

        @Test
        @DisplayName("try-catch로 예외처리")
        void checked_catch() {
            Service service = new Service();
            service.callCatch();
        }

        /**
         * Exception 을 상속받은 예외는 체크 예외가 된다.
         */
        static class MyCheckedException extends Exception {
            public MyCheckedException(String message) {
                super(message);
            }
        }

        /**
         * Checked 예외는
         * 예외를 잡아서 처리하거나, 던지거나 둘 중 하나를 선택해야 한다.
         */
        static class Service {
            Repository repository = new Repository();

            /**
             * 예외를 잡아서 처리하는 코드
             */
            public void callCatch() {
                try {
                    repository.call();
                } catch (MyCheckedException e) {
                    log.info("예외 처리, message={}", e.getMessage(), e);
                }
            }

            /**
             * 체크예외를 밖으로 던지는 코드
             * 체크 예외는 예외를 잡지 않고 밖으로 던지려면 throws 예외를 메서드에 필수로 선언해야 한다
             */
            public void callThrow() throws MyCheckedException {
                repository.call();
            }
        }

        static class Repository {
            public void call() throws MyCheckedException {
                throw new MyCheckedException("ex");
            }
        }
    }
    ```

### throws 로 예외 던지기

- `throws` 는 예외가 터진 부분에서 ==**해당 메서드를 호출한 쪽으로 계속 예외를 던지게 된다.**==
- 아래 테스트에서는 이런 흐름으로 에러가 터진다.

    1. `callThrow()` 호출
    2. `repository.call()` 이 호출하면서 예외발생, throws로 호출한쪽으로 예외던짐.
    3. `callThrow()`를 호출한 `checked_throw()`로 예외 던짐.
    4. `cheked_throw()` 를 호출한 Test 로 예외던짐
    5. 예외해결하지 못해서 테스트 실패

    ```java
    @Slf4j
    public class CheckedTest {

        @Test
        @DisplayName("throws 로 예외 던지기")
        void checked_throw() throws MyCheckedException {
            Service service = new Service();
    				service.callThrow();
        }

        /**
         * Exception 을 상속받은 예외는 체크 예외가 된다.
         */
        static class MyCheckedException extends Exception {
            public MyCheckedException(String message) {
                super(message);
            }
        }

        /**
         * Checked 예외는
         * 예외를 잡아서 처리하거나, 던지거나 둘 중 하나를 선택해야 한다.
         */
        static class Service {
            Repository repository = new Repository();

            /**
             * 예외를 잡아서 처리하는 코드
             */
            public void callCatch() {
                try {
                    repository.call();
                } catch (MyCheckedException e) {
                    log.info("예외 처리, message={}", e.getMessage(), e);
                }
            }

            /**
             * 체크예외를 밖으로 던지는 코드
             * 체크 예외는 예외를 잡지 않고 밖으로 던지려면 throws 예외를 메서드에 필수로 선언해야 한다
             */
            public void callThrow() throws MyCheckedException {
                repository.call();
            }
        }

        static class Repository {
            public void call() throws MyCheckedException {
                throw new MyCheckedException("ex");
            }
        }
    }
    ```

### 체크 예외의 장단점

- **장점** : 개발자가 실수로 예외를 누락해도 ==**컴파일에러를 통해 컴파일 전에 문제를 잡아줄 수 있다.**==
- **단점** : ==**모든 체크 예외를 반드시 잡거나 던지도록 처리**==해야 하기 때문에 번거롭다.

## 언체크 예외 기본 이해

---

언체크 예외는 `RuntimeException` 과 그 하위 예외들을 언체크 예외라 한다.  
언체크 예외는 말 그대로 ==**컴파일러가 체크하지 않는 예외**==를 말한다.  
기본적으로 `throws` 키워드를 사용하지 않고 ==**예외가 터질 경우 자동으로 던진다.**==

- 언체크 예외도 ==**try-catch문으로 잡아서 예외처리를 할 수 있다**==
- `throws` 키워드를 사용하지 않아도 ==**자동으로 던지기 때문에 해당 메서드에 throws 를 생략**== 가능하다.

    ```java
    @Slf4j
    public class uncheckedTest {

        @Test
        @DisplayName("언체크 예외 - catch로 예외잡기")
        void unchecked_catch() {
            Service service = new Service();
            service.callCatch();
        }

        @Test
        @DisplayName("언체크 예외 - 자동으로 throws")
        void unchecked_throw() {
            Service service = new Service();
            Assertions.assertThatThrownBy(() -> service.callThrow()).isInstanceOf(MyUncheckedException.class);
        }

        /**
         * RuntimeException 을 상속받는 예외는 언체크 예외가 된다.
         */
        static class MyUncheckedException extends RuntimeException {
            public MyUncheckedException(String message) {
                super(message);
            }
        }

        /**
         * Unchecked 예외는 예외를 잡거나, 던지지 않아도 된다.
         * 예외를 잡지 않으면 자동으로 밖으로 던진다.
         */
        static class Service {
            Repository repository = new Repository();

            /**
             * 필요한 경우 예외를 잡아서 처리하면 된다.
             */
            public void callCatch() {
                try {
                    repository.call();
                } catch (MyUncheckedException e) {
                    log.info("예외 처리, message={}", e.getMessage(), e);
                }
            }

            /**
             * 예외를 잡지 않아도 된다. 자동으로 상위로 넘어간다.
             * 체크예외와 다르게 throws 예외 선언을 하지 않아도 된다.
             */
            public void callThrow() {
                repository.call();
            }
        }

        static class Repository {
            public void call() {
                throw new MyUncheckedException("ex");
            }
        }
    }
    ```


### 언체크 예외의 장단점

- **장점** : 신경쓰고 싶지 않은 언체크 예외를 무시할 수 있다.
- **단점** : 언체크 예외는 개발자가 실수로 누락할 수 있다.

## 체크 예외 활용

---

체크예외 or 언체크예외를 사용하기 위해서는 다음 2가지 원칙을 기억하면 된다.

- ==**기본적으로 언체크(RuntimeException)예외를 사용하자**==
- 체크 예외는 ==**비즈니스 로직상 의도적으로 던지는 예외에만 사용**==하자(Custome Exception 등등)

### 체크 예외 문제점

![[Untitled 3 25.png|Untitled 3 25.png]]

- `Service` 단에서는 2**개의 예외를 처리할 수 없으므로 둘 다 밖으로 던지고 throws 키워드**를 사용한다.
- `Controller` 단에서도 **처리할 방법이 없기 때문에 throws 를 사용하여 밖으로 던진다.**
- 웹 어플리케이션일아면 **서블릿 오류 페이지**라던가 **스프링MVC** 가 제공하는 `ControllerAdvice` 를 통해 예외를 공통 처리한다.
- 이렇게 되면 사용자에게는 "서비스에 문제가 있습니다" 라고 메시지를 보여주고 개발자가 빨리 수정을 해야 한다.
    - 만약 `SQLException` 이라면 ==**개발자가 해당 SQL을 수정해서 배포하기 전까지 모든 사용자가 문제를 겪게 된다.**==

### 체크 예외 문제점 - 코드

- 위 그림 처럼 가상의 `Service` 와 `NetworkClient` , `Repository` , `Controller` 를 만들었다.
- `SQLException` 과 `ConnectExcepton` 이 터질 경우 **Service 와 Controller 가 처리를 못하기 때문에** `throws SQLException, ConnectException` 를 계속 선언해줘야 한다.

    ```java
    public class CheckedAppTest {

        @Test
        void checked() {
            Controller controller = new Controller();
            Assertions.assertThatThrownBy(() -> controller.request()).isInstanceOf(Exception.class);
        }

        static class Controller {
            Service service = new Service();

            public void request() throws SQLException, ConnectException {
                service.logic();
            }
        }

        static class Service {
            Repository repository = new Repository();
            NetworkClient networkClient = new NetworkClient();

            public void logic() throws SQLException, ConnectException {
                repository.call();
                networkClient.call();
            }
        }

        static class NetworkClient {
            public void call() throws ConnectException {
                throw new ConnectException("연결 실패");
            }
        }

        static class Repository {
            public void call() throws SQLException {
                throw new SQLException("SQL Ex");
            }
        }
    }
    ```


### 2가지 문제

1. **복구 불가능한 예외**
2. **의존 관계에 대한 문제**

    ### 복구 불가능한 예외

    - ==**대부분의 예외는 복구가 불가능**==하다.
    - SQLException 을 예로 들면 DB에 문제가 생겼을 때는 SQL 문법에 문제가 있거나 DB 자체의 문제가 발생했을 수 있다. 이런 문제들은 대부분 복구가 불가능하다.

    ### 의존 관계에 대한 문제

    - 위에 코드들을 보면 `SQLException` 과 `ConnectException` 을 **throws 하는 코드가 Service에도 있고 Controller에도 있다.**
    - 치명적인 문제는 `SQLException` 이다. **해당 예외는 JDBC에서 제공하는 예외**이므로 만약 JDBC가 아닌 **JPA로 바꾼다면 예외 자체를 바꿔야한다.** → ==**예외 때문에 Service,Controller 코드를 수정해야 한다.**==

### 정리

- ==**즉, 체크 예외때문에 throws 를 쓰면서 불필요한 의존관계가 생성된다.**==

## 언체크 예외 활용

---

![[Untitled 4 22.png|Untitled 4 22.png]]

- `SQLException` → `RuntimeSQLException` 으로 바꾸고 `ConnectException` → `RuntimeConnectException` 으로 바꾸었다.
    - ==**즉, 기존의 체크 예외를 RuntimeException 을 상속받는 언체크 예외로 교체해주었다.**==
- Repository 클래스에서 `runSQL()` 이라는 임의로 SQL을 실행하는 메서드를 생성하고 내부에는 ==**SQLException 이 터지도록 구현하였다.**==
- 그리고 실제 Service 에서 호출할 Repository 의 `call()` 메서드는 ==**SQLException 을 잡아서 RuntimeSQLException을 밖으로 던져주었다.**==
- ==**언체크 예외를 사용했기 때문에 Service 와 Controller 에는 기존의 의존적인 예외들이 사라졌다 !!**==

    ```java
    public class UnCheckedAppTest {

        @Test
        void unchecked() {
            Controller controller = new Controller();
            Assertions.assertThatThrownBy(() -> controller.request()).isInstanceOf(RuntimeSQLException.class);
        }

        static class Controller {
            Service service = new Service();

            public void request() {
                service.logic();
            }
        }

        static class Service {
            Repository repository = new Repository();
            NetworkClient networkClient = new NetworkClient();

            public void logic() {
                repository.call();
                networkClient.call();
            }
        }

        static class NetworkClient {
            public void call() {
                throw new RuntimeConnectException("연결 실패");
            }
        }

        static class Repository {
            public void call() {
                try {
                    runSQL();
                } catch (SQLException e) {
                    throw new RuntimeSQLException(e);
                }
            }

            public void runSQL() throws SQLException {
                throw new SQLException("ex");
            }

        }

        static class RuntimeConnectException extends RuntimeException {
            public RuntimeConnectException(String message) {
                super(message);
            }
        }

        static class RuntimeSQLException extends RuntimeException {
            public RuntimeSQLException(Throwable cause) {
                super(cause);
            }
        }
    }
    ```


## 예외 포함과 스택 트레이스

---

==**예외를 전환할 때는 꼭! 기존 예외를 포함해야 한다 !!!**== 그렇지 않으면 스택 트레이스를 확인할 때 심각한 문제가 발생한다.

- 예외를 전환하여 던질 때는 기존의 예외를 포함해서 던져야 한다.
- 그러기 위해서는 전환할 예외의 생성자중에 `Throwable` 객체를 받는 생성자가 필요하다.

    ```java
    public void call() {
    		try {
    				runSQL();
    		} catch (SQLException e) {
    				throw new RuntimeSQLException(e); 
    		}
    }

    static class RuntimeSQLException extends RuntimeException {
        public RuntimeSQLException(Throwable cause) {
            super(cause);
        }
    }
    ```

- `printStackTrace()` 는 `System.out` 을 통해 출력하지만 ==**실무에서는 무조건 로그로 출력해야 한다 !!**==

    ```java
    @Test
    void printEx() {
        Controller controller = new Controller();
        try {
            controller.request();
        } catch (Exception e) {
            //e.printStackTrace();
            log.info("ex", e);
        }
    }
    ```

- 출력 로그를 보면 Caused by 구문을 통해 무엇으로부터 예외가 터진지 추적이 가능하다 !!
