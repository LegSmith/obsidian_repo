## 애플리케이션 구조

---

- 애플리케이션 구조는 다양하지만 가장 많이 사용하고 간편한 구조는 ==**프레젠테이션 계층 ↔ 서비스 계층 ↔ 데이터 접근 계층**==이 있다.

    ![[Untitled 64.png|Untitled 64.png]]

- **프레젠테이션 계층**
    - UI 와 관련된 처리 담당
    - ==**웹 요청과 응답**==
- **서비스 계층**
    - ==**비즈니스 로직**==을 담당
- **데이터 접근 계층**
    - ==**실제 DB에 접근**==하는 코드

### 순수한 서비스 계층

- 위 구조 중 ==**가장 중요한 계층은 서비스 계층**==이다. 왜냐하면 UI를 담당하는 프레젠테이션 계층은 시간이 지남에 따라 바뀔 수 있고, 데이터 접근계층도 사용하는 DB에 따라 변경가능성이 있다. 하지만 **==서비스 계층은 최대한 변경없이 유지==**되어야 한다.
- ==**최대한 변경없이 유지할려면 서비스 계층을 특정 기술에 종속적이지 않게 개발**==해야 한다.


> 즉, **==서비스 계층은 가급적 비즈니스 로직만 구현하고 특정 구현 기술에 직접 의존해서는 안된다.==**

## 문제점들

---

- 이전에 만든 코드인 `MemberServiceV1` 을 보면 문제점을 알 수 있다.

    - 특정 기술에 의존하지 않고 코드를 짠 거 같지만 `SQLException` 을 throws 하는게 문제이다.
    - `SQLException` 은 ==**JDBC 에서 제공하는 예외**==이다. 즉, JDBC가 아닌 JPA 나 혹은 MongoDB 로 바꾸게 되면 다른 예외가 나가야한다.

    ```java
    import java.sql.SQLException;

    @RequiredArgsConstructor
    public class MemberServiceV1 {

        private final MemberRepositoryV1 memberRepository;

        public void accountTransfer(String fromId, String toId, int money) throws SQLException {
            Member fromMember = memberRepository.findById(fromId);
            Member toMember = memberRepository.findById(toId);

            memberRepository.update(fromId, fromMember.getMoney() - money);
            validation(toMember);
            memberRepository.update(toId, toMember.getMoney() + money);

        }

        private static void validation(Member toMember) {
            if(toMember.getMemberId().equals("ex")){
                throw new IllegalStateException("이체중 예외 발생");
            }
        }
    }
    ```

- 그리고 `MemberServiceV2` 에는 문제가 명확히 보인다.

    - 여기에서 `rollback()` 이나 `release()` 메서드도 **JDBC 기술에 의존**하고 있다.
    - 이러한 **==트랜잭션 기능들은 전부 JDBC에 의존적인 기능==**이다.

    ```java
    @Slf4j
    @RequiredArgsConstructor
    public class MemberServiceV2 {

        private final DataSource dataSource;
        private final MemberRepositoryV2 memberRepository;

        public void accountTransfer(String fromId, String toId, int money) throws SQLException {
            Connection con = dataSource.getConnection();
            try {
                con.setAutoCommit(false); // 트랜잭션 시작
                bizLogic(con, fromId, toId, money);
                con.commit(); // 로직이 예외없이 수행했기 때문에 commit 실행
            } catch (Exception e) {
                // 예외발생시 해당구문으로 이동
                con.rollback(); // 실패 했기 때문에 롤백 수행
                throw new IllegalStateException(e);
            } finally {
                if (con != null) {
                    try {
                        release(con);
                    } catch (Exception e) {
                        log.error("error", e);
                    }
                }
            }
        }

        private void bizLogic(Connection con, String fromId, String toId, int money) throws SQLException {
            // 비즈니스 로직
            Member fromMember = memberRepository.findById(con, fromId);
            Member toMember = memberRepository.findById(con, toId);

            memberRepository.update(con, fromId, fromMember.getMoney() - money);
            validation(toMember);
            memberRepository.update(con, toId, toMember.getMoney() + money);
        }

        private static void release(Connection con) throws SQLException {
            // Connection 을 설정할 때 autocommit 을 자동으로 했기 때문에 커넥션풀에 반환할 때도 자동인 채로 반환된다.
            // 다른 누군가 autocommit 이 자동인줄 알고 connection 을 획득할 수 있기 때문에 설정을 원래대로 하고 반환한다.
            con.setAutoCommit(true);
            con.close();
        }

        private static void validation(Member toMember) {
            if (toMember.getMemberId().equals("ex")) {
                throw new IllegalStateException("이체중 예외 발생");
            }
        }
    }
    ```


### 문제점들 정리

1. **트랜잭션 문제**
    - ==**JDBC 구현 기술이 서비스 계층에 누수**==되는 문제이다.
    - **트랜잭션을 사용하기 위해 JDBC 기술 코드**들이 서비스계층으로 들어왔다.
    - **같은 트랜잭션을 유지하기 위해 커넥션을 파라미터**로 넘겨야 한다. → 즉, ==**트랜잭션용 기능과 트랜잭션을 유지하지 않아도되는 기능으로 분리해**==야 한다.
    - 트랜잭션 적용 코드에 반복이 많다 → `try-catch-finally`
2. **예외 누수 문제**
    - `SQLException` 은 **JDBC 전용 예외**이다. → **==JDBC가 아닌 JPA 나 다른 기술을 쓰면 다른 예외가 사용되므로 수정이 필요해진다 !==**
3. **JDBC 반복 문제**
    - 유사한 코드의 반복이 너무 많다.
    - `try-catch-finally` 문으로 계속 커넥션을 열고 매핑하고 정리하고 이러한 로직이 계속 반복된다.


> **==이러한 문제들을 스프링이 다 해결해준다 !!!==**

## 트랜잭션 추상화

---

기존의 서비스 계층 코드는 JDBC 트랜잭션에 의존적인 코드였다. 하지만 **==트랜잭션을 인터페이스로 추상화시키고 JDBC나 JPA 와 같은 필요한 기술을 구체화시키고 서비스계층은 트랜잭션 인터페이스에 의존적==**이게 만들면 모든 문제가 해결된다 !!

- 우선 예시로 `TxManger` 라는 인터페이스를 생성하고, 트랜잭션에 필요한 `begin()`, `commit()`, `rollback()` 을 선언해준다.

    ```java
    public interface TxManager {
    		begin();
    		commit();
    		rollback();
    }
    ```

- 그리고 서비스 계층을 인터페이스에 의존적이게 연결하고 , 사용할 기술들을 구체화 시켜서 DI를 통해 서비스코드에 주입시키면 된다 !!
- 즉, ==**클라이언트인 서비스 코드는 인터페이스에 의존하고 DI를 사용한 덕분에 OCP원칙**==을 지키게 된다 !!

## 트랜잭션 동기화

---

스프링이 제공하는 트랜잭션 매니저는 2가지 역할을 한다.

- **트랜잭션 추상화**
- **리소스 동기화**

### 리소스 동기화

- ==**트랜잭션을 유지하기 위해서는 트랜잭션의 시작과 끝을 같은 DB 커넥션으로 유지**==해야 한다.
- 이전의 _**순수 자바코드로 트랜잭션을 구현할 때는 커넥션을 파라미터로 넘겨서 사용**_했다.
- 스프링을 사용하면 ==**트랜잭션 동기화 매니저를 통해 커넥션을 유지**==할 수 있다.

### 트랜잭션 매니저와 트랜잭션 동기화 매니저

- 스프링은 **트랜잭션 동기화 매니저**를 제공한다. 이것은 `쓰레드로컬(ThreadLocal)`을 사용해서 **커넥션을 동기화** 해준다.
- 트랜잭션 매니저는 내부에서 트랜잭션 동기화 매니저를 사용
- 쓰레드로컬을 사용하기 때문에 멀티쓰레드 상황에 안전하게 커넥션을 동기화 할 수 있다.

    ![[Untitled 3 24.png|Untitled 3 24.png]]

- 동작 방식을 정리하면 아래와 같다.
    1. 트랜잭션을 시작하면 _**트랜잭션 매니저 내부에서 데이터소스를 통해 커넥션을 만들고 트랜잭션을 시작**_한다.
    2. 트랜잭션 매니저는 _**트랜잭션이 시작된 커넥션을 트랜잭션 동기화 매니저에 보관**_한다.
    3. 데이터접근 계층인 _**레포지토리는 트랜잭션 동기화 매니저에 보관된 커넥션을 사용**_한다.
    4. _**트랜잭션이 종료되면 트랜잭션 동기화 매니저에 보관되어있는 커넥션을 통해 트랜잭션을 종료하고 커넥션도 닫는다.**_

## 트랜잭션 문제 해결 - 트랜잭션 매니저 1

---

스프링이 제공하는 트랜잭션 매니저를 사용하려면 `DataSourceUtils` 를 사용하면된다.

### Repository

- 기존의 `MemberRepository` 와 비교를 하면 ==**커넥션을 획득하고 닫을 때 JdbcUtils 를 사용하지 않고 DataSourceUtils 를 사용**==하는 점이다.

    - 기존의 커넥션을 파라미터로 넘겨서 커넥션을 유지했던 코드는 지웠다.
    - 우선 `getConnection()` 메서드를 보면 `DataSourceUtils.getConncetion()` 을 호출하여 DataSource 객체를 넘겨주었다.
    - `close()` 메서드에서 커넥션을 닫는 로직도 `DataSourceUtils.releaseConnection()` 을 호출하여 닫아줬다.

    ```java
    @Slf4j
    public class MemberRepositoryV3 {

        private final DataSource dataSource;

        public MemberRepositoryV3(DataSource dataSource) {
            this.dataSource = dataSource;
        }

        public Member save(Member member) throws SQLException {
    				 // 생략
        }

        public Member findById(String memberId) throws SQLException {
     				 // 생략
        }

        public void update(String memberId, int money) throws SQLException {
    				 // 생략
        }

        public void delete(String memberId) throws SQLException {
    				 // 생략
        }

        private void close(Connection con, Statement statement, ResultSet resultSet) { // Statement 는 일반 SQL 을 담는다.
            JdbcUtils.closeResultSet(resultSet);
            JdbcUtils.closeStatement(statement);
            // 주의! 트랜잭션 동기화를 사용하려면 DataSourceUtils 를 사용해야 한다
            DataSourceUtils.releaseConnection(con, dataSource);
        }

        private Connection getConnection() throws SQLException {
            // 주의 ! 트랜잭션 동기화를 사용하려면 DataSourceUtils 를 사용해야 한다.
            Connection con = DataSourceUtils.getConnection(dataSource);
            log.info("get connection={}, class={}", con, con.getClass());
            return con;
        }
    }
    ```


### **DataSourceUtils.getConnection()**

- ==**트랜잭션 동기화 매니저가 관리하는 커넥션이 있으면 해당 커넥션을 반환**==한다.
- 트랜잭션 도익화 매니저가 관리하는 **커넥션이 없는 경우 새로운 커넥션을 생성해서 반환**한다.

### **DataSourceUtils.releaseConnection()**

- 기존의 close() 메서드를 통해 커넥션을 닫으면 커넥션이 유지되지 않는 문제가 발생했다.
- `DataSourceUtils` 의 `releaseConnection()` 을 사용하면 커넥션을 바로 닫지 않는다.
    - ==**트랜잭션을 사용하기 위해 동기화된 커넥션은 커넥션을 닫지 않고 그대로 유지**==해준다.
    - 트랜잭션 동기화 매니저가 **관리하는 커넥션이 없는 경우 해당 커넥션을 닫는다.**

### Service

- 트랜잭션 추상화를 위해 스프링이 제공하는 `PlatformTranscationManger` 인터페이스를 주입받는다
    - 추후에 **JDBC를 쓰면 DataSourceTransactionManger 구현체를 주입**받으면 된다(테스트때 사용)
- 트랜잭션 시작하는 코드가 `PlatformTransactionManager` 인터페이스의 `getTransaction()` 을 호출하여 트랜잭션을 시작한다.
    - `getTransaction()` 메서드는 파라미터로 트랜잭션의 속성을 지정할 수 있다. 우선 지금은 기본인 `DefaultTransactionDefinition` 을 사용한다.
- 커밋도 `PlatformTransactionManager` 인터페이스가 제공하는 `commit()` 을 사용, 롤백도 `rollback()` 사용한다.
- `release()` 코드가 없어졌다. 왜냐하면 **트랜잭션 매니저가 릴리즈를 알아서 해준다**.

    ```java
    @Slf4j
    @RequiredArgsConstructor
    public class MemberServiceV3_1 {

        private final PlatformTransactionManager transactionManager;
        private final MemberRepositoryV3 memberRepository;

        public void accountTransfer(String fromId, String toId, int money) throws SQLException {
            // 트랜잭션 시작
            TransactionStatus status = transactionManager.getTransaction(new DefaultTransactionDefinition());// 트랜잭션의 속성을 정할 수 있다.

            try {
                bizLogic(fromId, toId, money);
                transactionManager.commit(status);
            } catch (Exception e) {
                // 예외발생시 해당구문으로 이동
                transactionManager.rollback(status);
                throw new IllegalStateException(e);
            }
        }

        private void bizLogic(String fromId, String toId, int money) throws SQLException {
            // 비즈니스 로직
            Member fromMember = memberRepository.findById(fromId);
            Member toMember = memberRepository.findById(toId);

            memberRepository.update(fromId, fromMember.getMoney() - money);
            validation(toMember);
            memberRepository.update(toId, toMember.getMoney() + money);
        }

        private static void validation(Member toMember) {
            if (toMember.getMemberId().equals("ex")) {
                throw new IllegalStateException("이체중 예외 발생");
            }
        }
    }
    ```


### ServiceTest

- Service 코드도 달라진점은 새로운 Repository 와 Service 를 주입받고, 데이터를 세팅하는 코드가 달라진다.
- `Service` 코드에서 ==**PlatformTransactionManager를 의존하기 때문에 여기서 구현체를 정해줘야 한다.**==

    - JDBC 는 `DataSourceTranscactionManger` 클래스가 구현체이다
    - 트랜잭션 매니저는 데이터소스를 통해 커넥션을 생성하므로 `DataSource` 가 필요 !

    ```java

            Member findMemberEx = memberRepository.findById(memberEx.getMemberId());

            assertThat(findMemberEx.getMoney()).isEqualTo(10000);
        }
    }
    ```

## 트랜잭션 문제 해결 - 트랜잭션 템플릿

---

트랜잭션을 사용하다 보면 비슷한 패턴이 반복되는 것을 알 수 있다. 이럴 때 ==**템플릿 콜백 패턴**==을 활용하면 이런 반복 문제를 깔끔하게 해결할 수 있다.

### 트랜잭션 템플릿

- **템플릿 콜백 패턴**을 적용할려면 템플릿을 제공하는 클래스가 필요하다, 스프링은 `TransactionTemplate` 라는 템플릿 클래스를 제공한다.

    - `excute()` : 응답 값이 있을 때 사용
    - `executeWithoutResult()` : 응답 값이 없을 때 사용

    ```java
    public class TransactionTemplate {
       private PlatformTransactionManager transactionManager;

       public <T> T execute(TransactionCallback<T> action){..}
       void executeWithoutResult(Consumer<TransactionStatus> action){..}
    }
    ```


### Service 코드 적용

- 트랜잭션 템플릿을 쓸려면 생성자 내부에서 별도의 로직이 필요하기 때문에 `@RequiredArgsConstructor` 를 지웠다.
- _**트랜잭션 템플릿은 트랜잭션 매니저가 필요**_하기 때문에 생성자 ==**파라미터로 트랜잭션 매니저를 받고, 내부에서는 트랜잭션 템플릿을 생성하면서 파라미터로 트랜잭션 매니저를 넘겨준다.**==
- 그리고 실제 로직인 `accountTransfer()` 는 반환값이 없으므로 트랜잭션 템플릿의 `executeWithoutResult()` 를 호출한다.
    - 람다식으로 `status(TransactionStatus)`를 넘기고 try-catch로 감싸서 `bizLogic()`을 호출한다. → **==bizLogin() 에서 SQLException 을 던지면 람다식 내부에서 예외를 밖으로 못던지기 때문에 try-catch 로 묶었다.==**
- 트랜잭션 템플릿의 기본동작은 다음과 같다.

    - `bizLogic()` 이 정상 수행하면 커밋
    - **언체크 예외가 발생하면 롤백**, 그 외의 경우 커밋한다.(체크예외의 경우 커밋한다)

    ```java
    @Slf4j
    public class MemberServiceV3_2 {

    //    private final PlatformTransactionManager transactionManager;
        private final TransactionTemplate txTemplate;
        private final MemberRepositoryV3 memberRepository;

        public MemberServiceV3_2(PlatformTransactionManager transactionManager, MemberRepositoryV3 memberRepository) {
            // TransactionTemplate 은 TransactionManager 가 필요하다.
            this.txTemplate = new TransactionTemplate(transactionManager);
            this.memberRepository = memberRepository;
        }

        public void accountTransfer(String fromId, String toId, int money) throws SQLException {
            txTemplate.executeWithoutResult((status) -> {
                try {
                    bizLogic(fromId, toId, money);
                } catch (SQLException e) {
                    throw new IllegalStateException(e);
                }
            });
        }

        private void bizLogic(String fromId, String toId, int money) throws SQLException {
    				// 이전과 동일
        }

        private static void validation(Member toMember) {
    				// 이전과 동일
        }
    }
    ```


### 정리

- `TransactionTemplate` 를 통해 반복적인 코드는 줄였다.
- 하지만 서비스 로직에 비즈니스 로직과 트랜잭션을 처리하는 로직이 함께 포함되어 있다.
    - 만약 트랜잭션을 바꾸거나 쓰지않으면 서비스코드를 수정해야 한다 !!(OCP 위반)
- **==애플리케이션을 구성하는 로직을 핵심 기능과 부가 기능으로 구분하면 비즈니스 로직은 핵심 기능이고, 트랜잭션은 부가 기능이다.==**
- 관심사를 나눠야 코드를 유지보수하기 쉽다 !

## 트랜잭션 문제 해결 - 트랜잭션 AOP 이해

---

서비스 로직에 비즈니스 로직과 트랜잭션 로직과 함께 있기 때문에 관심사를 구분할 필요가 있었다. ==**스프링 AOP를 통해 프록시를 사용하면 서비스로직에 비즈니스 로직만 둘 수 있다.**==

![[Untitled 4 21.png|Untitled 4 21.png]]

- 프록시를 도입하면 ==**클라이언트가 서비스코드를 호출할 때 프록시를 호출해서 프록시가 트랜잭션을 진행하고 실제 서비스를 호출**==하게 해준다.
- 이렇게 되면 ==**비즈니스 로직에는 핵심 기능**==만 들어가있고, ==**트랜잭션 프록시에 부가기능인 트랜잭션 기능**==이 있으므로 완벽하게 관심사가 분리되었다.

### 스프링이 제공하는 트랜잭션 AOP

- **스프링이 제공하는 AOP 기능**을 사용할려면 `@Aspect` , `@Advice` , `@Pointcut` 을 사용하여 트랜잭션을 작성할 수 있다.
- 하지만 **트랜잭션은 매우 중요한 기능**이기 때문에 스프링에서 `@Transacionial` 어노테이션을 통해 트랜잭션 AOP 기능을 제공한다.
    - 참고로 스프링 부트를 사용하면 트랜잭션 AOP를 처리하기 위해 필요한 스프링 빈들도 자동으로 등록해준다.

### Service 코드 적용

- 트랜잭션을 적용할 메서드 위에 `@Transactional` 어노테이션을 사용하면 ==**트랜잭션 프록시에서 트랜잭션이 처리**==된다.

### Service 테스트 코드 적용

- ==**스프링 AOP를 적용할려면 스프링 컨테이너가 필요**==하다
    - 이전 테스트 코드에서는 스프링을 사용하지 않았지만 이번 테스트에서는 스프링 컨테이너를 사용하기 위해 `@SpringbootTest` 어노테이션을 클래스 상단에 붙혀준다.
- `@TestConfiguration` 은 ==**테스트 안에서 내부설정 클래스를 만들어서 스프링부트가 자동으로 만들어주는 빈들에 추가로 빈에 등록하고 테스트를 수행**==할 수 있다.
- `@Autowired` 어노테이션을 통해 스프링 컨테이너에 등록된 빈을 가져온다.

## 스프링 부트의 자동 리소스 등록

---

스프링부트 이전에 개발자라면 `DataSource` 나 `TransactionManager` 를 직접 스프링 빈에 등록해서 사용했을 것이다. 하지만 스프링부트의 등장 이후로는 모든것이 자동화 되었다.

### DataSource - 자동등록

- 스프링 부트는 `DataSourcde`를 스프링 빈에 `dataSource` 라는 이름으로 등록한다.
    - 개발자가 직접 DataSource 를 등록하면 스프링부트가 자동으로 등록하지 않는다.
- 스프링 부트는 `application.properties` 에 있는 속성을 사용하여 DataSource 를 생성한다.

    ```xml
    spring.datasource.url=jdbc:h2:tcp://localhost/~/test
    spring.datasource.username=sa
    spring.datasource.password=
    ```

- 스프링 부트가 기본으로 생성하는 데이터소스는 `HikariDataSource` 이다.
- `spring.datasource.url` 속성이 없으면 내장 DB를 생성하려고 시도한다.

### TransactionManager - 자동 등록

- 스프리 부트는 `PlatformTransactionManager` 를 스프링 빈에 등록할 때 `transactionManager` 라는 이름으로 등록한다.
- 실제 구현체는 등록된 라이브러리를 보고 판단한다

    - **JDBC** : `DataSourceTransactionManger`
    - **JPA** : `JpaTransactionManager`
