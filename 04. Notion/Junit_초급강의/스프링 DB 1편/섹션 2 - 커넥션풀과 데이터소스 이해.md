  

# 커넥션 풀 이해

---

커넥션 풀을 이해하기 전에 데이터베이스 커넥션을 획득하는 흐름을 알아야 한다.

![[Untitled 62.png|Untitled 62.png]]

1. 클라이언트가 **서버로 요청**을 한다.
2. 서버 내부에 애플리케이션 로직에서 ==**등록된 DB 드라이버 목록에 커넥션을 조회**==한다.
3. DB 드라이버가 ==**실제 DB와 TCP/IP 커넥션을 연결**==한다.
4. 연결이 되면 ==**ID 와 PW 같은 정보들을 DB 에 전달**==한다.
5. DB 내부에서 ==**인증이 되고 DB 내부에 세션이 생성**==된다.
6. 애플리케이션 **==서버로 커넥션 생성이 완료되었다는 응답==**을 보낸다(**TCP/IP 의 3 way handkshake**)
7. ==**DB 드라이버는 커낵션 객체를 클라이언트에 반환**==  
    

> 매번 사용자 요청이 올 때마다 _==**SQL 을 실행시키는 시간 + 커넥션을 획득하는 시간까지 추가**==_되기 때문에 응답속도에 영향을 준다. 이 문제를 해결하기 위한 방법이 `**커넥션 풀**` 이다.

- ==**애플리케이션을 시작하는 시점에 커넥션 풀에 필요한 만큼 커넥션을 미리 확보해서 풀에 보관**==한다.
- 커넥션 갯수는 서비스의 특징과 서버 스펙에 따라 다르지만 **보통 10 개**이다.
    
    ![[Untitled 1 46.png|Untitled 1 46.png]]
    
- 그렇게 되면 **커넥션 풀과 DB 는 TCP/IP 연결 상태**이기 때문에 언제든지 _==**즉시 SQL을 DB에 전달**==_할 수 있다.
    
    ![[Untitled 2 32.png|Untitled 2 32.png]]
    
- 클라이언트로부터 요청이 오면 **_==애플리케이션 로직은 DB 드라이버를 통해 커넥션을 획득하는게 아닌 커넥션 풀을 통해서 커넥션을 획득==_**하게 된다.
    
    ![[Untitled 3 22.png|Untitled 3 22.png]]
    
- 이렇게 되면 커넥션을 획득하기 위한 DB 내부에 인증하는 시간이 줄고 즉시 SQL 요청시 그 결과를 받아서 처리할 수 있다.
- 주의할 점은 _==**사용하고 난 후에 커넥션은 종료가 아닌 다시 커넥션 풀에 반환하는 것**==_이다.
    
    ![[Untitled 4 19.png|Untitled 4 19.png]]
    

### 정리

- 적절한 커넥션 풀 숫자는 서비스의 특징과 서버 스펙등에 따라 다르기 때문에 ==**성능 테스트를 통해서 정해야 한다.**==
- 커넥션 풀은 서버당 최대 커넥션 수를 제한할 수 있다.
- 커넥션 풀을 사용함으로써 얻는 이점이 매우 크기 때문에 ==**실무에서는 항상 기본으로 사용**==한다.
- 커넥션 풀은 직접 구현할 수도 있지만, 오픈소스 커넥션 풀이 많기 때문에 오픈소스를 사용하는 것이 좋다.
    - 대표적인 오픈소스 커넥션 풀은 `commons-dbcp2` , `tomcat-jdbc pool` , `HikariCP` 등이 있다.
    - 실무에서는 레거스 프로젝트가 아닌 이상 대부분 `HicariCP`를 사용한다.

# DataSource 이해

---

커넥션을 획득하는 방법은 `DriverManager` 를 통해 **직접 DB 드라이버를 통해 커넥션을 얻는 방법**과, `커넥션 풀` 이란 기술을 사용하여 커넥션을 얻는 방법이 있다. 만약 기존의 `DriverManager`로 커넥션을 얻다가 커넥션 풀을 사용하게 되면 코드의 수정이 일어난다 이를 `DataSource` 를 통해 해결할 수 있다.

- 예를 들어 DriverManager 를 사용하다 HikariCP 같은 커넥션 풀을 사용하게 되면 애플리케이션 코드가 함께 변경되어야 한다. 왜냐하면 의존관계가 `DriverManager` → `HikariCP` 로 변경됐기 때문이다.
    
    ![[Untitled 5 16.png|Untitled 5 16.png]]
    
- 이런 코드 수정을 막아주기 위해 자바에서는 `javax.sql.DataSource` 라는 인터페이스를 제공한다.
- `DataSource` 는 ==**커넥션을 획득하는 방법을 추상화**== 하는 인터페이스이다.
- `DataSource` 는 여러 기능이 있지만 핵심 기능은 커넥션 조회 하나이다.
    
    ```java
    public interface DataSource{
    	Connection getConnection() throws SQLException;
    } // 일부분만 발췌
    ```
    

# DataSource 사용 예제 - HikariCP 오픈소스 사용

---

스프링부트 3.1 이상은 로그를 출력할 때 전체로그를 출력하지 않고 간략한 로그를 출력한다. 그래서 전체로그를 출력하기 위해 src/macin/resource/logback.xml 에 아래와 같은 코드를 추가해주자.

```xml
<configuration>
     <appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">
         <encoder>
             <pattern>%d{HH:mm:ss.SSS} [%thread] %-5level %logger{36} -%kvp-
 %msg%n</pattern>
         </encoder>
     </appender>
     <root level="DEBUG">
         <appender-ref ref="STDOUT" />
     </root>
 </configuration>
```

- `DataSource` 를 통해 커넥션 풀을 세팅해보자. 이 때 `HikariCP` 오픈소스를 사용한다.
    - `HikariCP` 는 `DataSource` 인터페이스의 구현체이다.
- `HikariCP` 는 **스프링부트에서 JDBC 라이브러리를 의존성추가하게 되면 자동으로 가져온다.**
- `URL` , `USERNAME` , `PASSWORD`, `최대 풀 갯수` , `풀 이름` 등등 다양한 세팅을 할 수 있다.
- 커==**넥션을 생성하는 작업은 애플리케이션 실행속도에 영향을 주지 않기 위해 별도의 쓰레드에서 동작**==이 되기 때문에 테스트가 먼저 종료될 수 있다. 그래서 `sleep()` 메서드로 잠시 멈추게 하면 커넥션풀이 생성되는 로그를 확인할 수 있다.
    
    ```java
    @Slf4j
    public class ConnectionTest {
    
    		@Test
        public void dataSourceConnectionPool() throws Exception {
            // 커넥션 풀링
            HikariDataSource dataSource = new HikariDataSource();
            dataSource.setJdbcUrl(URL);
            dataSource.setUsername(USERNAME);
            dataSource.setPassword(PASSWORD);
            dataSource.setMaximumPoolSize(10);
            dataSource.setPoolName("My Connection Pool");
    
            useDataSource(dataSource);
            Thread.sleep(1000);
        }
    
        private void useDataSource(DataSource dataSource) throws SQLException {
            Connection con1 = dataSource.getConnection();
            Connection con2 = dataSource.getConnection();
    
            log.info("connection={}, class={}",con1 ,con1 .getClass());
            log.info("connection={}, class={}",con2 ,con2 .getClass());
        }
    }
    ```
    

# DataSource 적용

---

어플리케이션에 직접 DataSource 를 적용해보자, 이전에 `DriverManager`를 사용할 때는 _==**MemberRepositoryV0 에서 직접 DriverManager 에 의존**==_했다. 만약 `HikariCP` 같은 오픈소스를 사용하게 된다면 `MemberRepositoryV0` 을 수정해야 한다. 그렇기 떄문에 새로운 레포지토리를 생성하여 DataSource 추상화를 직접 해본다.

### MemberRepositoryV1 → DataSource 인터페이스 의존

- 이전 `MemberRepositoryV0` 과 다른점은 **DataSource 를 생성자 주입**을 받는것과 **리소스를 정리하는 close() 메서드**, **커넥션을 획득하기 위한 getConnection() 메서드**가 다르다
- 기존의 `close()` 메서드에서는 try-catch 문에 예외발생시 예외를 던지게 짯지만 ==**SQLException 외에 다른 예외가 터지는 경우도 생각**==해야 한다. 이 떄 `JdbcUtils` 편의 메서드를 사용하면 좀 더 편리하게 리소스를 정리할 수 있다.
- 기존의 `getConnection()` 은 직접 DriverManager 에 연결시켰지만 이제는 _==**주입받은 DataSource 를 사용하여 커넥션을 연결**==_하였다.
    
    ```java
    @Slf4j
    public class MemberRepositoryV1 {
    
        private final DataSource dataSource;
    
        public MemberRepositoryV1(DataSource dataSource) {
            this.dataSource = dataSource;
        }
    
        public Member save(Member member) throws SQLException {
            // 이전과 동일하기 떄문에 생략
        }
    
        public Member findById(String memberId) throws SQLException {
            // 이전과 동일하기 떄문에 생략
        }
    
        public void update(String memberId, int money) throws SQLException {
            // 이전과 동일하기 떄문에 생략
        }
    
        public void delete(String memberId) throws SQLException {
            // 이전과 동일하기 떄문에 생략
        }
    
    
        private void close(Connection con, Statement statement, ResultSet resultSet) { // Statement 는 일반 SQL 을 담는다.
            JdbcUtils.closeResultSet(resultSet);
            JdbcUtils.closeStatement(statement);
            JdbcUtils.closeConnection(con);
        }
    
        private Connection getConnection() throws SQLException {
            Connection con = dataSource.getConnection();
            log.info("get connection={}, class={}",con, con.getClass());
            return con;
        }
    }
    ```
    

### 테스트 코드로 실행해보기

- `@BeforeEach` 로 테스트 시작전 ==**HikariCP 객체를 생성하여 세팅한 후 repository 에 생성자로 초기화**== 시키면 된다. 즉, ==**생성자 주입으로 사용하고 싶은 커넥션획득 방법으로 바꾸면 된다.**==
    
    ```java
    @Slf4j
    class MemberRepositoryV1Test {
    
        MemberRepositoryV1 repository;
    
        @BeforeEach
        void beforeEach() {
            // 기본 DriverManager - 항상 새로운 커넥션 획득
    //        DriverManagerDataSource dataSource = new DriverManagerDataSource(URL,USERNAME,PASSWORD);
    
            // 커넥션 풀링 : HikariCP 사용
            HikariDataSource dataSource = new HikariDataSource();
            dataSource.setJdbcUrl(URL);
            dataSource.setUsername(USERNAME);
            dataSource.setPassword(PASSWORD);
    
            repository = new MemberRepositoryV1(dataSource);
        }
    
        @Test
        void crud() throws SQLException {
    				// 이전과 같기 때문에 생략 ..
    
    				// 커넥션 생성은 애플리케이션 실행속도에 영향을 안주기 위해 별도의 쓰레드로 실행하기 때문에 sleep()을 걸어준다.
            try {
                Thread.sleep(2000);
            } catch (InterruptedException e) {
                throw new RuntimeException(e);
            }
        }
    }
    ```
