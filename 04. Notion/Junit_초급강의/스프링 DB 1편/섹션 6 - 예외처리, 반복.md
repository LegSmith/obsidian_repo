---

==_**서비스 계층은 특정 구현 기술에 의존하지 않고, 순수하게 유지**_==하는 것이 좋다.  
예를 들면 현재 서비스 계층에는 `throws SQLException` 구문이 있다. 이 코드도 **JDBC 에 종속된 예외**기 때문에 수정을 해야 한다.

- **서비스 계층이 순수하게 인터페이스에 의존하게 하고 필요한 기술을 인터페이스를 구현**하면 된다.
- 문제는 **인터페이스의 구현체가 체크 예외를 던지려면, 인터페이스 메서드에 먼저 체크 예외를 던지는 부분이 선언**되어 있어야 한다.
    
    - ==_**이렇게 되면 인터페이스도 특정 기술에 종속되게 된다!!**_==
    
    ```java
    public interface MemberRepositoryEx {
        Member save(Member member) throws SQLException;
        Member findById(String memberId) throws SQLException;
        void update(String memberId, int money) throws SQLException;
        void delete(String memberId) throws SQLException;
    }
    ```
    
- 위 문제를 해결하기 위해서는 체크예외를 언체크예외인 런타임 예외를 사용해야 한다 !!

## 런타임 예외 적용

---

### 언체크 예외를 상속받는 예외 생성

- 생성자로는 `Throwable cause` 를 파라미터로 받는 생성자를 꼭 만들어주자!!
- _==**실제 예외를 만들 때 예외의 원인을 담을 수 있어야 한다 !!**==_ 이 때 `cause` 를 통해 예외와 예외원인을 다 확인할 수 있다.
    
    ```java
    public class MyDbException extends RuntimeException{
    
        public MyDbException() {
        }
    
        public MyDbException(String message) {
            super(message);
        }
    
        public MyDbException(String message, Throwable cause) {
            super(message, cause);
        }
    
        public MyDbException(Throwable cause) {
            super(cause);
        }
    }
    ```
    

### 인터페이스 생성

- 실제 DB에 접근하는 `save()` , `findById()` , `update()` , `delete()` 메서드를 선언한다.
    
    ```java
    public interface MemberRepository {
        Member save(Member member);
    
        Member findById(String memberId);
    
        void update(String memberId, int money);
    
        void delete(String memberId);
    }
    ```
    

### 서비스 생성

- 인터페이스인 `MemberRepository` 를 의존하기 때문에 ==_**실제 구현 기술이 바뀌어도 서비스가 의존하는 대상인 인터페이스는 변경하지 않아도 된다 !!**_==
- 언체크 예외를 던지기 때문에 `thorws` 키워드가 사라져서 **순수하게 서비스 코드만 남게 된다!!**
    
    ```java
    @Slf4j
    public class MemberServiceV4 {
    
        private final MemberRepository memberRepository;
    
        public MemberServiceV4(MemberRepository memberRepository) {
            this.memberRepository = memberRepository;
        }
    
        @Transactional
        public void accountTransfer(String fromId, String toId, int money) {
            bizLogic(fromId, toId, money);
        }
    
        private void bizLogic(String fromId, String toId, int money) {
            // 비즈니스 로직
            Member fromMember = memberRepository.findById(fromId);
            Member toMember = memberRepository.findById(toId);
    
            memberRepository.update(fromId, fromMember.getMoney() - money);
            validation(toMember);
            memberRepository.update(toId, toMember.getMoney() + money);
        }
    
        private static void validation(Member toMember) {
            if (toMember.getMemberId().equals("ex")) {
                throw new IllegalStateException("이체중 예외 발생");
            }
        }
    }
    ```
    

## 데이터 접근 예외 직접 만들기

---

데이터베이스 오류에 따라서 특정 예외는 복구해야 하는 경우가 있다.  
예를 들어 **회원가입시 DB에 이미 같은 ID가 있다면 ID 뒤에 숫자를 붙여서 새로운 ID를 만들어야 한다.**  
이런경우는 ==**데이터베이스는 오류 코드를 JDBC 드라이버로 반환**==하고 JDBC 드라이버는 오류코드를 가진 채로 `SQLException` 을 반환한다. 이 때 ==**레포지토리에서 오류코드를 가지고 새로운 예외를 만들어 서비스계층으로 던지면 된다.**==

![[Untitled 66.png|Untitled 66.png]]

- H2 데이터베이스에서는 ==**키 중복 오류**==에 경우 `23505` 라는 코드를 내보낸다.
    - _**참고로 데이터베이스 마다 오류코드가 다르기 때문에 메뉴얼을 항상 확인하자 !!**_
- ==**서비스 계층에서는 예외 복구를 위해 키 중복 오류를 확인**==해야 한다. 레포지토리는 데이터베이스에서 받은 오류코드를 통해 `SQLException` 을 서비스 계층으로 던지게 된다.
- 이러면 ==**서비스계층에서는 JDBC에 의존적이게 되므로**== 레포지토리에서는 `SQLExcpetion` 을 새로운 예외인 `MyDuplicateKeyException` 으로 변환하여 던져주면 순수한 서비스계층이 유지된다.

### 특정 예외(MyDuplicateKeyException) 추가

- `RuntimeException` 을 상속받아도 되지만 `MyDbException` 을 `MyDbException` 을 상속받았다.
- 해당 예외는 JDBC 같은 특정기술에 의존적인 예외가 아닌 개발자가 직접 만든 예외기 때문에 서비스계층으로 해당예외를 던져도 서비스계층의 순수성을 유지할 수 있다.
    
    ```java
    public class MyDuplicateKeyException extends MyDbException {
        public MyDuplicateKeyException() {
            super();
        }
    
        public MyDuplicateKeyException(String message) {
            super(message);
        }
    
        public MyDuplicateKeyException(String message, Throwable cause) {
            super(message, cause);
        }
    
        public MyDuplicateKeyException(Throwable cause) {
            super(cause);
        }
    }
    ```
    

### 테스트 코드를 통해 확인

- 테스트코드내에서 임의로 `Repotiory` 를 만들어서 직접 DB에 커넥션을 받아와 저장하는 코드를 짯다.
- 우선 DB 에 `키 중복 오류코드인 23505` 가 레포지토리로 반환됐을 때는 위에서 만든 `MyDuplicateKeyException` 을 서비스코드로 던져 준다.
    
    ```java
    
    public class ExTranslatorV1Test {
    		@RequiredArgsConstructor
    		static class Repository {
    		    private final DataSource dataSource;
    		
    		    public Member save(Member member) {
    		        String sql = "INSERT INTO member(member_id, money) VALUES (?,?)";
    		
    		        Connection con = null;
    		        PreparedStatement pstmt = null;
    		
    		        try {
    		            con = dataSource.getConnection();
    		            pstmt = con.prepareStatement(sql);
    		            pstmt.setString(1, member.getMemberId());
    		            pstmt.setInt(2, member.getMoney());
    		            pstmt.executeUpdate();
    		            return member;
    		        } catch (SQLException e) {
    		            // h2 DB
    		            if (e.getErrorCode() == 23505) {
    		                throw new MyDuplicateKeyException(e);
    		            }
    		            throw new MyDbException(e);
    		        } finally {
    		            JdbcUtils.closeStatement(pstmt);
    		            JdbcUtils.closeConnection(con);
    		        }
    		    }
    		}
    		}
    ```
    
- `Repository`를 주입받아 `save()`를 호출한다.
- 그리고 `save()`시 중복된 키를 받았을때 `MyDuplicateKeyException` 이 터지기 때문에 `catch`로 예외를 잡는다.
- `generateNewId()`를 호출하여 새로운 아이디를 만들어 다시 저장하게 한다.
    
    ```java
    
    public class ExTranslatorV1Test {
    		@Slf4j
        @RequiredArgsConstructor
        static class Service {
            private final Repository repository;
    
            public void create(String memberId) {
                try {
                    repository.save(new Member(memberId, 0));
                    log.info("saveId={}", memberId);
                } catch (MyDuplicateKeyException e) {
                    log.info("키 중복, 복구 시도");
                    String retryId = generateNewId(memberId);
                    log.info("retryId={}", retryId);
                    repository.save(new Member(retryId, 0));
                } catch (MyDbException e) {
                    log.info("데이터 접근 계층 예외", e);
                    throw e;
                }
    
            }
    
            private String generateNewId(String memberId) {
                return memberId + new Random().nextInt(10000);
            }
    		}
    }
    ```

### 정리

- DB에 키중복 오류코드를 레포지토리에서 식별후 `SQLException` → `MyDuplicateKeyException` 으로 교체 후 서비스계층으로 넘겨준다. → ==_**런타임 예외(언체크 예외)기 때문에 특정 기술의 예외에 의존하지 않게 된다.**_==

## 스프링 예외 추상화 이해

---

위에 예제로 만든 `DbException` 이나 `MyDuplicateKeyException` 같은 예외들은 스프링에서 이미 제공한다 !!_==**각각의 예외는 특정 기술에 종속적이지 않게 설계**==_되어 있다. 즉, 서비스 계층에서도 스프링이 제공하는 예외를 사용하면 JDBC든 JPA 든 어느 기술로 바꿔도 서비스계층은 순수하게 유지 된다.

### 스프링 데이터 접근 예외 계층

- 스프링이 제공하는 예외의 최상위 예외는 `org.springframework.dao.DataAccessException` 이다.
    - ==_**런타임 예외를 상속받기 때문에 하위 예외들도 전부 런타임예외(언체크예외)이다.**_==
- `DataAccessException` 은 크게 `NonTransient` 와 `Transient` 로 나누어져 있다.
    
    - `Transient` : 일시적이라는 뜻이다. 해당 예외의 하위 예외들은 ==_**동일한 SQL을 다시 시도했을 때 성공할 가능성이 있다.**_==
        - 예를 들어 **쿼리 타임아웃**, **락**과 관련된 오류들이다.
    - `NonTransient` : 일시적이지 않다는 뜻이다. _==**같은 SQL을 그대로 반복해서 실행하면 실패한다.**==_
        - **SQL 문법 오류**, **데이터베이스 제약조건 위배** 등이 있다.
    
    ![[Untitled 3 26.png|Untitled 3 26.png]]
    
    > ==_**참고로 스프링 메뉴얼에는 모든 예외가 정리되어 있지 않기 때문에 직접 소스코드를 열어보는것이 좋다.**_==
    

### 스프링이 제공하는 예외 변환기

- 스프링은 ==**DB에서 발생하는 오류 코드를 스프링이 정의한 예외로 자동으로 변환해주는 변환기를 제공**==한다.
- 아래 코드의 ==_**문제점은 DB교체마다 에러코드를 수정해주고 그에맞는 예외를 던져줘야 하기 때문에 매우 번거롭다 !!**_==
    
    ```java
    @Slf4j
    public class SpringExceptionTranslatorTest {
    
        DataSource dataSource;
    
        @BeforeEach
        void init() {
            dataSource = new DriverManagerDataSource(URL, USERNAME, PASSWORD);
        }
    
        @Test
    		@DisplayName("스프링이 제공하는 예외 변환기를 안썻을 때 테스트")
        void sqlExceptionErrorCode() {
            String sql = "SELECT bad grammer";
    
            try {
                Connection con = dataSource.getConnection();
                PreparedStatement stmt = con.prepareStatement(sql);
                stmt.executeQuery();
            } catch (SQLException e) {
                Assertions.assertThat(e.getErrorCode()).isEqualTo(42122); // H2 DB
                int errorCode = e.getErrorCode();
                log.info("errorCode={}", errorCode);
                log.info("error", e);
            }
        }
    }
    ```
    
- 스프링이 제공하는 변환기를 사용할려면 `SQLErrorCodeSQLExceptionTranslator` 클래스를 사용하면 된다.
    
    - `SQLErrorCodeSQLExceptionTranslator` 의 인스턴스를 통해 `translate()` 메서드로 해당 예외를 가져온다.
        - 첫번째 파라미터는 `설명`을 넣어준다.
        - 두번째 파라미터는 실행할 `SQL`
        - 세번째 파라미터는 예외가 발생한 `SQLException` 을 넣어준다.
    - `translate()` 메서드가 _==**적절한 스프링 데이터 접근 계층의 예외로 변환하여 반환해준다.**==_
    - 해당 예제는 SQL 문법이 잘못됐으므로 `BadSqlGrammarException` 예외를 반환해준다.
    - `BadSqlGrammarException` 예외는 최상위 타입인 `DataAccessException` 을 상속받아서 만들어진다.
    
    ```java
    @Slf4j
    public class SpringExceptionTranslatorTest {
    
        DataSource dataSource;
    
        @BeforeEach
        void init() {
            dataSource = new DriverManagerDataSource(URL, USERNAME, PASSWORD);
        }
    
    		@Test
        @DisplayName("스프링이 제공하는 변환기 사용")
        void exceptionTranslator() {
            String sql = "select bad grammer";
    
            try {
                Connection con = dataSource.getConnection();
                PreparedStatement stmt = con.prepareStatement(sql);
                stmt.executeQuery();
            } catch (SQLException e) {
                SQLErrorCodeSQLExceptionTranslator exTranslator = new SQLErrorCodeSQLExceptionTranslator(dataSource);
                DataAccessException resultEx = exTranslator.translate("myTask", sql, e);
                log.info("resultEx", resultEx);
                assertThat(resultEx.getClass()).isEqualTo(BadSqlGrammarException.class);
            }
        }
    }
    ```
    
- **각각의 DB 마다 SQL ErrorCode 는 다르다**. 스프링은 각각의 DB를 고려해서 어떻게 `ErrorCode`를 변환할까 ? → `org.springframework.jdbc.support.sql-error-codes.xml` 을 통해서 반환한다.

### 정리

- ==**스프링은 데이터 접근 계층(Repository)에 대한 일관된 예외 추상화를 제공**==한다.
- 스프링이 제공하는 ==**예외 변환기를 통해 SQLException 의 ErrorCode 에 맞는 적절한 스프링데이터 접근 예외로 변환**==해준다.
- 만약 서비스나 컨트롤러 계층에서 ==**예외처리가 필요하면 특정 기술에 종속적인 SQLException 이 아닌 스프링이 제공하는 데이터 접근 예외를 사용**==하면 된다.
- _==**물론 스프링이 제공하는 예외를 사용하기 때문에 스프링에 대한 기술 종속성은 발생**==_한다.
    - 예외를 모두 직접 정의하고 예외 변환도 하면되지만 실용적인 방법은 아니다.

## 스프링 예외 추상화 적용

---

- 기존에는 DB에서 반환되는 특정 `ErrorCode`마다 _**예외를 직접 만들어서 사용했을 때는 DB 교체시 마다 ErrorCode를 수정**_해줘야 했지만 ==**스프링이 제공하는 변환기를 사용하면 코드가 일관성있게 변한다.**==
- 변환기인 `SQLExceptionTranslator` 클래스를 생성자 주입을 받고 파라미터로 `dataSource`를 넘겨준다.
    - `dataSource` 에는 **DB의 정보**들이 담겨있기 때문에 필요
- 실제 예외를 던지는 catch 문에서 `SQLExceptionTranslator` 참조변수를 통해 `translate()`를 호출하여 변환해주면 된다.
    
    - 반환타입이 `DataAccessException` 이기 때문에 throw 바로 뒤에 써줘도 된다.
    
    ```java
    @Slf4j

        private final DataSource dataSource;

        public MemberRepositoryV4_2(DataSource dataSource) {
            this.dataSource = dataSource;
            this.exTranslator = new SQLErrorCodeSQLExceptionTranslator(dataSource); 
        }
    
        @Override
        public Member save(Member member) {
            String sql = "INSERT INTO member(member_id, money) values(?, ?)";
    
    	      // 생략 ..
            try {
    	      // 생략 ..
            } catch (SQLException e) {
                throw exTranslator.translate("saveError", sql, e);
            } finally {
                close(con, pstmt, null);
            }
        }
    
        @Override
        public Member findById(String memberId) {
            // 위와동일
        }
    
        @Override
        public void update(String memberId, int money) {
            // 위와동일
        }
    
        @Override
        public void delete(String memberId) {
                    // 위와동일
        }
    
        private void close(Connection con, Statement statement, ResultSet resultSet) { // Statement 는 일반 SQL 을 담는다.
            JdbcUtils.closeResultSet(resultSet);
            JdbcUtils.closeStatement(statement);
            // 주의! 트랜잭션 동기화를 사용하려면 DataSourceUtils 를 사용해야 한다
            DataSourceUtils.releaseConnection(con, dataSource);
        }
    
        private Connection getConnection() throws SQLException {
            // 주의 ! 트랜잭션 동기화를 사용하려면 DataSourceUtils 를 사용해야 한다.
            Connection con = DataSourceUtils.getConnection(dataSource);
            log.info("get connection={}, class={}", con, con.getClass());
            return con;
        }
    }
    ```

## JDBC 반복 문제 해결 - JdbcTemplate

---

이전에는 스프링 예외 추상화 덕에 서비스 계층의 순수함을 유지시켰다. 하지만 ==_**레포지토리에서는 JDBC를 사용하기 때문에 반복적인 코드가 많아졌다.**_==

- 커넥션 조회, 커넥션 동기화
- PreparedStatement 생성 및 파라미터 바인딩
- 쿼리 실행
- 결과 바인딩
- 예외 발생 시 스프링 예외 변환기 실행
- 리소스 종료

### 템플릿 콜백 패턴

- 이런 ==**반복적인 코드를 효과적으로 제거해주는 방법이 템플릿 콜백 패턴**==이다.
- 스프링에서는 `JdbcTemplate` 이라는 템플릿을 제공한다.
- `JdbcTemplate` 은 ==**JDBC로 개발할 때 발생하는 반복을 대부분 해결**==해준다.
    - ==_**트랜잭션을 위한 커넥션 동기화는 물론이고 스프링 예외 변환기도 자동으로 실행**_==해준다.
- 쿼리 데이터가 변하는 경우에는 `update()` 를 호출해준다.
    - 파라미터로는 sql 과 파라미터 바인딩이 되는 변수를 넘겨준다.
- 단건 쿼리를 조회할 경우 `queryForObject()`를 호출한다.
    - sql 과 쿼리 결과를 어떻게 만들지 매핑정보를 넘겨준다.
    - 반환타입을 가지는 RowMapper 클래스를 가진다.

```java
@Slf4j
public class MemberRepositoryV5 implements MemberRepository {

    private final JdbcTemplate template;

    public MemberRepositoryV5(DataSource dataSource) {
        this.template = new JdbcTemplate(dataSource);
    }

    @Override
    public Member save(Member member) {
        String sql = "INSERT INTO member(member_id, money) values(?, ?)";
        template.update(sql, member.getMemberId(), member.getMoney());
        return member;
    }

    @Override
    public Member findById(String memberId) {
        String sql = "SELECT * FROM member WHERE member_id = ?";
        return template.queryForObject(sql, memberRowMapper(), memberId);//한건 조회는 queryForObject()
    }

    @Override
    public void update(String memberId, int money) {
        String sql = "UPDATE member SET money=? WHERE member_id=?";
        template.update(sql, money, memberId);
    }

    @Override
    public void delete(String memberId) {
        String sql = "DELETE FROM member WHERE member_id=?";
        template.update(sql, memberId);
    }

    private RowMapper<Member> memberRowMapper() {
        return (rs, rowNum) -> {
            Member member = new Member();
            member.setMemberId(rs.getString("member_id"));
            member.setMoney(rs.getInt("money"));
            return member;
        };
    }
}
```
