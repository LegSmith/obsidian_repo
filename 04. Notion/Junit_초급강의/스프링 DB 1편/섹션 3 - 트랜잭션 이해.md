## 트랜잭션 - 개념 이해

---

데이터를 저장할 때 파일에 저장하지 않고 데이터베이스에 저장하는 가장 큰 이유는 **데이터베이스의 트랜잭션** 때문이다.  
트랜잭션을 직역하면 거래인데, 쉽게 애기해서 ==_**데이터베이스에서 트랜잭션은 하나의 거래를 안전하게 처리하도록 보장**_==해주는 것이다.

- 예를 들어 5000원 계좌이체를 한다고 가정해보자.
    - 보내는 계좌에서 5000이 감소하고 받는 계좌에서 5000원이 증가한다. 하지만 5000원이 감소하고 중간에 오류가 떠서 상대계좌로 5000원이 증가하지 않았다고 해보자. 그렇게 되면 보내는 계좌만 돈이 감소하는 심각한 문제가 발생한다.
- 데이터베이스의 트랜잭션은 이런 거래 1개를 보장해준다. ==**만약 중간에 오류가 난다면 모든작업을 취소**==한다. 즉, 작업이 전부 성공해야 정상적으로 저장이되고 아니면 실패하게 된다.
- 모든 작업이 성공해서 ==**DB 에 정상 반영하는 것을 커밋(Commit)**== 이라 하고, 작업 중 하날도 실패해서 ==**거래 이전으로 되돌리는 것을 롤백(Rollback)**==이라 한다.

### 트랜잭션 ACID

- `원자성(Atomicity)` : 트랜잭션 내에서 실행한 작업들은 마치 ==**하나의 작업인 것처럼 모두 성공 하거나 모두 실패**==해야 한다.
- `일관성(Consistency)` : 모든 트랜잭션은 ==**일관성있는 데이터베이스 상태를 유지**==해야 한다.
- `격리성(Isolation)` : ==**동시에 실행되는 트랜잭션들이 서로에게 영향을 미치지 않도록 격리**==한다.
- `지속성(Durability)` : 트랜잭션을 ==**성공적으로 끝내면 그 결과가 항상 기록**==되어야 한다.
    
    **트랜잭션은 원자성, 일관성, 지속성을 보장**한다. 문제는 ==**격리성**==인데 **트랜잭션 간에 격리성을 완벽히 보장할려면 트랜잭션을 거의 순서대로 실행**해야 한다. 이렇게 되면 동시 처리 성능이 매우 나빠진다. 이런 문제로 인해 ANSI 표준은 _==**트랜잭션의 격리 수준을 4단계로 나누어 정의**==_했다.
    
    ### 트랜잭션 격리 수준 - Isolation level(단계가 높을수록 성능이 느려진다.)
    
    1. READ UNCOMMITED(커밋되지 않은 읽기)
    2. READ COMMITED(커밋된 읽기) → 주로 사용
    3. REPEATABLE READ(반복 가능한 읽기) → 주로 사용
    4. SERIALIZABLE(직렬화 가능)
    

## DB 연결 구조와 DB 세션

---

- 사용자는 WAS나 DB 접근 툴 같은 클라이언트를 사용해서 DB 서버에 접근할 수 있다.
- 클라이언트는 DB 서버에 연결을 요청하고 커넥션을 맺게 된다. **커넥션을 맺는 시점에 DB 내부에서 DB 세션이 생성**된다.
- 해당 ==**커넥션을 통한 모든 요청은 이 DB 세션을 통해서 실행**==하게 된다.
    - 즉, _==**개발자가 클라이언트를 통해 SQL을 전달하면 커넥션에 연결된 DB 세션이 SQL을 실행**==_한다.
- DB 세션은 **트랜잭션을 시작하고, 커밋 또는 로백을 통해 트랜잭션을 종료**한다.
- 사용자가 커넥션을 닫거나, 또는 DBA가 세션을 강제로 종료하면 세션은 종료된다.

## 트랜잭션 DB 예제 - 자동커밋, 수동커밋

---

### 자동 커밋

- 자동커밋이란 `commit` 명령어를 실행하지 않아도 ==**SQL 실행시 자동으로 커밋을 호출하여 실행**==시키는 것을 말한다.
- 편리한 기능이지만 **트랜잭션 기능을 제대로 사용할려면 수동커밋으로 바꿔야 한다.**
    
    - 보통 `autocommit` 이 기본값이다.
    
    ```sql
    set autocommit true;
    INSERT INTO member(member_id, money) VALUES ('data1',10000); // 자동커밋
    INSERT INTO member(member_id, money) VALUES ('data2',10000); // 자동커밋
    ```
    

### 수동 커밋

- 수동 커밋모드로 설정하는 것을 트랜잭션을 시작한다고 표현할 수 있다.
- 수동 커밋으로 설정하면 반드시 `commit` , `rollback` 을 호출해야 한다.
    
    ```sql
    set autocommit false;
    INSERT INTO member(member_id, money) VALUES ('data1',10000); 
    INSERT INTO member(member_id, money) VALUES ('data2',10000);
    commit;
    ```

## DB 락 - 개념 이해

---

세션1 이 데이터를 수정하는 동안 커밋을 수행하지 않았는데 세션2가 똑같은 데이터를 수정하게 되면 트랜잭션의 원자성이 깨지게 된다. 이 문제를 방지하기 위해 _==**세션이 트랜잭션을 시작하고 데이터를 수정하는 동안 다른 세션이 접근못하게 잠궈야 한다.**==_

- 데이터베이스는 이러한 문제를 해결학 위해 `락(Lock)`이라는 개념을 제공한다.

  

- 세션1이 `memberA` 의 **데이터를 수정하기 위해 트랜잭션을 시작**한다.
- **데이터를 수정하기 위해 우선 락을 획득**한다.
- 락을 획득했으므로 `UPDATE` 쿼리를 실행하여 데이터를 수정한다.
    
    ![[Untitled 6 13.png|Untitled 6 13.png]]
    
- 그리고 세션2가 `memberA` 를 수정하기 위해 트랜잭션을 시작한다.
- _==**데이터를 수정할려면 락을 획득해야 하므로 세션1이 끝날때까지 대기를 한다.**==_
    
    ![[Untitled 7 9.png|Untitled 7 9.png]]
    
    - 참고로 락을 획득하기 위해 무한정 대기하는것은 아니다. **락 대기 시간을 넘어가면 락 타임아웃 오류가 발생**한다(락 대기 시간 설정가능)
- 데이터를 수정하고 실제 DB 에 반영하기 위해 세션1이 `commit` 을 수행한다. _==**트랜잭션이 종료되었으므로 락도 반납**==_한다.
    
    ![[Untitled 8 6.png|Untitled 8 6.png]]
    
- 세션1이 락을 반납했기 떄문에 대기하고 있던 세션2가 락을 획득하여 memberA 데이터를 수정한다.
    
    ![[Untitled 9 5.png|Untitled 9 5.png]]
    
- 세션2가 `commit`을 하고 트랜잭션이 종료됐기 때문에 락을 반납한다.
    
    ![[Untitled 10 4.png|Untitled 10 4.png]]
    

### 정리

- 즉, **여러 세션이 데이터를 동시에 수정할려고 할 때 원자성이 깨지므로** 데이터베이스 `락(Lock)`을 통해 원자성을 지킬 수 있다.

## DB 락 - 조회

---

보통 조회할 때는 락(Lock)없이 여러 세션이 동시에 데이터를 조회할 수 있다. 하지만 _**==조회를 한 이후에 꼭 데이터를 수정해야하 하는 작업이 필요한 경우 조회할 때 락을 획득하여 데이터 수정이 가능==**_하다. 이때는 `select for update` 구문을 사용하면 된다.

- 우선 세션1 , 세션2를 준비해두고 **기본데이터를 입력**하여 DB 를 통일 시키자.
    
    ```sql
    set autocommit true;
    delete from member;
    insert into member(member_id, money) values ('memberA',10000);
    ```
    
- 먼저 세션 1에서 `select for update` 구문을 사용해보자.
    
    ```sql
    set autocommit false;
    select * from member where member_id='memberA' for update;
    ```
    
- 그리고 세션 2에서 세션1이 조회한 `memberA` 를 수정할려고 하면 대기를 하다가 락 타임아웃 시간이 지나면 오류가 뜨게 된다.
    
    ![[Untitled 16 2.png|Untitled 16 2.png]]
    
- 세션1이 commit 을 해서 트랜잭션을 종료해야 세션2가 memberA 의 데이터를 수정할 수 있다.

### 정리

- 트랜잭션과 락은 DB 마다 동작하는 방식이 조금씩 다르기 떄문에 해당 데이터베이스의 메뉴얼을 확인하여 테스트를 한 이후에 실제 어플리케이션에 적용하는게 좋다 !!

## 트랜잭션 적용 1

---

계좌이체를 예시로 autocommit 이 자동으로 설정되어있다는 전제하에 실습을 한다.

- A 가 B 에게 돈을 계좌이체하면 우선 ==**A 계좌잔액 금액감소 → B 계좌잔액 금액감소**==가 이루어진다.
- 하지만 **트랜잭션이 없기 때문에 중간에 체크하는 로직**을 넣어두고 `memberId` 가 `EX` 일 때 예외가 터지도록 코딩을 해둔다.
    
    ```java
    @RequiredArgsConstructor
    public class MemberServiceV1 {
    
        private final MemberRepositoryV1 memberRepository;
    
        public void accountTransfer(String fromId, String toId, int money) throws SQLException {
            Member fromMember = memberRepository.findById(fromId);
            Member toMember = memberRepository.findById(toId);
    
            memberRepository.update(fromId, fromMember.getMoney() - money);
            validation(toMember);
            memberRepository.update(toId, toMember.getMoney() + money);
    
        }
    
        private static void validation(Member toMember) {
            if(toMember.getMemberId().equals("ex")){
                throw new IllegalStateException("이체중 예외 발생");
            }
        }
    }
    ```
    
- 먼저 DB 커넥션을 얻는 로직을 `@BeforeEach` 로 테스트 전 세팅을 해둔다.
- 그리고 정상이체에 경우에는 `memberA` 에 금액이 감소되고 `memberB`에 금액이 감소되기 때문에 _==**DB 조회를 해도 값이 정상적으로 들어가 있다.**==_
    
    ```java
    class MemberServiceV1Test {
    
        public static final String MEMBER_A = "memberA";
        public static final String MEMBER_B = "memberB";
        public static final String MEMBER_EX = "ex";
    
        private MemberRepositoryV1 memberRepository;
        private MemberServiceV1 memberService;
    
        @BeforeEach
        void before(){
            DriverManagerDataSource dataSource = new DriverManagerDataSource(URL, USERNAME, PASSWORD);
            memberRepository = new MemberRepositoryV1(dataSource);
            memberService = new MemberServiceV1(memberRepository);
        }
    
        @Test
        @DisplayName("정상 이체")
        public void accountTransfer() throws Exception {
            //given
            Member memberA = new Member(MEMBER_A, 10000);
            Member memberB = new Member(MEMBER_B, 10000);
            memberRepository.save(memberA);
            memberRepository.save(memberB);
    
            //when
            // memberA 가 memberB 에게 2000원을 계좌이체
            memberService.accountTransfer(memberA.getMemberId(),memberB.getMemberId(),2000);
    
            //then
            Member findMemberA = memberRepository.findById(memberA.getMemberId());
            Member findMemberB = memberRepository.findById(memberB.getMemberId());
            assertThat(findMemberA.getMoney()).isEqualTo(8000);
            assertThat(findMemberB.getMoney()).isEqualTo(12000);
        }
    }
    ```

- 문제는 이체중간에 예외가 터지는 경우이다.

## 트랜잭션 적용 2

---

위 문제를 해결하기 위해서는 트랜잭션을 사용해야 한다. 즉, `autocommit` 을 ==**false(수동)으로 설정해서 트랜잭션을 설정**==해줘야 한다. 그리고 _==**Connection 1개를 이용하여 이체로직을 실행**==_해야 한다.

- 우선 계좌이체는 `findById()` 와 `update()` 를 수행하는데 이 로직은 **Connection 을 파라미터로 받아야한다**. 그래야 ==**autocommit 을 false 로 바꾼 Connection 을 가지고 계좌이체를 실행**==할 수 있다.
- 변경점은 `Connection` 을 파라미터로 받는것과 `getConnection()` 이 사라지고 Connection 을 닫지 않는 부분이다.
    
    - 파라미터로 받는 이유는 **autocommit 이 수동인 커넥션을 가지고 수행**해야 하기 때문
    - `getConnection()` 을 실행하면 **autocommit 이 true 인 커넥션을 연결**시키기 때문
    - `Connection` 을 닫으면 **트랜잭션이 종료**되기 때문
    
    ```java
    /**
     * <h2>JDBC - Connection을 Parameter 로 넘긴다.</h2>
     */
    @Slf4j
    public class MemberRepositoryV2 {
    
        private final DataSource dataSource;
    
        public MemberRepositoryV2(DataSource dataSource) {
            this.dataSource = dataSource;
        }
    
        public Member findById(Connection con, String memberId) throws SQLException {
            String sql = "SELECT * FROM member WHERE member_id = ?";
    
            PreparedStatement pstmt = null;
            ResultSet rs = null;
    
            try {
                pstmt = con.prepareStatement(sql);
    
                pstmt.setString(1, memberId);
                rs = pstmt.executeQuery(); // 데이터변경일 경우엔 executeUpdate() , 데이터를 조회할 때는 executeQuery()
    
                if (rs.next()) {
                    Member member = new Member();
                    member.setMemberId(rs.getString("member_id"));
                    member.setMoney(rs.getInt("money"));
                    return member;
                } else {
                    throw new NoSuchElementException("Member not found memberId = " + memberId);
                }
            } catch (SQLException e) {
                log.error("DB Error", e);
                throw e;
            } finally {
                // Connection 은 닫지 않는다.
                JdbcUtils.closeResultSet(rs);
                JdbcUtils.closeStatement(pstmt);
            }
        }
    
        public void update(Connection con, String memberId, int money) throws SQLException {
            String sql = "UPDATE member SET money=? WHERE member_id=?";
    
            PreparedStatement pstmt = null;
    
            try {
                pstmt = con.prepareStatement(sql);
                pstmt.setInt(1, money);
                pstmt.setString(2, memberId);
                int resultSize = pstmt.executeUpdate(); // 쿼리를 실행하고 영향받은 row 수를 반환
                log.info("resultSize={}", resultSize);
            } catch (SQLException e) {
                log.error("DB Error", e);
                throw e;
            } finally {
                // Connection 은 닫지 않는다.
                JdbcUtils.closeStatement(pstmt);
            }
        }
    }
    ```
    
- 직접적으로 트랜잭션을 사용하는 _==**서비스 계층에서 커넥션을 연결하고 autocommit 을 false 로 하고 connection 을 반환하는 로직**==_이 들어간다.
- 비즈니스 로직을 메서드 추출한 이유는 try-catch 문 안에는 대부분 Connection 과 관련된 코드가 있기 때문에 따로 분리하였다.
- try 문 마지막 코드에는 `commit`을 해주고 _==**예외가 터져서 catch 문으로 가면 rollback**==_ 을 하고 에러코드를 남긴다.
- finally 구문에서는 커넥션이 있을경우 ==**autocommit 을 원래대로 설정해주고 반환**==해주면 된다.
    
    - autocommit 이 false 인 상태로 커넥션 풀에 반환하면 다른 세션이 커넥션을 획득할 때 autocommit 이 false 인 커넥션을 획득할 수 있기 때문.
    
    ```java
    /**
     * 트랜잭션 - 파라미터 연동, 풀을 고려한 종료
     */
    @Slf4j
    @RequiredArgsConstructor
    public class MemberServiceV2 {
    
        private final DataSource dataSource;
        private final MemberRepositoryV2 memberRepository;
    
        public void accountTransfer(String fromId, String toId, int money) throws SQLException {
            Connection con = dataSource.getConnection();
            try {
                con.setAutoCommit(false); // 트랜잭션 시작
                bizLogic(con, fromId, toId, money);
                con.commit(); // 로직이 예외없이 수행했기 때문에 commit 실행
            } catch (Exception e) {
                // 예외발생시 해당구문으로 이동
                con.rollback(); // 실패 했기 때문에 롤백 수행
                throw new IllegalStateException(e);
            } finally {
                if (con != null) {
                    try {
                        release(con);
                    } catch (Exception e) {
                        log.error("error", e);
                    }
                }
            }
        }
    
        private void bizLogic(Connection con, String fromId, String toId, int money) throws SQLException {
            // 비즈니스 로직
            Member fromMember = memberRepository.findById(con, fromId);
            Member toMember = memberRepository.findById(con, toId);
    
            memberRepository.update(con, fromId, fromMember.getMoney() - money);
            validation(toMember);
            memberRepository.update(con, toId, toMember.getMoney() + money);
        }
    
        private static void release(Connection con) throws SQLException {
            // Connection 을 설정할 때 autocommit 을 자동으로 했기 때문에 커넥션풀에 반환할 때도 자동인 채로 반환된다.
            // 다른 누군가 autocommit 이 자동인줄 알고 connection 을 획득할 수 있기 때문에 설정을 원래대로 하고 반환한다.
            con.setAutoCommit(true);
            con.close();
        }
    
        private static void validation(Member toMember) {
            if (toMember.getMemberId().equals("ex")) {
                throw new IllegalStateException("이체중 예외 발생");
            }
        }
    }
    ```
