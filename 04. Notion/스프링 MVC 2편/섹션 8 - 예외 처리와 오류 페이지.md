## 서블릿 예외 처리 - 시작

---

스프링이 아닌 ==**서블릿 컨테이너는 예외를 2가지 방법으로 처리**==한다.  
1) `Exception(예외)`  
2) `HttpServletResponse.sendError(HTTP상태코드, 오류 메시지)`

### Exception

- 자바의 메인 메서드에서 실행하는 경우 main 이라는 쓰레드가 실행되고 실행 도중에 예외를 처리하지못하면 예외 정보를 남기고 해당 쓰레드는 종료된다.
- 웹 어플리케이션에서는 어플리케이션에서 예외를 잡지 못하면 서블릿 밖까지 예외가 전달된다.
    
    WAS ← 필터 ← 서블릿 ← 인터셉터 ← 컨트롤러(예외가 발생)
    
- 임의로 컨트롤러에서 예외를 터뜨리고 WAS 에서 어떻게 오류화면을 보여주는지 보자
- 그전에 스프링이 제공하는 기본 오류 페이지를 꺼야한다.
    
    ```yaml
    server.error.whitelabel.enabled=false
    ```
    
- 컨트롤러에서 예외를 터뜨려보자
    
    ```java
    @Slf4j
    @Controller
    public class ServletExController {
    
        @GetMapping("/error-ex")
        public void errorEx(){
            throw new RuntimeException("공습경보!"); // 임의로 예외 발생
        }
    }
    ```
    
    ![[Untitled 10.png|Untitled 10.png]]
    
- ==_**서버내에서 Exception 을 처리하지 못했기 때문에 HTTP 상태코드 500 을 반환한것이다.**_==

### response.sendError()

- `sendError()` 를 호출한다고 예외가 발생하는건 아니고 ==**서블릿 컨테이너에게 오류가 발생했다는 점을 전달할 수 있다.**==
    
    `response.sendError(HTTP 상태코드)`
    
    `response.sendError(HTTP 상태코드, 오류메시지)`
    
- Exception 과는 흐름이 조금 다르다.
    
    WAS(sendError 호출 기록 확인) ← 필터 ← 서블릿 ← 인터셉터 ← 컨트롤러(response.sendError())
    

  

> _**서블릿 컨테이너가 제공하는 기본 예외 처리 화면은 사용자가 보기에 불편하다 !**_

## 서블릿 예외 처리 - 오류 화면 제공

---

==_**서블릿이 제공하는 기본 오류 화면은 사용자에게 불친절하다. 개발자가 임의로 오류화면을 만들고 서블릿에 등록해보자 !!**_==

- 우선 `resources/templates/error-page` 경로에 `404.html` , `500.html` 파일을 만들어서 오류 화면을 만들자.
- 그리고 서블릿이 제공하는 `WebServerFactoryCustomizer<>` 인터페이스를 구체화시킨 클래스에서 임의로 만든 오류 화면페이지를 등록하면 된다.
- `ErrorPage` 클래스에서 ==**HTTP 상태코드와 오류화면 경로를 생성자**==로 받는다. 그렇게 되면 _==**해당 HTTP 상태코드가 반환될 때 해당 경로로 이동**==_시킨다.
    
    ```java
    @Component
    public class WebServerCustomizer implements WebServerFactoryCustomizer<ConfigurableWebServerFactory> {
        @Override
        public void customize(ConfigurableWebServerFactory factory) {
            ErrorPage errorPage404 = new ErrorPage(HttpStatus.NOT_FOUND, "/error-page/404");
            ErrorPage errorPage505 = new ErrorPage(HttpStatus.INTERNAL_SERVER_ERROR, "/error-page/500");
    
            ErrorPage errorPageEx = new ErrorPage(RuntimeException.class, "/error-page/500");
    
            factory.addErrorPages(errorPage505,errorPage404,errorPageEx);
        }
    }
    ```
    
- 해당 경로와 같은 컨트롤러를 구현해주면 된다.
    
    ```java
    @Slf4j
    @Controller
    public class ErrorPageController {
    
        @RequestMapping("/error-page/404")
        public String errorPage404(HttpServletRequest request, HttpServletResponse response){
            log.info("errorPage 404");
            return "error-page/404";
        }
    
        @RequestMapping("/error-page/500")
        public String errorPage500(HttpServletRequest request, HttpServletResponse response){
            log.info("errorPage 500");
            return "error-page/500";
        }
    }
    ```
    

  

> _**RuntimeException 이나 서버에서 발생한 예외인 500 에러가 터지면 /error-page/500 컨트롤러가 호출되면서 오류화면페이지가 띄어지게 된다.**_

## 서블릿 예외 처리 - 오류 페이지 작동 원리

---

- WAS 는 예외 처리를 하는 오류 페이지 정보를 확인한다.
    
    `new ErrorPage(RuntimeException.class, "/error-page/500");`
    
- RuntimeException 예외가 터지면 위에 `ErrorPage` 정보를 보고 `/error-page/500` 경로를 _==**재요청**==_ 한다.
    
    WAS ‘/error-page/500’ 재요청 → 필터 → 서블릿 → 인터셉터 → 컨트롤러(/error-page/500) → View
    
- ==_**웹 브라우저(클라이언트)는 서버 내부에서 이렇게 재요청을 하는지 전혀 모른다.**_==

  

> **예외가 발생해서 WAS 까지 전달이 되면 WAS 는 오류 페이지 경로를 찾아서 내부에서 오류 페이지를 호출한다. 이 때 오류 페이지 경로로 필터 → 서블릿 → 인터셉터 → 컨트롤러가 다시 호출된다.**

### 오류 정보 추가

- WAS 는 오류 페이지 경로를 재호출만 하는게 아니라 오류 정보를 request 의 attribute 에 추가해서 넘겨준다.
    
    `ERROR_EXCEPTION` : 예외
    
    `ERROR_EXCEPTION_TYPE` : 예외 타입
    
    `ERROR_MESSAG` : 오류 메시지
    
    `EERROR_REQUEST_URI` : 클라이언트 요청 URI
    
    `ERROR_SERVLET_NAME` : 오류가 발생한 서블릿 이름
    
    `ERROR_STATUS_CODE` : HTTP 상태 코드
    
    ```java
    @Slf4j
    @Controller
    public class ErrorPageController {
    
        //RequestDispatcher 상수로 정의되어 있음
        public static final String ERROR_EXCEPTION = "javax.servlet.error.exception";
        public static final String ERROR_EXCEPTION_TYPE = "javax.servlet.error.exception_type";
        public static final String ERROR_MESSAGE = "javax.servlet.error.message";
        public static final String ERROR_REQUEST_URI = "javax.servlet.error.request_uri";
        public static final String ERROR_SERVLET_NAME = "javax.servlet.error.servlet_name";
        public static final String ERROR_STATUS_CODE = "javax.servlet.error.status_code";
    
        @RequestMapping("/error-page/404")
        public String errorPage404(HttpServletRequest request, HttpServletResponse response){
            log.info("errorPage 404");
            printErrorInfo(request);
            return "error-page/404";
        }
    
        @RequestMapping("/error-page/500")
        public String errorPage500(HttpServletRequest request, HttpServletResponse response){
            log.info("errorPage 500");
            printErrorInfo(request);
            return "error-page/500";
        }
    
        private void printErrorInfo(HttpServletRequest request){
            log.info("ERROR_EXCEPTION : {}", request.getAttribute(ERROR_EXCEPTION));
            log.info("ERROR_EXCEPTION_TYPE : {}", request.getAttribute(ERROR_EXCEPTION_TYPE));
            log.info("ERROR_MESSAGE : {}", request.getAttribute(ERROR_MESSAGE));
            log.info("ERROR_REQUEST_URI : {}", request.getAttribute(ERROR_REQUEST_URI));
            log.info("ERROR_SERVLET_NAME : {}", request.getAttribute(ERROR_SERVLET_NAME));
            log.info("ERROR_STATUS_CODE : {}", request.getAttribute(ERROR_STATUS_CODE));
    
            log.info("dispatchType={}", request.getDispatcherType());
        }
    }
    ```
    

## 서블릿 예외 처리 - 필터

---

오류가 발생하면 WAS 에서 오류 페이지를 재요청하면서 다시 필터 → 서블릿 → 인터셉터 → 컨트롤러가 호출이 된다. 만약 로그인 인증 체크 로직이라면 또 필터나 인터셉터에서 로그인 인증을 체크한다면 비효율적이다. 그래서 ==**필터에는 클라이언트가 요청했는지 아니면 WAS 내부에서 요청했는지 구분하기 위해 DispatcherType 옵션을 제공**==한다.

### DispatcherType

- 필터에서는 ==_**클라이언트가 직접 요청한것인지 WAS 가 오류 페이지를 호출하기 위한 요청인지 DispatcherType 을 통해 구분**_==한다.
- 디폴트 값은 `DispatcherType.REQUEST` 이다.
- WAS에서 오류페이지를 호출할 때 재요청을 하면 `DispatcherType.ERROR` 이다.
    
    - `REQUEST` : 클라이언트 요청
    - `ERROR` : 오류 요청
    - `FORWARD` : 서블릿에서 다른 서블릿이나 JSP 를 호출할 때
    - `INCLUDE` : 서블릿에서 다른 서블릿이나 JSP의 결과를 포함할 때
    - `ASYNC` : 서블릿 비동기 호출
    
    ```java
    public enum DispatcherType{
    	FORWARD,
    	INCLUDE,
    	REQUEST,
    	ASYNC,
    	ERROR
    }	
    ```
    
- 필터를 등록할 때 setDispatcherType() 메서드에 DispatcherType 을 넣으면 된다.
    
    ```java
    @Configuration
    public class WebConfig implements WebMvcConfigurer {
    
        @Bean
        public FilterRegistrationBean logFilter(){
            FilterRegistrationBean<Filter> filterFilterRegistrationBean = new FilterRegistrationBean<>();
            filterFilterRegistrationBean.setFilter(new LogFilter());
            filterFilterRegistrationBean.setOrder(1);
            filterFilterRegistrationBean.addUrlPatterns("/*");
            filterFilterRegistrationBean.setDispatcherTypes(DispatcherType.REQUEST,DispatcherType.ERROR);
            return filterFilterRegistrationBean;
        }
    }
    ```
    

## **서블릿 예외 처리 - 인터셉터**  

---

==**스프링 인터셉터는 WAS의 중복요청을 필터와 다르게 오류페이지 경로를 인터셉터 대상에서 제외시키기만 해도 된다.**==

- 아래와 같이 인터셉터 등록 시에 경로를 빼서 설정이 가능하다.
    
    ```java
    @Configuration
    public class WebConfig implements WebMvcConfigurer {
    
        @Override
        public void addInterceptors(InterceptorRegistry registry) {
            registry.addInterceptor(new LogInterceptor())
                    .order(1)
                    .addPathPatterns("/**")
                    .excludePathPatterns(
                            "/css/**", "/*.ico", "/error", "/error-page/**"//오류 페이지 경로
                    );
        }
    }
    ```
    

## 스프링 부트 - 오류 페이지

---

_**예외 처리 페이지를**_ `_HTML_` _**로 만들고 또**_ `_WebServerFactoryCustomizer_` _**인터페이스를 구현화하는 복잡한 과정을 스프링부트에서는 간단하게 지원을 한다.**_

### BasicErrorController

- 스프링부트는 자체적으로 오류페이지를 자동으로 등록하는 `BasicErrorController` 를 지원한다.
- 즉, 개발자는 오류페이지만 템플릿엔진이나 정적리소스를 룰과 규칙을 지켜 만들어주기만 하면 스프링부트가 제공하는 `BasicErrorController` 에서 알아서 처리해준다.
- 스프링부트는 예외처리 페이지를 찾을때 `/error` 로 찾아간다.
- 뷰 선택 우선순위
    
    1. 뷰 템플릿
        - `resources/templates/error/500.html`
        - `resources/templates/error/5xx.html`
    2. 정적 리소스
        - `resources/static/error/400.html`
        - `resources/static/error/404.html`
        - `resources/static/error/4xx.html`
    3. 적용 대상이 없을 때 (error 라는 이름의 뷰를 찾는다)
        - `resources/templates/error.html`
    
    > _**기본적으로 디테일한 부분부터 먼저 찾는다(500>5xx)**_
    

  

> _==**개발자는 HTTP 상태코드에 따라 클라이언트에게 보여주는 뷰만 만들면 스프링부트가 알아서 해준다.**==_
