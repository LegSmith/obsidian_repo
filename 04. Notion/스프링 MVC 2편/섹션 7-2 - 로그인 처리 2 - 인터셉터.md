# 스프링 인터셉터 - 소개

---

==_**스프링 인터셉터는 필터와 같이 공통 관심 사항을 해결하지만 좀더 스프링 MVC 에 특화된 기술**_==이다. 필터와는 적용되는 순서와 범위가 다르다.

### 스프링 인터셉터 흐름

- 서블릿은 스프링 MVC 기준으로 한다면 디스패처 서블릿이 맞다. ==**디스패처 서블릿 까지 HTTP 요청이 도달하고 그 다음 스프링 인터셉터가 동작**==한다.
    
    ```xml
    HTTP 요청 -> WAS -> 필터 -> 서블릿(디스패터 서블릿) -> 스프링 인터셉터 -> 컨트롤러
    ```
    

### 인터셉터와 필터의 차이

- 필터와 똑같이 인터셉터에서 _**적절하지 않은 요청이라 판단되면 컨트롤러까지 가지 않는다.**_
- 필터와 같이 _**체인으로 구성**_이 가능하다.
- 필터보다 더 편리하고 정교하고 다양한 기능을 제공하므로 _**==꼭 필터가 사용한 경우가 아니라면 스프링 인터셉터를 사용하는게 좋다.==**_

### 스프링 인터셉터 인터페이스

```java
public interface HandlerInterceptor {

	default boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler)
			throws Exception {

		return true;
	}

	default void postHandle(HttpServletRequest request, HttpServletResponse response, Object handler,
			@Nullable ModelAndView modelAndView) throws Exception {
	}

	default void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler,
			@Nullable Exception ex) throws Exception {
	}

}
```

### 스프링 인터셉터 호출 흐름

- `preHandle()` : 정확히는 핸들러 어댑터 호출 전에 호출된다.
    - 응답값이 `true` 이면 다음으로 동작하고 `false` 이면 더는 진행하지 않는다.
- `postHandle()` : 컨트롤러 호출 후에 호출된다.
    - 만약 ==_**컨트롤러에서 예외가 발생되면 postHandle 은 호출되지 않는다.**_==
- `afterCompletion()` : 뷰가 랜더링 된 이후에 호출된다.
    
    - _==**예외가 발생해도 무조건 호출**==_된다. 예외(Exception) 을 파라미터로 받아서 로그로 출력 가능하다.

## 스프링 인터셉터 - 요청 로그

---

### preHandle()

- `preHanel()` 에 ==**handler 파라미터는 호출할 컨트롤러 메서드의 모든 정보가 포함**==되어 있다.
- 필터 때와 마찬가지고 UUID 를 사용하여 HTTP 요청을 식별해야 하는데 동일한 uuid 를 preHandle() , postHandle() 과 같이 다른 메서드에서도 사용해야 한다.
- ==_**UUID 를 멤버 변수로 빼버리면 싱글톤으로 작동하는 인터셉트에게 치명적**_==이다. → setAttribute() 메서드를 이용하여 request 객체에 담으면 된다.
- ==**return 값은 반드시** **`true`** **가 되야 어댑터 핸들러까지 가고** **`false`** **이면 여기서 멈춘다.**==
    
    ```java
    public static final String LOG_ID = "logId";
    
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
    
        String requestURI = request.getRequestURI();
        String uuid = UUID.randomUUID().toString();
    
        request.setAttribute(LOG_ID, uuid);
    
        log.info("REQUEST [{}][{}][{}]",uuid, requestURI, handler);
    
        return true;
    }
    ```
    

### postHandle()

- `postHandle()` 은 MoelAndView 를 가지고 있어서 View 에 정보를 담을 수 있다
    
    ```java
    @Override
    public void postHandle(HttpServletRequest request, HttpServletResponse response, Object handler, ModelAndView modelAndView) throws Exception {
        log.info("postHandle [{}]",modelAndView);
    }
    ```
    

### afterCompletion()

- 인터셉터 종료 로그를 postHandle() 에 담지 않은 이유는 ==**컨트롤러에서 예외가 터지면 postHandle() 은 호출되지 않지만 afterCompletion() 은 무슨일이 있어도 호출**==되기 때문에 종료 로그를 담았다.
- 예외 파라미터가 null 이 아니면 예외가 발생했다는 뜻이므로 log.error를 실행
    
    ```java
    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex) throws Exception {
        String requestURI = request.getRequestURI();
        String logId = (String) request.getAttribute(LOG_ID);
    
        log.info("RESPONSE [{}][{}][{}]",logId,requestURI,handler);
        if(ex != null){
            log.error("afterCompletion error!!", ex);
        }
    }
    ```
    

### 스프링 인터셉터 등록

- 필터와 다르게 등록 클래스에 `WebMvcConfigurer` 인터페이스를 받아 구현화 해야 한다.
- `addInterceptors()` 메서드를 오버라이딩 하여 등록한다.
- `addInterceptors()` 메서드로 스프링 인터셉터를 등록한다.
- ==_**필터와 다른점은 인터셉터에 적용할 URI 패턴과 적용하지 않아도 되는 URI 패턴을 구분하여 등록할 수 있다.**_==
    
    ```java
    @Configuration
    public class WebConfig implements WebMvcConfigurer {
    
        @Override
        public void addInterceptors(InterceptorRegistry registry) {
            registry.addInterceptor(new LogInterceptor())
                    .order(1)
                    .addPathPatterns("/**")
                    .excludePathPatterns("/css/**","/*.ico","/error");
        }
    		//생략...
    }
    ```
    

> 패턴 방식은 스프링 공식 문서에서 찾으면 된다.
> 
> > [!info] PathPattern (Spring Framework 6.0.13 API)  
> > declaration: package: org.  
> > [https://docs.spring.io/spring-framework/docs/current/javadoc-api/org/springframework/web/util/pattern/PathPattern.html](https://docs.spring.io/spring-framework/docs/current/javadoc-api/org/springframework/web/util/pattern/PathPattern.html)  

## 스프링 인터셉터 - 인증 체크

---

- 로그인 인증은 컨트롤러 호출 전에만 호출 되면 된다 → `preHandle` 만 구현
    
    - 필터와 다르게 `HandlerInterceptor` 는 모든 메서드가 `default` 로 설정되기 때문에 ==**필요한 메서드만 오버라이딩이 가능하다.**==
    
    ```java
    @Slf4j
    public class LoginCheckInterceptor implements HandlerInterceptor {
        @Override
        public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
            String requestURI = request.getRequestURI();
            log.info("인증 체크 인터셉터 실행 -> {}", requestURI);
    
            HttpSession session = request.getSession();
            if(session == null || session.getAttribute(SessionConst.LOGIN_MEMBER) == null){
                log.info("미인증 사용자 요청");
    
                response.sendRedirect("/login?redirectURL=" + requestURI);
                return false;
            }
    
            return true;
        }
    }
    ```
    
- 등록도 일일이 `@Bean` 어노테이션으로 스프링 빈 등록 없이 `addInterceptor()` 로 추가만 해주면 된다.
    
    ```java
    @Override
    public void addInterceptors(InterceptorRegistry registry) {		
    		//생략
    
        registry.addInterceptor(new LoginCheckInterceptor())
                .order(2)
                .addPathPatterns("/**")
                .excludePathPatterns("/","/members/add","/login","/logout","/css/**","/*.ico","/error");
    }
    ```
    

## ArgumentResolver 활용

---

`ArgumentResolver` 란 _==**어노테이션 기반의 컨트롤러에서 필요한 파라미터를 미리 생성해서 컨트롤러를 호출하면서 파라미터를 넘겨주는 역할**==_을 한다. → ArgumentResolver 덕분에 Model , @RequestParam , @ModelAttribute 등 다양한 어노테이션을 유연하게 처리할 수 있다.

- 임의의 어노테이션을 HomeController 에서 임의의 ==_**@Login 어노테이션을 생성하여 Member 객체를 받아보자**_==
- 이렇게만 하고 실행하면 `@Login` 어노테이션을 ==_**ArgumentResolver 가 인식을 못해서 null 값을 표시할 것이다.**_==
    
    ```java
    @GetMapping("/")
    public String homeLoginV3ArgumentResolver(@Login Member loginMember, Model model){
    
        if(loginMember == null)
            return "home";
    
        model.addAttribute("member",loginMember);
        return "loginHome";
    }
    ```
    
    ![[Untitled 1 7.png|Untitled 1 7.png]]
    
- 이제 @Login 어노테이션을 생성하고 ArgumentResolver 에 등록해주면 된다.
    
    - `@Target(ElementType.``_PARAMETER_``)` : 파라미터에만 사용
    - `@Retention(RetentionPolicy.``_RUNTIME_``)` : 런타임까지 어노테이션 정보가 남아있게 한다.
    
    ```java
    @Target(ElementType.PARAMETER)
    @Retention(RetentionPolicy.RUNTIME)
    public @interface Login {}
    ```
    
- `HandlerMethodArgumentResolver` 인터페이스를 받아 구현체로 만들어서 ArgumentResolver 가 인식할 수 있게 해준다.
- `supportsParameter()` : ==**@Login 어노테이션이 있으면서 Member 타입**==이면 `ArgumentResolver` 가 사용되게 한다. ( return 값이 `false` 이면 `resolveArgument()` 가 실행 안된다.)
- `resolveArgument()` : 세션이 없으면 null 을 리턴해서 member에 null 값을 담고 세션에 있는 사용자면 세션정보를 갖고와 반환하면 된다.
    
    ```java
    @Slf4j
    public class LoginMemberArgumentResolver implements HandlerMethodArgumentResolver {
        @Override
        public boolean supportsParameter(MethodParameter parameter) {
            log.info("supportsParameter 실행");
    
            boolean hasLoginAnnotation = parameter.hasParameterAnnotation(Login.class);
            boolean hasMemberType = Member.class.isAssignableFrom(parameter.getParameterType());
    
            return hasLoginAnnotation && hasMemberType;
    
        }
    
        @Override
        public Object resolveArgument(MethodParameter parameter, ModelAndViewContainer mavContainer, NativeWebRequest webRequest, WebDataBinderFactory binderFactory) throws Exception {
            log.info("resolveArgument 실행");
    
            HttpServletRequest request = (HttpServletRequest) webRequest.getNativeRequest();
    
            HttpSession session = request.getSession(false);
            if(session == null){
                return null;
            }
    
            return session.getAttribute(SessionConst.LOGIN_MEMBER);
        }
    }
    ```
    
- 새롭게 만든 `LoginMemberArgumentResolver` 를 `WebConfig` 에 등록 시켜주면 된다.
    
    ```java
    @Configuration
    public class WebConfig implements WebMvcConfigurer {
    
        @Override
        public void addArgumentResolvers(List<HandlerMethodArgumentResolver> resolvers) {
            resolvers.add(new LoginMemberArgumentResolver());
        }
    		//생략..
    }
    ```
    

==_**ArgumentResolver 를 활용하면 공통 작업이 필요할 때 컨트롤러를 더욱 편리하게 사용할 수 있따.**_==
