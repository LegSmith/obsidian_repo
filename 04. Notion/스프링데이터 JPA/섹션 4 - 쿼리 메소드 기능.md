## 메소드 이름으로 쿼리 생성

---

==_**스프링 데이터 JPA 의 공통인터페이스도 강력한 기능이지만 짧은 쿼리나 간단한 쿼리를 인터페이스 선언만으로 구현체를 만들지 않고 쿼리를 작성해주는 기능이 있다 !**_==

- username과 age 를 where 조건절에 거는 쿼리가 있다고 하자
    
    `SELECT * FROM Member m WHERE m.username = :username AND m.age >`
    
- 스프링 데이터 JPA 없이 짠다면 createQuery() 를 통해 JPQL 을 작성해야 한다. 
    
    ```java
    em.createQuery("SELECT m FROM Member m WHERE m.username = :username AND m.age > :age")
          .setParameter("username",username)
          .setParameter("age",age)
          .getResultList();
    ```
    
- 하지만 스프링 데이터 JPA 는 메서드 이름으로 쿼리를 생성해주는 기능이 있다.
    
    ```java
    public interface MemberRepository extends JpaRepository<Member,Long> {
      List<Member> findByUsernameAndAgeGreaterThan(String username, int age);
    }
    ```
    
- 하지만 규칙이 있다.
    - 공식문서에 자세하게 나와있다.
        
        > [!info] JPA Query Methods :: Spring Data JPA  
        > This section describes the various ways to create a query with Spring Data JPA.  
        > [https://docs.spring.io/spring-data/jpa/reference/jpa/query-methods.html](https://docs.spring.io/spring-data/jpa/reference/jpa/query-methods.html)  
        

> 프로젝트가 클 수록 간단한 쿼리가 많아진다. 하나하나 만들기보다는 스프링 데이터 JPA 의 기능을 활용하는게 좋다. 조건이 복잡해지면 메서드명이 길어지기 때문에 짧은 쿼리에 사용하는게 좋다.

  

## @Query

---

_**JPA레포지토리 인터페이스에서**_ `@Query` _**어노테이션으로 JPQL을 직접적어서 메서드에 적용할 수 있다.**_

- `@Query` 어노테이션 안에 JPQL 을 실행하여 해당 메서드 호출시 JPQL을 실행시키게 된다.
    
    ```java
    public interface MemberRepository extends JpaRepository<Member,Long> {
        @Query("SELECT m FROM Member m WHERE m.username = :username AND m.age = :age")
        List<Member> findUser(@Param("username") String username, @Param("age") int age);
    }
    ```
    

> _**장점은 애플리케이션 로딩시에 JPQL을 파싱하기 때문에 실수로 오타를 입력해도 에러를 찾아준다**_

## @Query - DTO 로 조회

---

`_**@Query**_` _**어노테이션을 사용했을 때는 반환값이 인터페이스에 선언된 타입이어야 하지만 실무에서는 DTO를 반환할 떄가 많다. 이 떄는**_ ==_new 키워드_==_**를 사용한다.**_

- 우선 Member 엔티티의 `username` Team 엔터티의 `teamName`을 반환하는 DTO를 생성한다.
    
    ```java
    @Data
    public class MemberDto {
        public MemberDto(Long id, String username, String teamName) {
            this.id = id;
            this.username = username;
            this.teamName = teamName;
        }
    
        private Long id;
        private String username;
        private String teamName;
    }
    ```
    
- 쿼리에서 출력하는 속성값들이 DTO 이므로 SELECT 키워드 뒤에 오는 값들을 new 키워드로 묶어준다. → 이 때 ==**DTO의 경로를 함께 적어줘야 한다 !**==
    
    ```java
    public interface MemberRepository extends JpaRepository<Member,Long> {
        @Query("SELECT new study.datajpa.dto.MemberDto(m.id, m.username, t.name) FROM Member m JOIN m.team t")
        List<MemberDto> findMemberDto();
    }
    ```
    

## 파라미터 바인딩

---

_==**파라미터 바인딩에는 위치기반 , 이름기반이 있다. 가급적 이름기반으로 쓰자 !**==_

### 위치기반 과 이름기반

- 가독성에 있어서 이름기반이 유지보수하기 쉽다.
    
    ```java
    SELECT m FROM Member m WHERE m.username = ?0; // 위치기반
    SELECT m FROM Member WHERE m.username = :name; // 이름기반
    ```
    

### 파라미터 바인딩

- 파라미터로 컬렉션을 받아서 SQL 에 IN 키워드를 편리하게 쓸 수있다.
    
    ```java
    @Query("SELECT m FROM Member m WHERE m.username in :names")
    List<Member> findByNames(@Param("names") List<String> names);
    ```
    

## 반환 타입

---

==**스프링 데이터 JPA 는 유연하게 반환 타입을 지원한다.**==

- 컬렉션이나 단건, Optional 로 감싸진 단건 등 다양한 리턴타입을 지원한다.
    
    ```java
    List<Member> findListByUsername(String username); // 컬렉션
    Member findMemberByUsername(String username); // 단건
    Optional<Member> findOptionalByUsername(String username); // 단건 Optional
    ```
    
    > _**스프링 데이터 JPA 공식문서**_
    > 
    > > [!info] Spring Data JPA - Reference Documentation  
    > > © 2008-2023 The original authors.  
    > > [https://docs.spring.io/spring-data/jpa/docs/current/reference/html/#repository-query-return-types](https://docs.spring.io/spring-data/jpa/docs/current/reference/html/#repository-query-return-types)  
    
- ==_**컬렉션에 경우 결과가 없으면 Null 이 아닌 Empty Collection 을 반환한다.**_==
- 단건 조회는 null 을 반환하기 때문에 Java 8 이 지원하는 Optional 을 사용하는게 좋다.
    - 단, JPA 에서는 `javax.persistence.NoResultException` 예외가 발생
    - 결과과 2건 이상일 때는 `javax.persistence.NonUniqueResultException` 예외가 발생.

## 순수 JPA 페이징과 정렬

---

- JPA 에서 제공하는 페이징은 시작페이지(offset) , 조회할 데이터 수(limit)을 파라미터로 받아 `setFirstResult()` , `setMaxResults()` 메서드로 간편하게 페이징을 할 수 있다.
    
    ```java
    public List<Member> findByPage(int age, int offset, int limit){
        return em.createQuery("SELECT m FROM Member m WHERE m.age = :age ORDER BY m.username DESC")
                .setParameter("age",age)
                .setFirstResult(offset) // 몇 번 째 부터
                .setMaxResults(limit) // 몇 개 까지
                .getResultList();
    }
    ```
    

## 스프링 데이터 JPA 페이징과 정렬

---

스프링 데이터 JPA 에서 지원하는 Page 인터페이스와 Slice 인터페이스가 있다.

- 쿼리메서드 기능으로 findByxxx 를 하면 xxx를 조회할 수 있다.
- 첫번째 파라미터에는 조건을 받고, 두번재 파라미터는 `Pageable` 을 받으면 된다.
    
    ```java
    public interface MemberRepository extends JpaRepository<Member,Long> {
    
        Page<Member> findByAge(int age, Pageable pageable);
    
    }
    ```
    
- PageRequest 은 Pageable 자식 클래스이고 offset , limie , 정렬방식을 파라미터로 받아서 페이징조건을 정할 수 있다.
    
    ```java
    @Test
    public void paging(){
        //given
        memberRepository.save(new Member("member1",10));
        memberRepository.save(new Member("member2",10));
        memberRepository.save(new Member("member3",10));
        memberRepository.save(new Member("member4",10));
        memberRepository.save(new Member("member5",10));
        memberRepository.save(new Member("member6",10));
    
        int age = 10;
        PageRequest pageRequest = PageRequest.of(0, 3, Sort.by(Sort.Direction.ASC,"username"));
    
        //when
        Page<Member> page = memberRepository.findByAge(age, pageRequest);
    
        //then
        List<Member> content = page.getContent();
        long totalCount = page.getTotalElements();
    
        assertThat(content.size()).isEqualTo(3);
        assertThat(totalCount).isEqualTo(6);
        assertThat(page.getNumber()).isEqualTo(0);
        assertThat(page.getTotalPages()).isEqualTo(2);
        assertThat(page.isFirst()).isTrue();
        assertThat(page.isLast()).isFalse();
        assertThat(page.hasNext()).isTrue();
    }
    ```
    

### Page , Slice 인터페이스

- Page 인터페이스는 Slice 인터페이스의 자식인터페이스이기 때문에 Slice 인터페이스의 다양한 메서드를 사용할 수 있다.
- Page 인터페이스는 Slice 인터페이스와 다르게 count 를 계산하는 쿼리가 나가기 때문에 전체 페이지 수를 알 수 있다.
    
    ```java
    public interface Page<T> extends Slice<T> {
    		int getTotalPages(); //전체 페이지 수 longgetTotalElements();//전체 데이터 수
    		<U> Page<U> map(Function<? super T, ? extends U> converter); //변환기
    }
    ```
    
- Slice 는 count 쿼리가 나자기 않는다.
    
    ```java
    public interface Slice<T> extends Streamable<T> {
    		intgetNumber();  // 현재 페이지
    		intgetSize();  // 페이지 크기
    		int getNumberOfElements(); // 현재 페이지에 나올 데이터 수
    		List<T> getContent(); // 조회된 데이터
    		boolean hasContent(); // 조회된 데이터 존재 여 부
    		Sort getSort();  // 정렬 정보
    		boolean isFirst();  // 현재 페이지가 첫 페이지 인지 여부
    		boolean isLast();  // 현재 페이지가 마지막 페이지 인지 여부
    		boolean hasNext();  // 다음 페이지 여부
    		boolean hasPrevious();  // 이전 페이지 여부
    		Pageable getPageable();  // 페이지 요청 정보
    		Pageable nextPageable();   // 다음 페이지 객체
    		PageablepreviousPageable();  // 이전 페이지 객체
    		<U> Slice<U> map(Function<? super T, ? extends U> converter); //변환기
    }
    ```
    

### 실무에서는 count 쿼리를 분리해야함

- 실무 쿼리는 매우 길기 때문에 Page 로 count 쿼리가 같이나가면 복잡한 쿼리에 count 가 나가기 때문에 성능이 안나올 수 있다.
- 예를 들어 Member 와 Team 을 Left Outer Join 을 하면 count 도 똑같이 Left Outer Join 을 하면서 구하기 때문에 성능이 안나온다.
    
    ```java
    @Query(value = "select m from Member m left outer join m.team",
            countQuery = "select count(m.username) from Member m")
    Page<Member> findbyAge(int age , pageable);
    ```
    

## 벌크성 수정 쿼리

---

- 다량의 데이터를 한꺼번에 수정해야할 일이 생길때가 있다.
- DB의 변동을 줄 때는 `@Modifying` 어노테이션을 사용해야 한다. 사용하지 않으면 JPQL 문법이후에 `getResultList()` 나 `getSingleResult()` 를 호출한다.
- `clearAutomatically = true` 로 설정한 이유는 bulkAgePlus() 를 호출하면 ==**JPQL 문법이 실행되면서 영속성 컨텍스트는 flush()를 호출해 SQL을 DB로 전송**==한다. 하지만 ==**1차 캐시에는 데이터가 최신화되지 않기때문에 DB와 데이터가 일치하지 않는다. → 데이터 동기화를 위해 !!**==
- 그래서 로직에서 `EntityManager.clear()` 를 날려줘야한다. 하지만 `clearAutomatically = true` 를 설정하면 clear()를 알아서 해준다.
    
    ```java
    public interface MemberRepository extends JpaRepository<Member,Long> {
        @Modifying(clearAutomatically = true)
        @Query("UPDATE Member m SET m.age = m.age + 1 WHERE m.age >= :age")
        int bulkAgePlus(@Param("age") int age);
    }
    ```
    

## @EntityGraph

---

`@EntityGraph` 는 ==_**스프링데이터 JPA가 패치조인(Fetch Join)을 간편하게 제공**_==할려고 지원하는 어노테이션ㅇ디ㅏ.

- 일반적으로 패치조인을 사용할 때는 아래와 같이 JPQL문법을 직접 작성하는 경우가 있다.
    
    ```java
    @Query("SELECT m FROM Member m LEFT JOIN FETCH m.team")
    List<Member> findMemberFetchJoin();
    ```
    
- 하지만 스프링데이터 JPA를 사용할 때 패치조인을 사용할려면 메서드를 게속 추가해야하는 불편함이 있다. → `@Override` 어노테이션을 사용하면 `findAll()`같이 자주사용하는 메서드에 패치조인을 입힐 수 있다.
- 그리고 `@EntityGraph(attributePath = {”연관엔티티명”})` 을 메서드 위에 붙혀주면 된다.
    
    ```java
    @Override
    @EntityGraph(attributePaths = {"team"}) //JPQL을 직접 안짜도 된다 !
    List<Member> findAll();
    ```
    
- 또는 JPQL 로 주 엔티티만 작성하고 연관엔티티는 @EntityGraph 로 지정할 수 있다.
    
    ```java
    @EntityGraph(attributePaths = {"team"})
    @Query("SELECT m FROM Member m")
    List<Member> findMemberEntityGraph();
    ```
    

  

> 즉, JPA의 패치조인을 간편하게 사용할려면 `@EntityGraph` 를 사용하면 된다.
