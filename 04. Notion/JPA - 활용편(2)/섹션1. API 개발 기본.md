# 회원 등록 API

---

==**패키지를 구성할때 API Controller와 Controller를 구분해서 만드는게 좋다!!**==

## 회원등록 API - V1

- `@RestController`는 기존의 `@Controller`와 `@ResponseBody`의 기능이 합쳐진 어노테이션이다.
    - `@ResponseBody` : 값을 return한다.
- id값만 리턴하는 static class를 만들고 Member를 파라미터로 받아서 id값을 리턴한다.
    
    ```java
    @RestController //@Controller + @ResponseBody
    @RequiredArgsConstructor
    public class MemberApiController {
    
        private final MemberService memberService;
    
        @PostMapping("/api/v1/members")
        public CreateMemberResponse saveMemberV1(@RequestBody @Valid Member member){
            //@RequestBody : Json 으로 온 body 를 Member 로 변환
            Long id = memberService.join(member);
            return new CreateMemberResponse(id);
        }
    
        @Data
        static class CreateMemberResponse{
            private Long id;
    
            public CreateMemberResponse(Long id) {
                this.id = id;
            }
        }
    }
    ```
    
    ### V1 의 문제점
    
    1. 직접적으로 ==_**Entity를 @RequestBody로 받기 때문에 Entity의 컬럼명이 바뀐다면 API 스펙 자체가 바뀐다**_==
    2. 실무에서는 회원(Member) 엔티티를 위한 다양한 API가 만들어진다, 한 엔티티에 각각의 API를 위한 모든 요청 요구사항을 담기는 어렵다.
    3. ==**API 통신을 할 때 에는 DTO를 별도로 생성해서 파라미터로 받는다**==

## 회원등록 API - V2

- CreateMemberRequest 라는 별도의 DTO 를 만들어서 파라미터로 받는다.
    
    ```java
    @RestController //@Controller + @ResponseBody
    @RequiredArgsConstructor
    public class MemberApiController {
    
        /**
         * 등록 V2 : 요청 값으로 Member 엔티티 대신에 별도의 DTO 를 받는다.
         */
        @PostMapping("/api/v2/members")
        public CreateMemberResponse saveMemberV2(@RequestBody @Valid CreateMemberRequest request){
            Member member = new Member();
            member.setName(request.getName());
    
            Long id = memberService.join(member);
            return new CreateMemberResponse(id);
        }
    
        @Data
        static class CreateMemberRequest{
            private String name;
        }
    
        @Data
        static class CreateMemberResponse{
            private Long id;
    
            public CreateMemberResponse(Long id) {
                this.id = id;
            }
        }
    }
    ```
    
    ### V2 의 장점
    
    - 엔티티와 API 스펙을 명확하게 분리할 수 있다.
    - 나중에 유지보수 할 때에도 알아보기 쉽다.
    - 엔티티가 변해도 API 스펙이 변하지 않는다.

==**실무에서는 절대 엔티티를 API 스펙에 노출하면 안된다 !!**==

# 회원 수정 API

---

==**회원수정 예제에서는 부분수정을 한다. 부분 수정할때는 Patch 메서드를 사용하는게 REST 스타일에 맞다!!**==  
==**Put 메서드는 전체수정할 때 사용하는 메서드이다.**==

- 회원수정 URI 는 회원등록과 같은 URI에 뒤에 id값을 받는 URI 를 사용
- `@PathVariable` 어노테이션으로 id값을 받고 `UpdateMemberRequest`라는 DTO를 파라미터로 받는다.
- update메서드에서 수정한 엔티티를 반환받는 방법도있지만, 김영한선생님은 커멘드와 쿼리를 분리하는 방식을 선호하신다.
    - update메서드에서 변경감지로 수정을하고 ==**수정한 엔티티를 id값을 이용해 찾아 반환하면 한개의 커멘드(update)에서 반환할 엔티티를 찾는 쿼리를 한번더 하게 되는 셈이다.**== 문제는 없지만 추후에 유지보수에 용이성을 가진다고 한다.
- update를 했으면 거기서 끝내는게 좋다고한다.
- `@AllArgsConstructor` 어노테이션은 모든 필드를 받는 생성자를 만들어주는 어노테이션.

```java
@RestController //@Controller + @ResponseBody
@RequiredArgsConstructor
public class MemberApiController {

    private final MemberService memberService;

    //회원 수정
    @PutMapping("/api/v2/members/{id}")
    public UpdateMemberResponse updateMemberV2(@PathVariable Long id,
                                               @RequestBody @Valid UpdateMemberRequest request){
        memberService.update(id, request.getName());
        Member findMember = memberService.findOne(id);

        return new UpdateMemberResponse(findMember.getId(), findMember.getName());
    }

    @Data
    static class UpdateMemberRequest{
        private String name;
    }

    @Data
    @AllArgsConstructor
    static class UpdateMemberResponse{
        private Long id;
        private String name;
    }
}
```

# 회원 조회 API

---

### 회원 조회 V1

- 문제점
    1. 엔티티에 모든 정보가 화면에 노출된다.
    2. @JsonIgnore 어노테이션으로 몇몇 필드 노출을 막을 수 있지만 임시방편이다.
    3. 엔티티가 변경되면 API 스펙이 변한다.
    4. 컬렉션을 반환하면 API 스펙 확장이 어렵다.
- 결론 : API 스펙에 맞춘 DTO 로 반환하자 !
    
    ```java
    @RestController //@Controller + @ResponseBody
    @RequiredArgsConstructor
    public class MemberApiController {
    
        private final MemberService memberService;
    
        //회원 수정
        @GetMapping("/api/v1/members")
        public List<Member> membersV1(){
            return memberService.findMembers();
        }
    }
    ```

### 회원 조회 V2

- MemberDto를 하나 생성하고 이름만 노출시킨다는 가정하에 name 필드만 추가해준다.
    - id나 Addreess 등도 추가하면 된다.
- 우선 memberService로 List 타입인 Member를 다 가져오고 MemberDto로 전부 변환해준다.
- 그리고 Rsult 클래스로 한번더 감싸서 반환해주면 JSON에서 Array 자체를 넘겨주는게 아닌 확장성을 용이하게 아래와 같이 할 수 있따.
    
    ```java
    @RestController //@Controller + @ResponseBody
    @RequiredArgsConstructor
    public class MemberApiController {
    
        private final MemberService memberService;
    
        @GetMapping("/api/v2/members")
        public Result memberV2(){
            List<Member> findMembers = memberService.findMembers();
    
            // 엔티티 -> DTO
            // Java 8 버전으로 변경
    //        List<MemberDto> collect = findMembers.stream()
    //                .map(m -> new MemberDto(m.getName()))
    //                .collect(Collectors.toList());\
    
            List<MemberDto> collect = new ArrayList<>();
            for (Member findMember : findMembers) {
                collect.add(new MemberDto(findMember.getName()));
            }
    
            return new Result(collect);
        }
    
        @Data
        @AllArgsConstructor
        static class Result<T>{
            private T data;
            //private int count; 필요한 필드를 추가할 수 있다.
        }
    
        @Data
        @AllArgsConstructor
        static class MemberDto{
            private String name;
            // 필요한거만 노출하면 된다.
            //private Long id;
            //private Address address;
        }
    }
    ```
