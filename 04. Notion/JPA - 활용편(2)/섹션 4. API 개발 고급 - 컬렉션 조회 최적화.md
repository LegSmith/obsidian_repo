**==Order 엔터티와 컬렉션인 OrderItem , Item 엔터티를 같이 조회활때 어떻게 로직을 짜고 최적화를 어떻게 하는지 알아보자==**

## 주문 조회 V1 : 엔티티 직접 노출

---

==**이 방법은 실무에서 절대 사용해서는 안된다 Why? 엔터티의 스펙이 바뀌면 API 스펙 전체가 망가지기 때문이다 !**==

- 컬렉션이 아닌 엔터티를 조회할 때는 `orderRepository.findAllByStyring()` 으로 전체 조회 결과를 가져오고 forEach 를 돌려서 Lazy 강제 초기화를 해서 데이터를 출력했다.
- Lazy 초기화를 안하면 null 로 나온다.

    ![[Untitled 32.png|Untitled 32.png]]

- 컬렉션이 있으면 forEach 문안에서 한번더 forEach 문으로 Lazy 초기화를 해주면 된다.

    ```java
    @GetMapping("/api/v1/orders")
        public List<Order> ordersV1(){
            List<Order> all = orderRepository.findAllByString(new OrderSearch());
            for (Order order : all) {
                order.getMember().getName();
                order.getDelivery().getAddress();
                List<OrderItem> orderItems = order.getOrderItems();
                orderItems.stream()
                        .forEach(o->o.getItem().getName());
            }

            return all;
        }
    ```

    ![[Untitled 1 22.png|Untitled 1 22.png]]


> _**엔티티를 직접 반환하면 엔티티 정보들이 노출되기 때문에 향후에 엔티티 필드명이 바뀌거나 하면 API 스펙이 다 꺠진다.**_

## 주문 조회 V2 : 엔티티를 DTO 로 변환하여 노출

---

- 컬렉션이 있는 엔터티를 DTO 로 변환할 때는 DTO 안에 있는 엔터티도 한번 더 DTO 로 변환해야 하는 번거로움이 있다.
- **==이런식으로 Order 에 있는 OrderItem 은 컬렉션이기 때문에 OrderItem 을 한번더 DTO 로 감싸야 한다.==**

    ```java
    @Data
    static class OrderDto{

        private Long orderId;
        private String name;
        private LocalDateTime orderDate;
        private OrderStatus orderStatus;
        private Address address;
        private List<OrderItemDto> orderItems;

        public OrderDto(Order order) {
            orderId = order.getId();
            name = order.getMember().getName();;
            orderDate = order.getOrderDate();
            orderStatus = order.getStatus();
            address = order.getDelivery().getAddress();
            orderItems = order.getOrderItems().stream()
                    .map(o -> new OrderItemDto(o))
                    .collect(Collectors.toList());
        }
    }

    @Data
    static class OrderItemDto{

        private String itemName; // 상품명
        private int orderPrice; // 주문 가격
        private int count; // 주문 수량

        public OrderItemDto(OrderItem orderItem) {
            itemName = orderItem.getItem().getName();
            orderPrice = orderItem.getOrderPrice();
            count = orderItem.getCount();
        }
    }
    ```

- 배열을 JSON 출력할 때는 한번 감싸서 출력하는게 좋다.

    ```java
    @GetMapping("/api/v2/orders")
    public Result<?> ordersV2(){
        List<Order> orders = orderRepository.findAllByString(new OrderSearch());
        List<OrderDto> collect = orders.stream()
                .map(o -> new OrderDto(o))
                .collect(Collectors.toList());
        return new Result<>(collect.size(), collect);
    }

    // DTO 생략

    @Data
    @AllArgsConstructor
    static class Result<T>{
        private int count;
        private T data;
    }
    ```


## 주문 조회 V3 : 엔티티를 DTO로 변환 - 페치 조인 최적화

---

**==V2 에서는 지연로딩으로 인해 너무 많은 SQL이 실행됐다. 패치조인으로 쿼리를 최적화 시킬 수 있다.==**

- `OrderService` 코드에서 패치조인 로직을 담당하는 메서드를 작성

    ```java
    public List<Order> findAllWithItem() {
        return em.createQuery(
            "SELECT o FROM Order o" +
                        " join fetch o.member m" +
                        " join fetch o.delivery d" +
                        " join fetch o.orderItems oi" +
                        " join fetch oi.item i", Order.class)
                .getResultList();
    }
    ```

- 컨트롤러에서 `findAllWithItem()` 메서드를 실행시킨다.

    ```java
    @GetMapping("/api/v3/orders")
    public List<OrderDto> ordersV3(){
        List<Order> orders = orderRepository.findAllWithItem();
        return orders.stream()
                .map(o -> new OrderDto(o))
                .collect(Collectors.toList());
    }
    ```

- V2 에서 문제였던 여러개의 쿼리는 나가지 않고 1개의 쿼리만 나가게 된다.
- 하지만 출력되는 컬럼은 4개이다.

- 하지만 Order 와 OrderItem 은 1:N 조인이므로 Order 데이터가 중복이 생긴다.

    ![[Untitled 3 9.png|Untitled 3 9.png]]

- JPQL 에서 `distinct` 를 사용하면 같은 엔티티를 애플리케이션에서 중복을 걸러준다. (DB에서는 걸러지는게 아님)

    ```java
    public List<Order> findAllWithItem() {
        return em.createQuery(
            "SELECT distinct o FROM Order o" +
                        " join fetch o.member m" +
                        " join fetch o.delivery d" +
                        " join fetch o.orderItems oi" +
                        " join fetch oi.item i", Order.class)
                .getResultList();
    }
    ```

## 주문 조회 V4 : DTO 로 직접 조회

---

앞서 컬렉션이 아닌 엔티티를 DTO로 직접조회할 때는 원하는 JPQL 에서 new 키워드를 통해 원하는 컬럼만 뽑아서 마치 SQL 처럼 작성하면 된다. 하지만 컬렉션이 있는 DTO는 JPQL 을 2번 짜야한다.

- 우선 첫 JPQL 을 작성할 때는 컬렉션이 아닌 컬럼만 있는 DTO로 작성한다.

    ```java
    @Data
    public class OrderQueryDto {

        private Long orderId;
        private String name;
        private LocalDateTime orderDate;
        private OrderStatus orderStatus;
        private Address address;
        private List<OrderItemQueryDto> orderItems;

        public OrderQueryDto(Long orderId, String name, LocalDateTime orderDate, OrderStatus orderStatus, Address address) {
            this.orderId = orderId;
            this.name = name;
            this.orderDate = orderDate;
            this.orderStatus = orderStatus;
            this.address = address;
        }
    }

    --------------

    private List<OrderQueryDto> findOrders() {
        return em.createQuery(
                        "SELECT NEW jpabook.jpashop.repository.order.query.OrderQueryDto(o.id, m.name, o.orderDate, o.status, d.address)" +
                                " FROM Order o" +
                                " JOIN o.member m" +
                                " JOIN o.delivery d", OrderQueryDto.class)
                .getResultList();
    }
    ```

- 그런다음 컬렉션인 컬럼인 OrderItem 전용 DTO도 생성한다.

    ```java
    @Data
    public class OrderItemQueryDto {

        private Long orderId;
        private String itemName;
        private int orderPrice;
        private int count;

        public OrderItemQueryDto(Long orderId, String itemName, int orderPrice, int count) {
            this.orderId = orderId;
            this.itemName = itemName;
            this.orderPrice = orderPrice;
            this.count = count;
        }
    }
    ```

- 맨 처음 JPQL에서 결과값인 List를 forEach를 돌려 컬렉션을 찾는 OrderItem 쿼리문으로 setter 해준다.

    ```java
    public List<OrderQueryDto> findByOrderQueryDtos() {
        List<OrderQueryDto> result = findOrders();
        result.forEach(o->{
            List<OrderItemQueryDto> orderItems = findOrderItems(o.getOrderId());
            o.setOrderItems(orderItems);
        });
        return result;
    }

    private List<OrderItemQueryDto> findOrderItems(Long orderId) {
        return em.createQuery(
                "SELECT NEW jpabook.jpashop.repository.order.query.OrderItemQueryDto(oi.order.id, i.name, oi.orderPrice, oi.count)" +
                        " FROM OrderItem oi"+
                        " JOIN oi.item i" +
                        " WHERE oi.order.id = :orderId" , OrderItemQueryDto.class)
                .setParameter("orderId", orderId)
                .getResultList();
    }
    ```

- 즉, 정리하면 이렇게 된다.
    1. **컬렉션을 제외한 컬럼만 DTO로 조회한다.**
    2. **조회환 결과값을 List 를 돌려 데이터 하나당 OrderItem 을 찾는 쿼리문을 조회하여 setter 로 데이터를 넣어준다.**
- 이렇게 되면 쿼리는 루트 1번 , 컬렉션 N 번으로 실행된다.
- ToMany 관계는 조인하면 row 수가 증가하기 때문에 이런방법으로 하는게 좋다.

## 주문 조회 V5 : DTO 로 직접 조회 - 컬렉션 조회 최적화

---

```java
public List<OrderQueryDto> findAllByDto_optimization() {
    List<OrderQueryDto> result = findOrders();

    Map<Long, List<OrderItemQueryDto>> orderItemMap = findOrderItemMap(toOrderIds(result));

    result.forEach(o->o.setOrderItems(orderItemMap.get(o.getOrderId())));

    return result;
}

private Map<Long, List<OrderItemQueryDto>> findOrderItemMap(List<Long> orderIds) {
    List<OrderItemQueryDto> orderItems = em.createQuery("SELECT NEW jpabook.jpashop.repository.order.query.OrderItemQueryDto(oi.order.id, i.name, oi.orderPrice, oi.count)" +
                    " FROM OrderItem oi" +
                    " JOIN oi.item i" +
                    " WHERE oi.order.id in :orderIds", OrderItemQueryDto.class)
            .setParameter("orderIds", orderIds)
            .getResultList();

    Map<Long, List<OrderItemQueryDto>> orderItemMap = orderItems.stream()
            .collect(Collectors.groupingBy(OrderItemQueryDto::getOrderId));
    return orderItemMap;
}

private static List<Long> toOrderIds(List<OrderQueryDto> result) {
    List<Long> orderIds = result.stream()
            .map(o -> o.getOrderId())
            .collect(Collectors.toList());
    return orderIds;
}
```

- V4에서는 N+1 문제가 있었다. 왜냐하면 루트쿼리 1개와 관련되 OrderItem 갯수만큼 쿼리를 전송하기 때문이다.
- V5에서는 OrderItem 을 찾는 쿼리를 IN 키워드를 통해 1개만 날리고 IN 키워드에 넣을 데이터를 Map 컬렉션을 사용해 메모리에 띄어놓는 방법이다.
- 우선 루트 쿼리를 날리는 findOrders() 를 사용한다.(컬렉션을 빼고 우선 조회한다.)

    ```java
    private List<OrderQueryDto> findOrders() {
        return em.createQuery(
                        "SELECT NEW jpabook.jpashop.repository.order.query.OrderQueryDto(o.id, m.name, o.orderDate, o.status, d.address)" +
                                " FROM Order o" +
                                " JOIN o.member m" +
                                " JOIN o.delivery d", OrderQueryDto.class)
                .getResultList();
    }
    ```

- 그리고 orderId 만 뽑아서 List로 데이터를 뽑아온다.

    ```java
    private static List<Long> toOrderIds(List<OrderQueryDto> result) {
        List<Long> orderIds = result.stream()
                .map(o -> o.getOrderId())
                .collect(Collectors.toList());
        return orderIds;
    }
    ```

- 그리고 IN 키워드에 위에서 뽑은 데이터를 넣으면 된다. 그리고 Key 값이 id이고 value 가 데이터인 Map으로 뽑아서 반환

    ```java
    private Map<Long, List<OrderItemQueryDto>> findOrderItemMap(List<Long> orderIds) {
        List<OrderItemQueryDto> orderItems = em.createQuery("SELECT NEW jpabook.jpashop.repository.order.query.OrderItemQueryDto(oi.order.id, i.name, oi.orderPrice, oi.count)" +
                        " FROM OrderItem oi" +
                        " JOIN oi.item i" +
                        " WHERE oi.order.id in :orderIds", OrderItemQueryDto.class)
                .setParameter("orderIds", orderIds)
                .getResultList();

        Map<Long, List<OrderItemQueryDto>> orderItemMap = orderItems.stream()
                .collect(Collectors.groupingBy(OrderItemQueryDto::getOrderId));
        return orderItemMap;
    }
    ```

- 루트 쿼리로 가져온 데이터에 컬렉션에 값을 넣어주기만 하면 된다.

    ```java
    public List<OrderQueryDto> findAllByDto_optimization() {
        List<OrderQueryDto> result = findOrders();

        Map<Long, List<OrderItemQueryDto>> orderItemMap = findOrderItemMap(toOrderIds(result));

        result.forEach(o->o.setOrderItems(orderItemMap.get(o.getOrderId())));
    ```

## API 개발 고급 정리

---

### 권장 순서

1. **엔티티 조회 방식으로 우선 접근**
    1. **==패치조인으로 쿼리 수를 최적화==**
    2. 컬렉션이 있다면
        1. 페이징필요 → `hibernate.default_batch_fetch_size` , `@BatchSize` 로 최적화
        2. 페이징 불필요 → 페치 조인 사용
2. 엔티티 조회 방식으로 해결이 안되면 DTO 조회 방식 사용 → JPQL `new` 키워드 사용
3. NativeSQL or 스프링 JdbcTemplate
