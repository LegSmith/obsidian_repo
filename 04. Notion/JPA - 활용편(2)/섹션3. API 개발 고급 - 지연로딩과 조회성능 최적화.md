# 간단한 주문 조회 V1 : 엔티티를 직접 노출

---

- xxToOne 관계에 있는(컬렉션이 아닌) 엔티티를 조회하는 메서드이다.
- findAllByString() 메서드를 호출하여 그대로 출력하면 어떻게 되는가

    - findAllByString은 아무조건없이 모두 출력하는 메서드.

    ```java
    @RestController
    @RequiredArgsConstructor
    public class OrderSimpleApiController {

        private final OrderRepository orderRepository;

        @GetMapping("/api/v1/simple-orders")
        public List<Order> orderV1(){
            List<Order> all = orderRepository.findAllByString(new OrderSearch());
            return all;
        }
    }
    ```

- ==**무한루프가 돌게되면서 먹통이 된다.**==
- ==**왜냐하면 Order를 가보면 Member가 있고 Member에 가면 Order가 있기때문에 양방향 관계때문에 무한루프가 돌게 된다.**==
- 이를 막기위한 임시방편으로는 `@JsonIgnore`가 있다.
- 하지만 그렇게 해도 아래와 같은 오류가 뜬다.

    ![[Untitled 31.png|Untitled 31.png]]

- 오류 내용을 보면 `bytebuddy.ByteBuddyInterceptor` 라는 오류내용이 있다.
    - bytebuddy는 최근 프록시 기술 라이브러리이다.
    - Order 엔티티를 까보면 Member 엔티티가 Lazy(지연로딩) 전략을 사용하고 있다.
    - ==**따라서 실제 엔티티가 아닌 프록시로 존재하게 된다. 하지만 jackson 라이브러리는 실제 Member엔티티 타입으로만 알고있지 프록시 객체를 모르기 때문에 여기서 예외가 발생한다.**==

        ```java
        @Entity
        @Table(name = "orders")
        @Getter @Setter
        @NoArgsConstructor(access = AccessLevel.PROTECTED)
        public class Order {

            @Id @GeneratedValue
            @Column(name = "order_id")
            private Long id;

            @ManyToOne(fetch = LAZY)
            @JoinColumn(name = "member_id")
            private Member member;
        }
        ```

    - 이 때 `Hibernate5Module` 을 스프링 빈으로 등록하면 해결된다(스프링부트 2.xx 버전기준)
    - JpashopApplication 클래스에 추가해주자

        ```java
        @Bean
        	Hibernate5Module hibernate5Module() {
        		Hibernate5Module hibernate5Module = new Hibernate5Module(); //강제 지연 로딩 설정
        //		hibernate5Module.configure(Hibernate5Module.Feature.FORCE_LAZY_LOADING, true);
        		return hibernate5Module;
        	}
        ```

        ![[Untitled 1 21.png|Untitled 1 21.png]]

- ==**member , orderItems, delivery 가 null 로 나오는 이유는 Lazy(지연로딩) 전략이기 때문이다.**==
- Hibernate5Module 기본 전략이 지연로딩은 그냥 무시하고 출력하기 때문에 null로 되있다.
- 강제로 지연로딩을 설정할 수 있다.

    ```java
    @Bean
    	Hibernate5Module hibernate5Module() {
    		Hibernate5Module hibernate5Module = new Hibernate5Module(); //강제 지연 로딩 설정
    		hibernate5Module.configure(Hibernate5Module.Feature.FORCE_LAZY_LOADING, true);
    		return hibernate5Module;
    	}
    ```

    ![[Untitled 2 12.png|Untitled 2 12.png]]

- 이렇게 되면 하나의 주문에 모든 정보들이 노출된다.
- 하지만 이런방법은 좋지 않은 방법이다. ==**엔티티의 정보들을 노출시키게 되면 나중에 API 스펙에 추가 노출이 되고 성능상의 문제도 있다.**==
- 필요한 정보들만 노출시키는 방법이 있다.
- order.getMember() 까지는 Lazy 강제 초기화가 아니다. ==**getName() 부분부터 Lazy 강제 초기화이다.**==
- 원하는 내용만 골라서 출력하게 된다.

    - `hibernate5Module.configure(Hibernate5Module.Feature.FORCE_LAZY_LOADING, true);` 이 부분은 주석 처리 해줘야 한다.

    ```java
    @RestController
    @RequiredArgsConstructor
    public class OrderSimpleApiController {

        private final OrderRepository orderRepository;

        @GetMapping("/api/v1/simple-orders")
        public List<Order> orderV1(){
            List<Order> all = orderRepository.findAllByString(new OrderSearch());
            for (Order order : all) {
                order.getMember().getName(); //Member의 name을 건들면서 Lazy 강제 초기화(쿼리를 날린다)
                order.getDelivery().getAddress(); //Lazy 강제 초기화
            }

            return all;
        }
    }
    ```

### 정리 !!

1. 엔티티를 직접 노출할 때는 양방향 연관관계가 걸린 곳은 꼭 한곳에 @JsonIgnore 처리를 해줘야 한다.
2. 정말 혼자하는 간단한 어플리케이션이 아니라면 ==**엔티티를 API 응답으로 외부에 노출시키는것은 좋지않다. DTO 로 변환해서 노출시키는게 가장 좋은 방법**==

# V2 : 엔티티를 DTO 로 전환

---

### DTO 설계

- 배송아이디, 주문자이름, 주문날짜, 주문상태, 배송지를 받는 API 라고 가정하자

    ```java
    @Data
        static class SimpleOrderDto{
            private Long orderId;
            private String name;
            private LocalDateTime orderDate;
            private OrderStatus orderStatus;
            private Address address; // 배송지 정보

            public SimpleOrderDto(Order order){
                orderId = order.getId();
                name = order.getMember().getName(); // LAZY 초기화
                orderDate = order.getOrderDate();
                orderStatus = order.getStatus();
                address = order.getDelivery().getAddress(); // LAZY chrlghk
            }
        }
    ```

- 우선 V1과 똑같이 orderRepository를 통해 주문조회 리스트를 받는다
- 자바8 stream을 이용하여 SimpleOrderDto 타입인 리스트로 받아서 뿌려주면된다.

    - for 문을 통해 변환해도 된다.
    - 원래는 List로 뿌리면 API 스펙확장에 문제가 생기기 때문에 한번더 감싸서 뿌려줘야 한다.

    ```java
    @GetMapping("/api/v2/simple-orders")
        public List<SimpleOrderDto> orderV2(){
            //ORDER 2개 조회(N : 2개)
            //N + 1 -> 1 + 회원*N + 배송*N
            List<Order> orders = orderRepository.findAllByString(new OrderSearch());

            //Entity -> DTO
            // 루프 1번당 2개의 쿼리(Member, Delivery) 가 나간다.
            List<SimpleOrderDto> result = orders.stream()
                    .map(o -> new SimpleOrderDto(o))
                    .collect(Collectors.toList());

            return result;

        }
    ```

- 하지만 여기서 N + 1 문제가 생긴다.
- 맨 처음 orderRepository로 받을때 쿼리가 1개 나간다.
- 그리고 2번의 루프가 돌면서 총 4개의 쿼리( Member 2개, Delivery 2개)가 나간다.

    - Member 와 Delivery는 Order 의 Lazy(지연로딩) 전략에 의해서 나간다.

# V3 : 페치 조인 최적화

---

==**V2 에서는 지연로딩 때문에 N+1 문제가 있었다, 이를 패치조인으로 한방쿼리를 이용하여 지연로딩이 안 일어나게 막을 수 있다.**==  
==**(Eager 전략으로 하면 안된다 ! )**==

- V2와 코드는 같고 orderRepository 로 불러오는 메서드 로직만 달라진다.

    ```java
    @GetMapping("/api/v3/simple-orders")
    public List<SimpleOrderDto> orderV3(){
        List<Order> orders = orderRepository.findAllWithMemberDelivery();
        List<SimpleOrderDto> result = orders.stream()
                .map(o -> new SimpleOrderDto(o))
                .collect(Collectors.toList());

        return result;
    }
    ```

- 패치조인을 이용하여 Order와 관련된 Member 와 Delivery 를 한번에 가져와서 반환한다.

    ```java
    public List<Order> findAllWithMemberDelivery() {
        return em.createQuery(
                "select o from Order o" +
                        " join fetch o.member m" +
                        " join fetch o.delivery d", Order.class
        ).getResultList();
    }
    ```

- 결과는 같지만 쿼리1개만 나가는걸 확인할 수 있다.

# V4 : JPA에서 DTO로 바로 조회

---

> _**기존 방식은 엔티티를 조회하고 엔티티를 DTO로 전환한 다음 뿌려줬었다.**_

==**기존 V1,2,3 는 OrderRepository를 직접 조회했지만 Repository는 엔티티만 건드는게 좋으므로 Repository에 order.simplequery라는 계층을 만들어 여기에 Dto로 바로조회하는 레포지토리를 생성한다 !!**==

- 이 DTO의 생성자에서 Order order를 안받는 이유는 아래의 ==**JPQL에서 Order로 받아버리면 o 자체가 o.id(식별자)로만 인식이 되서 orderId만 넘어가게 되기 때문에 필요한 값들을 다 적는다**==

    ```java
    @Data
    public class OrderSimpleQueryDto {

        private Long orderId;
        private String name;
        private LocalDateTime orderDate;
        private OrderStatus orderStatus;
        private Address address; // 배송지 정보

        public OrderSimpleQueryDto(Long orderId, String name, LocalDateTime orderDate, OrderStatus orderStatus, Address address){
            this.orderId = orderId;
            this.name = name;
            this.orderDate = orderDate;
            this.orderStatus = orderStatus;
            this.address = address;
        }
    }
    ```

- 쿼리를 보면 new 키워드가 생성된 걸 알 수 있다.

    - JPQL은 엔티티만 건들수 있고 DTO는 직접 건들지 못하기에 new 키워드를 사용하여 DTO패키지명과 반환값들을 적어준다.

    ```java
    @Repository
    @RequiredArgsConstructor
    public class OrderSimpleQueryRepository {

        private final EntityManager em;

        public List<OrderSimpleQueryDto> findOrderDtos() {
            return em.createQuery(
                            "select new jpabook.jpashop.repository.order.simplequery.OrderSimpleQueryDto(o.id,m.name,o.orderDate,o.status,d.address) "+
                                    "from Order o" +
                                    " join o.member m" +
                                    " join o.delivery d", OrderSimpleQueryDto.class)
                    .getResultList();

        }
    }
    ```

- 1개의 쿼리가 나가면서 모든 컬럼을 갖고오는게 아닌 DTO로 뽑아내고싶은 컬럼만 가져오는 쿼리가 되어 성능면에서 더 낫다.

## 정리

1. V3와 V4의 성능차이는 생각보다 크게 나지 않는다.
2. 리포지토리의 재사용성이 떨어진다(API 스펙에 맞춘 코드가 리포지토리에 들어가기 때문에)
3. 가장 좋은방법
    1. ==**엔티티를 DTO로 변환하는 방법을 최우선으로 하자.**==
    2. 필요하면 패치 조인으로 성능을 최적화하자!(대부분 성능 이슈 해결)
    3. 그래도 안된다면 DTO로 직접조회
    4. 그래도 안되면 JPA에서 제공하는 네이티브 SQL 이나 JDBC Template를 사용하여 직접 SQL 사용
