`OSIV(Open Session In View)` : Hibernate 에서 사용되는 언어  
`Open EntityManager In View` : JPA 에서 사용 되는 언어 → 관례상 ==**OSIV**== 라 한다.

> ==_명확한 이해가 필요하다!! 잘못사용하면 장애가 발생한다!!_==

## OSIV ON

---

![[Untitled 33.png|Untitled 33.png]]

- `spring.jpa.open-in-view` : true 가 기본값이다.
- 어플리케이션이 ==**데이터베이스 커넥션을 가져오는 시점은 영속성 컨텍스트가 트랜잭션을 실행시킬 때 데이터베이스 커넥션을 가져온다.**==
- **==OSIV 가 true 라면 트랜잭션이 종료되도 데이터베이스 커넥션을 반환하지 않는다 !!==** → 컨트롤러 단에서 Lazy 로딩초기화 같은것을 사용할 수 있는 이유이다 !!!
- 즉 OSIV를 true 로 설정하면 API 라면 클라이언트에게 반환할 때까지, View Templates 라면 View를 랜더링할때까지 데이터베이스 커넥션을 가지고 있게 한다.
- 지연로딩(Lazy)은 영속성 컨텍스트가 살아있어야 가능하고, 영속성 컨텍스트는 기본적으로 데이터베이스 커넥션을 유지한다.
- 하지만 **==실시간 트래픽이 중요한 애플리케이션에서는 커넥션이 모자랄 수 있다.==**

## OSIV OFF

---

![[Untitled 1 23.png|Untitled 1 23.png]]

- OSIV 를 끄게되면 트랜잭션이 종료되는 시점에 영속성 컨텍스트를 닫고 데이터베이스 커넥션을 반환한다. 즉, 실시간 트래픽이 중요한 어플리케이션에선 커넥션 리소스를 낭비하지 않기 때문에 장점이 된다.
- 다만, 지연로딩을 트랜잭션 안에서 처리해야 하는 단점이 있다.

### OSIV OFF 시 복잡성을 해결

- 커멘드와 쿼리를 완벽히 분리한다.
- 컨트롤러에 있던 쿼리 로직을 따로 클래스로 빼서 트랜잭션을 걸로 로직을 짜면 된다.
- `OrderService`
    - `OrderService`: 핵심 비즈니스 로직
    - `OrderQueryService`: 화면이나 API에 맞춘 서비스 (주로 읽기 전용 트랜잭션 사용)
