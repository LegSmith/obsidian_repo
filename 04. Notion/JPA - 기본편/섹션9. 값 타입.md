# 기본값 타입

---

### JPA 최상위 타입

- 엔티티 타입
    - @Entity로 정의하는 객체
    - 데이터가 변해도 식별자(ID값)로 지속해서 ==**추적 가능**==
- 값 타입
    - int, Integer, String처럼 값으로 사용하는 자바 기본타입이나 객체
    - 식별자가 없고 값만 있으므로 ==**추적 불가**==

### 값 타입 분류

- **기본값 타입**
    - 자바 기본 타입(int, double 등등)
    - 래퍼 클래스(Integer, Long 등등)
    - String
- **임베디드 타입(embedded type, 복합 값 타입)**
- **컬렉션 값 타입(collection value type)** → ==**자바컬렉션에 기본값타입이나 임베디드 타입을 넣는것**==

### 기본값 타입(String name, int age 등등)

- ==**생명주기를 엔티티의 의존**==
    - 예) 회원엔티티를 삭제하면 이름, 나이 필드도 함께 삭제
- 값 타입은 공유하면 안된다
    
    > _**자바 기본 타입은 절대 공유하면 안된다!!**  
    > **Integer같은 래퍼 클래스는 공유는 가능하지만 변경X**_
    

# 임베디드 타입

---

- 새로운 값 타입을 직접 정의할 수 있다.
    - Member엔티티의 street, city, zipcode 와 같이 주소를 세분화한 값들은 애플리케이션 전체에서 재사용이 가능하기 때문에 Address라는 클래스고 정의할 수 있다.
- 기본 값 타입을 모아서 만들었기 때문에 복합 값 타입이라고 한다.

### 임베디드 타입 사용법

- `@Embeddable` : 값 타입을 정의하는 곳(클래스)에 표시
- `@Embedded` : 값 타입을 사용하는 곳에 표시
- ==**기본 생성자 필수**==

### 임베디드 타입과 테이블 매핑

- ==**임베디드 타입을 사용하기 전과 후에 매핑하는 테이블은 변화가 없다.**==
- ==**잘 설계한 ORM 애플리케이션은 매핑한 테이블의 수보다 클래스의 수가 더 많음.**==

# 값 타입과 불변 객체

---

- 하나의 임베디드 값 타입을 여러 엔티티가 공유하고 있을 시에 부작용(side effect)가 생긴다. 
    
    ```java
    Address address = new Address("City", "Street", "10000");
    
    Member member = new Member();
    member.setUsername("Ronaldo");
    member.setHomeAddress(address);
    em.persist(member);
    
    Member member2 = new Member();
    member2.setUsername("Messi");
    member2.setHomeAddress(address);
    em.persist(member2);
    
    member.getHomeAddress().setCity("newCity");
    ```
    
- member 엔티티에 city값만 수정하고 싶지만 address는 객체타입이기 때문에 인스턴스 값을 공유하고 있다. 그래서 address를 공유하고 있는 모든 엔티티의 값이 수정된다.
- 두 번의 update쿼리가 나가면서 공유하고 있는 2개의 엔티티의 값이 수정된다 !!
    
    ![[Untitled 27.png|Untitled 27.png]]
    
- 그래서 임베디드 값 타입을 복사하여 사용하면 된다.
    
    ```java
    Address address = new Address("City", "Street", "10000");
    
    Member member = new Member();
    member.setUsername("Ronaldo");
    member.setHomeAddress(address);
    em.persist(member);
    
    //부작용을 막기 위해서는 새로운 엔티티를 생성해서 써야한다.
    Address copyAddress = new Address(address.getCity(), address.getStreet(), address.getStreet());
    Member member2 = new Member();
    member2.setUsername("Messi");
    member2.setHomeAddress(copyAddress);
    em.persist(member2);
    
    member.getHomeAddress().setCity("newCity");
    ```
    
- ==**하지만 자바에서는 객체 타입의 공유참조를 피할 수 없다. (누군가는 반드시 실수할 수 있다.)**==

### 불변 객체

- 객체 타입을 수정할 수 없게 만들면 부작용을 원천 차단할 수 있다.
- 값 타입은 ==**불변 객체(immutable object)로 설계**==해야 한다.
- ==**불변객체란 생성 시점 이후 절대 값을 변경할 수 없는 객체이다.**==
- 불변객체는 ==**생성자로만 값을 설정하고 수정자(setter)를 만들지 않으면 된다.**==

  

==**즉, 값 타입은 무조건 불변객체로 만들자 !! 나중에 오류 추적하기가 어렵다 !!**==

# 값 타입 비교

---

- 동일성(identity) 비교 : 인스턴스의 참조값을 비교한다. ( == 연산자 사용 )
    - int와 같은 자바 기본 타입은 참조값이 없고 실제 값을 비교하므로 true를 반환하지만
    - Address와 같은 객체 타입은 참조값을 가지기 때문에 == 비교시 false 를 반환한다.
        
        ```java
        int a = 10;
        int b = 10;
        
        System.out.println("a == b : "+ (a==b));
        
        Address address1 = new Address("city", "street", "10000");
        Address address2 = new Address("city", "street", "10000");
        
        // 동일성(identity) 비교 : 인스턴스의 참조 값을 비교 , == 연산자
        System.out.println("address1 == address2 : " + (address1==address2));
        ```
        
- 동등성(equivalence) 비교 : 인스턴스의 값을 비교한다. ( equlas() 메서드 사용 )
    - 주의할 점은 Java에서 제공하는 기본 equlas() 메서드를 사용하면 false를 반환한다.
    - 비교하는 객체에서 equals() 메서드를 오버라이딩(재정의)하여 사용해야 한다.
        
        ```java
        int a = 10;
        int b = 10;
        
        System.out.println("a == b : "+ (a==b));
        
        Address address1 = new Address("city", "street", "10000");
        Address address2 = new Address("city", "street", "10000");
        
        // 동일성(identity) 비교 : 인스턴스의 참조 값을 비교 , == 연산자
        System.out.println("address1 == address2 : " + (address1==address2));
        // 동등성(equivalence) 비교 : 인스턴스의 값을 비교, equlas()사용
        // equals() 메서드는 비교하는 클래스에서 재정의하여 사용해햐 한다.
        System.out.println("address1 equals address2 : " + (address1.equals(address2)));
        ```
        

# 값 타입 컬렉션

---

==값 타입을 컬렉션으로 사용하는 것을 말한다.==

> 객체지향에서 List , Set같은 컬렉션은 관계형 DB에서 매칭이 안된다.  
> 즉, 별도의 테이블을 생성해야 한다.

- ==**값 타입을 하나 이상 저장할 때 사용**==
- `@ElementCollection` , `@CollectionTable` 사용

### 값 타입 저장

- 따로 컬렉션을 persist할 필요 없이 애플리케이션에서 컬렉션에 데이터를 추가하고 메인 엔티티만 persist하면 컬렉션으로 만든 값 타입들도 insert 문이 나간다.
    
    ```java
    Member member = new Member();
    member.setUsername("Ronaldo");
    member.setHomeAddress(new Address("City1", "Street", "100000"));
    
    member.getFavoriteFoods().add("닭가슴살");
    member.getFavoriteFoods().add("고구마");
    member.getFavoriteFoods().add("브로콜리");
    
    member.getAddressesHistory().add(new Address("Old1", "Street", "100000"));
    member.getAddressesHistory().add(new Address("Old2", "Street", "100000"));
    
    em.persist(member);
    
    
    tx.commit(); //트랜잭션 종료
    ```

### 값 타입 조회

- 조회를 하면 쿼리문은 Member 쿼리문 1개만 나온다.
- ==**즉, 컬렉션 값 타입은 기본이 지연로딩(LAZY)전략이다.**==

### 값 타입 수정

- 수정자(Setter)를 사용하면 부작용이 생기므로 컬렉션에 있는 값을 지우고 새로 넣는 방법밖에 없다.
- member 클래스를 조회할때 select 쿼리 1개
- Old1인 데이터를 삭제할때 delete 쿼리1개, 새로운 데이터를 add 할때 insert 쿼리1개가 나가야하는데 2개가 나간다 왜 ????
    
    ```java
    Member member = new Member();
    member.setUsername("Ronaldo");
    member.setHomeAddress(new Address("City1", "Street", "100000"));
    
    member.getFavoriteFoods().add("닭가슴살");
    member.getFavoriteFoods().add("고구마");
    member.getFavoriteFoods().add("브로콜리");
    
    member.getAddressesHistory().add(new Address("Old1", "Street", "100000"));
    member.getAddressesHistory().add(new Address("Old2", "Street", "100000"));
    
    em.persist(member);
    
    em.flush();
    em.clear();
    
    System.out.println("================Strat===============");
    Member findMember = em.find(Member.class, member.getId());
    
    //old1 -> new1
    //컬렉션마다 다르긴하지만 똑같은 Object를 가져와서 equals를 하기 때문에 그대로 가져와야한다.
    findMember.getAddressesHistory().remove(new Address("Old1", "Street", "100000"));
    findMember.getAddressesHistory().add(new Address("new1", "Street", "100000"));
    ```

### 값 타입 컬렉션의 제약사항

- ==**값 타입 컬렉션에 변경 사항이 발생하면, 주인 엔티티와 연관된 모든 데이터를 삭제하고, 값 타입 컬렉션에 있는 현재 값을 모두 다시 저장한다.**==
- 즉, 데이터를 삭제하는 기능이므로 쓰면 안된다….
- 실무에서는 ==**값 타입 컬렉션 대신에 일대다 관계를 고려하는것이 좋다.**==
