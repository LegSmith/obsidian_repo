## 전략 별 장단점

---

### 조인 전략

- 장점
    - 테이블이 정규화 되어있다.
    - 저장공간이 효율화
- 단점
    - 조회시 조인을 많이 사용하기 때문에 성능이 저하된다. → 조회 쿼리 복잡
    - 데이터 저장시 INSERT SQL 2번 호출

### 단일 테이블 전략

- 장점
    - 조인이 필요 없으므로 조회 성능이 빠르다. → 조회 쿼리 단순
- 단점
    - 자식 엔티티가 매핑한 컬럼은 모두 null 허용
    - 한 테이블에 다 때려박았기 대문에 테이블이 커질 수 있고 상황에 따라서 조회 성능이 오히려 느려질 수 있음.

# **@MappedSuperclass**  

==**공통 매핑 정보**==가 필요할 때 사용한다. (id , name 등)

- 공통속성을 가지는 클래스를 생성해서 `@MappedSuperclass` 어노테이션을 붙히면 된다.
- 그리고 엔티티를 extends하면 된다.
    
    ```java
    package hellojpa;
    
    import javax.persistence.MappedSuperclass;
    import java.time.LocalDateTime;
    
    @MappedSuperclass
    public class BaseEntity {
    
        private String createdBy; //누가 생성했는지
        private LocalDateTime createdDate; //언제 생성했는지
        private String modifiedBy; //누가 수정했는지
        private LocalDateTime lastModifiedDate; //마지막 수정시간
    
        public String getCreatedBy() {
            return createdBy;
        }
    
        public void setCreatedBy(String createdBy) {
            this.createdBy = createdBy;
        }
    
        public LocalDateTime getCreatedDate() {
            return createdDate;
        }
    
        public void setCreatedDate(LocalDateTime createdDate) {
            this.createdDate = createdDate;
        }
    
        public String getModifiedBy() {
            return modifiedBy;
        }
    
        public void setModifiedBy(String modifiedBy) {
            this.modifiedBy = modifiedBy;
        }
    
        public LocalDateTime getLastModifiedDate() {
            return lastModifiedDate;
        }
    
        public void setLastModifiedDate(LocalDateTime lastModifiedDate) {
            this.lastModifiedDate = lastModifiedDate;
        }
    }
    ```
    

### 정리

- 상속관계 매핑이랑 관계없다.
- 엔티티가 아니기 때문에 DB 테이블과 매핑이 안된다. → 그냥 클래스임.
- 당연히 조회도 안된다. → ==**상속관계 매핑에서는 조회가 됐다(엔티티이기 때문에)**==
- ==**추상클래스**==로 만드는게 추천. 왜냐하면 직접 생성해서 사용할 일이 없으므로
