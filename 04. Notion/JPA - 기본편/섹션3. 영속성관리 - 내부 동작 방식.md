### JPA 에서 가장 중요한 2가지

1. 매핑하기 ← 정적인 부분
2. ==**영속성 컨텍스트**==

## 영속성 컨텍스트

==**“엔티티를 영구 저장하는 환경”**== 이라는 뜻  
`EntityManager.persist(entity);`←DB에 저장하는게 아닌 영속성 컨텍스트에 저장

- 영속성 컨텍스트는 ==**논리적인 개념**==
- `EntityManger` 를 통해서 영속성 컨텍스트에 접근

## 엔티티의 생명주기

- **비영속(new / transient)** : 새로운 상태, ==**객체를 새로 생성한 상태**==
- **영속(managed)** : **==persist()==** 메서드에 객체를 넘겨주면 영속상태
- **준영속(detached)** : ==**영속성 컨텍스트에 저장되었다가 분리**==된 상태,
    - `EntityManager.detach(entity);` 로 객체를 준영속상태로 만든다.
- **삭제(removed)**
    - `EntityManager.remove(entity);` 로 객체를 삭제한다.

## 1차 캐시

- `persist(entity);` 시점에 영속성 컨텍스트에 1차캐시 라는 곳에 entity가 저장이된다.
- 그리고 나중에 `find(Entity.class,”Entity 2”);` 로 _**==객체를 찾을 때 가장 먼저 1차 캐시에서 조회를 한다==**_
- 객체가 있으면 1차캐시에서 반환해주고 _**==없다면 DB조회를 해서 데이터를 가져온 후 1차캐시에 저장 후 반환한다.==**_

```java
Member member = new Member();
member.setId(101L);
member.setName("Son");

em.persist(member);

Member findMember = em.find(Member.class, 101L);

System.out.println("findMember.id = " + findMember.getId());
System.out.println("findMember.name = " + findMember.getName());
```

- persist()로 영속화 시켜 1차캐시에 저장후
- 영속화 시킨 객체를 find() 로 찾으면 select 쿼리문이 나갈까 ?
    
    ![[Untitled 23.png|Untitled 23.png]]
    
- 나가지 않는다. **==JPA는 1차캐시에서 조회 후 데이터가 있으면 반환하기 때문에 DB조회를 하지 않는다. 즉, select 쿼리문이 나가지 않는다.==**

```java
Member findMember1 = em.find(Member.class, 101L);
Member findMember2 = em.find(Member.class, 101L);
      
tx.commit(); //트랜잭션 종료
```

- 객체를 영속화 하지않고(1차캐시에 저장되지 않았다, 영속화(persist)를 하지 않았기 때문에) 객체를 2번 조회한다면 쿼리문이 2개나갈까 ?
    
    ![[Untitled 1 15.png|Untitled 1 15.png]]
    
- 첫 조회때 1차캐시에 없어서 DB에서 조회를 한다 → 쿼리문이 나감
- 두번째 조회때 1차캐시에 있기 때문에 쿼리문이 나가지 않는다.

  

## 영속 엔티티의 동일성 보장

```java
Member findMember1 = em.find(Member.class, 101L);
Member findMember2 = em.find(Member.class, 101L);

System.out.println("result = " + (findMember1 == findMember2));
// 동일성 비교 true
```

- 1차 캐시가 있기에 가능하다.

  

## 엔티티 등록시에 쓰기 지연

- 객체를 영속화를 하는 시점에 Insert 쿼리문이 DB로 가지않고, commit() 시점에 쿼리문이 DB로 전송된다.
- ==**persist() 할 때마다 insert 쿼리문이 쓰기지연 SQL 저장소에 쌓는다.**==
- commit() 시점에 한번에 DB로 쿼리문이 전송된다. → `flush`
    
    ```java
    Member member1 = new Member(150L, "A");
    Member member2 = new Member(151L, "B");
    
    em.persist(member1); //insert 쿼리문이 쓰기지연 Sql 저장소에 쌓임
    em.persist(member2); //insert 쿼리문이 쓰기지연 Sql 저장소에 쌓임
    System.out.println("===========================");
    //commit() 시점에 쿼리문이 나간다.
    tx.commit(); //트랜잭션 종료
    ```

## 변경 감지 (Dirty Checking)

- JPA 는 마치 객체를 다루듯이 자바코드로 객체를 수정해주면 실제 DB에 있는 데이터도 수정이된다.
    
    ```java
    Member findMember = em.find(Member.class, 150L);
    
    System.out.println("==== findMember.name 수정 전 ====");
    System.out.println("findMember = " + findMember.getName());
    
    System.out.println("==== findMember.name 수정 후 ====");
    findMember.setName("zzzzzzzzzzz");
    System.out.println("findMember = " + findMember.getName());
    
    System.out.println("========================");
    //commit() 시점에 쿼리문이 나간다.
    tx.commit(); //트랜잭션 종료
    ```
    
    ![[Untitled 3 4.png|Untitled 3 4.png]]
    
- commit()을 하면 내부적으로 flush()가 호출이 된다
- 엔티티와 스냅샷을 비교하여 변경을 감지하여 변경이 있을시 update 쿼리문을 쓰기지연 SQL 저장소에 저장후 DB에 쿼리문을 전송한다

## 플러시(flush)

**영속성 컨텍스트의 변경내용을 데이터베이스에 반영**

### 플러시 발생

- 변경 감지
- 수정된 엔티티 쓰기지연 SQL 저장소에 등록

### 영속성 컨텍스트를 플러시하는 방법

- `em.flush()` → 직접 호출
- **트랜잭션 커밋** → 플러시 자동 호출
- **JPQL 쿼리 실행** → 플러시 자동 호출

```java
//트랜잭션 commit()전에 쿼리문이 날라가는걸 보고싶다면 ?
Member member = new Member(400L, "member@2400");
em.persist(member);

System.out.println("==========flush 호출===========");
em.flush(); //강제로 flush 호출되서 쿼리문이 db로 날라감
//commit() 시점에 쿼리문이 나간다.
System.out.println("============트랜잭션 commit전에 쿼리문 전송");
tx.commit(); //트랜잭션 종료
```

### 플러시 모드 옵션

- 기본값은 JPQL쿼리를 실행하거나 트랜잭션 커밋시에 플러시가 되는것이 기본값이지만
- `em.setFlushMode(FlushModeType.COMMIT)` **를 지정해주면 커밋할 때만 플러시이다.**

  

플러시는 !!  
영속성 컨텍스트를 비우지 않는다**영속성 컨텍스트의 변경내용을 데이터베이스에 동기화!!**

  

## 준영속 상태

- 영속 → 준영속
- 즉, 영속성 상태(JPA가 관리)에서 관리되지 않은 상태이다.
- `detach(entity)` 메서드를 이용하여 ==**특정 엔티티를 준영속 상태로 전환한다.**==
    
    ```java
    Member findMember = em.find(Member.class, 2L); //이때 findMember가 영속상태가 된다.
    findMember.setName("Cristiano");
    
    em.detach(findMember); //준영속 상태로 전환
    
    //트랜잭션 커밋을해도 아무일도 일어나지 않는다.
    System.out.println("==================");
    tx.commit(); //트랜잭션 종료
    ```
    
    ![[Untitled 6 2.png|Untitled 6 2.png]]
    
- `EntityManager.clear();` 메서드는 ==_**영속성 컨텍스트를 완전히 초기화한다. 즉, 1차캐시를 비운다는 말과 같다.**_==
- `EntityManager.close();` 메서드는 ==_**영속성 컨텍스트를 종료**_==한다.
