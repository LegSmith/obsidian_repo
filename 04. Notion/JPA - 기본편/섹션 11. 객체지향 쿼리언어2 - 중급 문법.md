# JPQL - 경로 표현식

---

## 경로 표현식

- .(점)을 찍어 객체 그래프를 탐색 하는것

    ```sql
    select m.username -> 상태 필드
    from Member m
    	join m.team t -> 단일 값 연관 필드
    	join m.orders o -> 컬렉션 값 연관 필드
    where t.name = '팀A'
    ```

- 상태 필드(state filed) : 단순히 값을 저장하기 위한 필드( ex : m.username )
- 연관 필드(association field) : 연관관계를 위한 필드
    - 단일 값 연관 필드 : @ManyToOne, @OneToOne, 대상이 엔티티(ex : m.team)
    - 컬렉션 값 연관 필드 : @OneToMany , @ManyToMany , 대상이 컬렉션(ex : m.orders )

## 경로 표현식 특징

- 상태 필드(state field) : 경로 탐색의 끝, 탐색X
- 단일 값 연관 경로 : ==**묵시적 내부 조인(inner join)발생**==, 탐색O
- 컬렉션 값 연관 경로 : 묵시적 내부 조인 발생, ==**탐색X**==

### 묵시적 내부 조인

- 단일 값 연관 경로를 사용하였을 때 사용자의 의도와 관계없이 묵시적 내부 조인이 발생한다.

    ```java
    Member member = new Member();
    member.setUsername("Cristiano");
    member.setAge(37);
    em.persist(member);

    em.flush();
    em.clear();

    String query = "SELECT m.team From Member m";
    List<Team> result = em.createQuery(query, Team.class).getResultList();
    ```

    ![[Untitled 29.png|Untitled 29.png]]

- 나중에 SQL 튜닝할때 많이 불편하기 때문에 묵시적 내부조인은 조심히 써야한다.

## 명시적 조인, 묵시적 조인

- 명시적 조인 : join 키워드 직접 사용
    - `SELECT m FROM Member m` **`join m.team t`**
- 묵시적 조인 : 경로 표현식에 의해 묵시적으로 SQL 조인 발생( 내부 조인만 가능)
    - `SELECT` **`m.team`** `FROM Member m`

실무에서는 묵시적 조인을 사용하면 SQL 튜닝시 힘들기 때문에 ==**웬만하면 명시적 조인을 하자 !!!**==

## 경로 탐색을 사용한 묵시적 조인 시 주의사항

- ==**항상 내부 조인**==
- 컬렉션은 경로 탐색의 끝.
- ==**즉, 가급적 묵시적 조인 대신에 명시적 조인 사용**==
- 조인은 SQL 튜닝에 중요 포인트

# 페치조인 1 - 기본

---

==**JPQL - 페치 조인(fetch join) : 실무에서 매우 중요함 !!!**==

## 페치조인(Fetch Join)

- SQL 조인이 아니다
- ==**JPQL 에서 성능 최적화를 위해 제공하는 기능**==
- **==연관된 엔티티나 컬렉션을 SQL 한 번에 함께 조회==**하는 기능
- `join fetch` 명렁어 사용


### 페치조인을 사용하지 않을때

- 아래와 같은 테이블 설계일때 팀이 있는 회원만 조회한다고 가정하자

    ![[Untitled 1 19.png|Untitled 1 19.png]]

    ```java
    Team teamA = new Team();
    teamA.setName("팀A");
    em.persist(teamA);

    Team teamB = new Team();
    teamB.setName("팀A");
    em.persist(teamB);

    Member member1 = new Member();
    member1.setUsername("회원1");
    member1.setTeam(teamA);
    em.persist(member1);

    Member member2 = new Member();
    member2.setUsername("회원2");
    member2.setTeam(teamA);
    em.persist(member2);

    Member member3 = new Member();
    member3.setUsername("회원3");
    member3.setTeam(teamB);
    em.persist(member3);

    em.flush();
    em.clear();

    String query = "SELECT m From Member m";
    List<Member> result = em.createQuery(query, Member.class).getResultList();

    for (Member member : result) {
        System.out.println("member = " + member.getUsername() + ", " + member.getTeam().getName());
    }

    tx.commit(); //트랜잭션 종료
    ```

- Member 엔티티의 username만 가져올 때는 Team 엔티티를 지연로딩 전략을 사용했기 때문에 Member 를 조회하는 SQL 쿼리만 나간다.
- 하지만 Team이름까지 가져오는 코드가 있을 때는 해당 코드 실행시에 Team을 조회하는 SQL 이 나가게 된다.

    ![[Untitled 2 11.png|Untitled 2 11.png]]

- 팀A에 소속된 멤버가 2명이므로 팀A는 한번만 SQL쿼리가 나가게 되서 총 3번의 쿼리가 나간다. 최악의 경우는 4번(각각 다른팀일떄)
- 만약 게시판 프로그램에서 100명이 회원을 조회하면 100번이나 101번 쿼리가 나가게 되는 문제가 생긴다 → N + 1 문제
- ==**즉, 패치 조인으로 해결하면 된다 !!!!**==

    ```java
    //....생략
    String query = "SELECT m From Member m join fetch m.team";
    List<Member> result = em.createQuery(query, Member.class).getResultList();

    for (Member member : result) {
        System.out.println("member = " + member.getUsername() + ", " + member.getTeam().getName());
    }

    tx.commit(); //트랜잭션 종료
    ```

## 컬렉션 조인 패치

- 컬렉션도 패치조인을 사용하지 않으면 지연로딩이 발생하므로 패치조인을 사용해야 한다.

    ```java
    // .....생략

    String query = "SELECT t FROM Team t join fetch t.members";
    //String query = "SELECT t FROM Team t"; // 패치 조인을 사용하지 않으면 지연로딩 발생
    List<Team> result = em.createQuery(query, Team.class).getResultList();

    for (Team team : result) {
        System.out.println("team.getName() " + team.getName() + "|members = " + team.getMembers().size());
    }
    ```

    ![[Untitled 4 5.png|Untitled 4 5.png]]

- 위와 같이 팀A가 두번 출력되는 이유는 DB입장에서 Team 과 Member를 조인할때 TeamA를 가지는 Member가 2명이 므로 팀A를 두번 반환하게 된다.

    ![[Untitled 5 5.png|Untitled 5 5.png]]

- ==**이때 DISTINCT 로 중복을 제거할 수 있다.**==

## 패치 조인과 DISTINCT

- SQL 의 중복을 제거하는 DISTINCT와 같다.
- JPQL 에서 제공하는 DISTINCT 는 2가지 기능을 제공한다.
    1. SQL에 DISTINCT 추가
    2. ==**애플리케이션단에서 엔티티 중복 제거**==
- SQL에서 DISTINCT는 중복되는 컬럼이 모두 같아야 한다.
- 하지만 컬렉션 패치 조인을 했을때 출력되는 컬럼은 Team 이름만 같을뿐 완벽히 같지않다.
- ==**그래서 JPQL에서 DISTINCT가 추가로 애플리케이션에서 중복 제거를 해준다.**==
- 같은 식별자를 가진 Team 엔티티를 제거하여 출력해준다.

    ```java
    // .......생략
    String query = "SELECT DISTINCT t FROM Team t join fetch t.members";
    List<Team> result = em.createQuery(query, Team.class).getResultList();

    System.out.println("result.size() = " + result.size());
    for (Team team : result) {
        System.out.println("team.getName() " + team.getName() + "|members = " + team.getMembers().size());
        for (Member member : team.getMembers()) {
            System.out.println("-> member = " + member);
        }
    }
    ```

# **페치 조인의 특징과 한계**

---

1. 패치 조인 대상에는 별칭을 줄 수 없다 !!!
    - As 키워드를 사용해서 별칭을 주면 안된다 , 가급적 사용 X, Hibernate는 가능

        `String query = "SELECT t FROM Team t join t.members AS m";`

    - 페치 조인은 연관된 데이터를 모두 끌어오는 것이 목적이다!!
    - 그래프 탐색에서 몇개만 걸러서 데이터를 가져오는것은 좋지 않은 방법.
2. 둘 이상의 컬렉션은 페치 조인 할 수 없다 !
3. 컬렉션을 페치 조인하면 페이징 API를 사용할 수 없다 !

    - ==**Hibernate 에서는 경고로그(Warn)를 남기고 메모리에서 페이징(매우 위험)**==
    - 다대일 관계로 뒤집어서 쿼리를 짜면 해결할 수 있다.
    - 컬렉션 변수에 `@BatchSize` 어노테이션을 이용하여 N+1 문제를 해결할 수 있다.
    - 또는 persistence.xml 파일에 아래와 같은 코드를 적어 글로벌 세팅으로 해결할 수 있다.(1000이하에 값을 적절하게 주면 된다.
        - `<property name="hibernate.default_batch_fetch_size" value="100"/>`


- 연관된 엔티티들을 SQL 한 번으로 조회 → 성능 최적화
- 엔티티에 직접 적용하는 글로벌 로딩 전략보다 우선시 된다.
    - `@OneToMany(fetch=FetchType.LAZY)`-> 글로벌 로딩 전략
- 실무에서는 글로벌 로딩 전략은 모두 지연 로딩
- ==**최적화가 필요한 곳에 페치 조인 적용**==

## 페치 조인 - 정리

- 페치 조인은 객체 그래프를 유지할 때 사용하면 효과적

**==여러 테이블을 조인해서 엔티티가 가진 모양이 아닌 전혀 다른 결과를 내야하면==** 페치 조인보다는 일반 조인을 사용하고 **==필요한 데이터들만 조회해서 DTO로 반환하는 것이 효과적==**

## 기본 키 값 사용

- JPA에서 엔티티를 직접 사용하면 엔티티의 기본 키값을 사용하게 바꿔준다.
    - `select count(m.id) from Member m;`

# Named 쿼리

---

**쿼리를 미리 정의하여 이름을 부여하고 나중에 그 이름으로 쿼리를 재활용 하는 것이다.**

### 정적 쿼리

- 동적 쿼리는 안되고 ==**정적 쿼리만 가능하다.**==
- 어노테이션, XML 에 정의
- 애플리케이션 로딩 시점에 초기화 후 재사용
- ==**애플리케이션 로딩 시점에 쿼리를 검증**==

    ```java
    @NamedQuery(
            name = "Member.findByUsername", // 
            query = "select m from Member m where m.username = :username"
    )

    List<Member> resultList = em.createNamedQuery("Member.findByUsername", Member.class)
                        .setParameter("username", "회원1")
                        .getResultList();
    ```


# 벌크 연산

---

- 예를 들어 재고가 10개 미만인 모든 상품의 가격을 10% 상승하는 작업을 한다면 많은 SQL 쿼리가 나간다.
    1. 재고가 10개 미만인 상품을 리스트로 조회
    2. 상품 엔티티의 가격을 10% 증가
    3. 트랜잭션 커밋 시점에 변경감지가 동작
- 변경된 데이터가 100건이라면 100번의 update sql 실행 → ==**JPA 벌크연산으로 해결가능**==

### 벌크 연산 예제

- 쿼리 한 번으로 여러 엔티티 로우 변경
- `executeUpdate()` 메서드로 한번에 변경하고 ==**반환값은 영향받은 엔티티 수 반환.**==

    ```java
    int resultCount = em.createQuery("update Member m set m.age=20")
              .executeUpdate();
    System.out.println("resultCount = " + resultCount);
    ```


### 벌크 연산 주의

- ==**벌크 연산은 영속성 컨텍스트를 무시하고 데이터베이스에 직접 쿼리를 전달한다**== → 매우 위험함
    - 예를들어 예전에 조회했던 Member가 영속성 컨텍스트에 있는데 벌크연산을 하면서 Member의 age값이 바꼈다. 그럼 DB에는 바뀐 Member엔티티가 있지만 영속성 컨텍스트에는 이전 Member가 존재하므로 데이터가 꼬인다 !!!!
- 해결방법

    1. 벌크 연산을 먼저 실행
    2. ==**벌크 연산 수행 후 영속성 컨텍스트를 초기화 시킨다.**==
