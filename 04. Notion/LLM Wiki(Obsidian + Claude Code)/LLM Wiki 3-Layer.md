### RAG의 한계

- 문서가 많을때 찾을 수 있음
- 단점
    1. 지식의 누적이 안됨 → 매번 모든 데이터를 다시 검색한다
    2. 여러 문서를 종합하는 추론이 약함 → 여러 소스를 종합해서 분석하는 질문에 약함(여러 소스들의 관계를 잘 모름)
    3. **복리 효과가 없음 → 자료를 10개 읽는거랑 자료를 100개 읽는거랑 성능차이가 거의없다.(지식이 늘어날수록 검색범위만 늘어나는거임)**

### LLM Wiki

- RAG는 인터프리터처럼 동작한다 → 질문이 들어올때마다 문서를 읽고 파싱하여 답변한다
- LLM Wiki는 컴파일러처럼 동작한다 → 소스를 미리 분석해서 실행가능한 Wiki를 미리 만들어두고 Wiki안에서 검색한다(속도가 빠름)
    - 소스(자료)가 추가될때마다 증분되서 컴파일된다.
    - 사용자입장에서 질문시점에서는 원본문서(로우데이터)에서 하는게 아닌 LLM이 분석해서 만든 Wiki에서 답변을 해준다.
- 사람이 하는 일은 ‘소스수집’ , ‘탐색’ , ‘올바른 질문’ 이다.
    - 소스수집,탐색은 에이전트로 자동화가 가능하다
    - 하지만 어떤 종류의 소스를 수집할지, 탐색할지는 사람이 정한다.
- Obsidian 은 IDE이고 LLM은 프로그래머 Wiki는 코드베이스로 생각하면 된다.
- LLM Wiki는 개인용, 리서치, 책읽기, 비즈니스 등에 활용된다.

## LLM Wiki 3 Layer 분석

### Layer 1 - Raw Sources(원본소스)

- obsidian 을 기준으로 `raw/` 폴더 안으로 각 목적별로 `articles/ papers/ notes/ ...` 등으로 폴더안에 원본소스가 들어간다
- 핵심은 ==**수정하지 않는다**== 이다.

### Layer 2 - Wiki(위키)

- obsidian 을 기준으로 `wiki/` 폴더 안으로 각 목적별로 `concepts/ java/ kafka/ ...` 등으로 마크다운 파일들이 들어간다.
- ==**사람이 쓰는게 아니고 LLM 이 쓴다.**==
- 하나의 소스(Raw Data)가 Wiki 페이지 다수의 영향을 줄 수 있다.

### Layer 3 - Schema(스키마)

- LLM 에게 이 Wiki가 어떻게 구성이 되어있는지 어떤 규칙을 따르는지 새 소스가 들어오면 어떻게 처리해야하는지 알려주는 설정 파일이다. → Claude Code 로 이용한다(`CLAUDE.md`)
- 폴더의 구조, 파일의 명칭, 처리순서, 위키페이지 표준형식 등등의 설정이 들어간다.
- 스키마 파일은 고정이 아니고 계속 대화를 하면서 점점 수정되고 변경된다.

### 3가지 기본작업

1. `Ingest` : 새 자료(Raw Data)가 생겼을 때 LLM은 raw를 읽고 요약을하고 기존페이지를 업데이트
2. `Query` : Wiki가 어느정도 쌓이면 자유롭게 질문 → 쿼리를 통해 나온 답변은 지식베이스에 저장을 한다.(좋은 답변은 새 페이지로 저장)
3. `Lint` : 주기적으로 수행하는 정비

### 정리

- LLM Wiki는 자료가 들어올때마다 구조화된 지식으로 편찬
- 3 layer 구조 `Raw(원본)` → `Wiki(편찬된 지식)` → `Schema(규칙)`
