## 1. 멀티태스킹과 멀티프로세싱

### 1.1. 단일프로그램 실행

**프로그램 실행** 이란 **==프로그램을 구성하는 코드를 순서대로 CPU 에서 실행==**(연산) 하는 일이다.

- 예를 들어 2개의 프로그램을 실행하고, CPU 에서 연산을 처리하는 코어가 1개라면 다음 그림의 순서대로 연산이 된다.

    ![[image 17.png|image 17.png]]

    ![[image 1 8.png|image 1 8.png]]

    ![[image 2 6.png|image 2 6.png]]

    ![[image 3 5.png|image 3 5.png]]

- 연산을 하는 CPU가 1개이기 때문에 프로그램 A의 모든 연산이 끝난 후, 프로그램 B 의 연산이 시작된다.
- 이는 사용자 입장에서 답답하게 느껴지기 때문에 **==CPU 코어로 여러 프로그램을 동시에 실행==**하는 `멀티 태스킹` 기술이 등장한다.

### 1.2. 멀티 태스킹

- 멀티태스킹을 쉽게 예로 들면 **==순서대로 촬영한 사진을 빠르게 교차해서 보여줄 경우 사람은 이를 영상으로 인지하게 된다.==**
- 위 예시처럼 `멀티 태스킹`은 **==하나의 CPU 코어로 두 프로그램의 코드를 빠르게 번갈아 가면서 수행하는 기술==**이다.

    ![[image 4 5.png|image 4 5.png]]

    ![[image 5 5.png|image 5 5.png]]

    ![[image 6 5.png|image 6 5.png]]

- 위 그림과 같이 **==각 프로그램의 실행 시간을 분할해서 마치 동시에 실행==**되는 것 처럼 하는 기법을 `시분할 기법(Time Sharing) 기법`이라 한다.
    - OS나 실행환경에 따라 다르겠지만 대략 10m/s 단위로 번갈아가면서 실행된다.
- 즉, **==하나의 컴퓨터에서 시스템이 동시에 여러 작업을 수행하는 능력==**을 `멀티태스킹(MultiTasking)`이라 한다.

### 1.3. 스케줄링

- **==CPU 에 어떤 프로그램이 얼마만큼 실행될지는 운영체제가 결정==**하는데 이것을 `스케줄링(Scheduling)`이라 한다.
- 이 때, 단순히 시간으로만 작업하는 시분할 기법이 아닌 CPU를 최대한 활용할 수 있는 다양한 우선순위와 최적화 기법을 사용.

### 1.4. 멀티프로세싱

- **멀티프로세싱(Multi Processing)**은 **==컴퓨터 시스템에서 둘 이상의 프로세서(CPU 코어)를 사용하여 여러 작업을 동시에 처리하는 기술==**을 의미한다.
- 여러 개의 CPU 코어를 활용하여 실제로 동시에 다수의 작업을 병렬 처리할 수 있다.

    ![[image 7 5.png|image 7 5.png]]

    ![[image 8 5.png|image 8 5.png]]

    ![[image 9 5.png|image 9 5.png]]


### 1.5. 멀티 프로세싱 vs 멀티 태스킹

|`멀티 프로세싱(Multi Processing)`|`멀티 태스킹(Multi Tasking)`|
|---|---|
|여러 CPU를 사용하여 동시에 여러 작업 수행|단일 CPU가 여러 작업을 동시에 수행하는 것처럼 보이게 실행|
|하드웨어 기반으로 성능 향상|소프트웨어 기반으로 CPU 시간을 분할하여 각 작업 할당|

## 2. 프로세스와 스레드

### 2.1. 프로세스와 스레드란?

- 쉽게말하면 프로세스는 운영체제(OS) 안에 있고, 스레드는 프로세스안에 있다.

    ![[image 10 5.png|image 10 5.png]]


### 2.2. 프로세스

- 프로그램은 실제로 실행하기 전까지 단순한 파일에 불과하다.
- 하지만 **프로그램을 실행하는 순간 프로세스가 만들어지고 프로그램이 실행**된다.
- 즉, **실행중인 프로그램을 프로세스**라 한다.
- 프로세스는 실행 중인 프로그램의 인스턴스이다.
- Java 로 비유하자면 _클래스는 프로그램, 인스턴스는 프로세스_이다.

`프로세스` 는 **==운영체제 안에서 독립적인 메모리 공간을 갖고 있다==**.

각 프로세스는 별도의 메모리 공간을 갖고 있기 때문에 서로 간섭하지 않는다(==**리소스를 공유하지 않는다.**==)

### **2.3. 프로세스의 메모리 구성**

- `코드 섹션(Code Section)` : 실행할 프로그램의 코드가 저장되는 부분
- `데이터 섹션(Data Section)` : 전역 변수 및 정적 변수가 저장되는 부분
- `힙(Heap)` : 동적으로 할당되는 메모리 영역
- `스택(Stack)` : 메서드(함수) 호출 시 생성되는 지역 변수와 반환 주소가 저장되는 영역(**스레드에 포함**)

### 2.4. 스레드(Thread)

**프로세스(Process)**는 ==**하나 이상의 스레드를 반드시 포함**==한다.

**스레드(Thread)**는 **==프로세스 내에서 실행되는 작업의 단위==**이다. 프로세스가 제공하는 ==**동일한 메모리 공간을 공유**==한다.

- **공유 메모리** : 같은 프로세스의 `코드 섹션`, `데이터 섹션`, `힙`은 프로세스 안의 모든 **스레드가 공유**한다.
- **개별 스택** : 각 **스레드는 자신의 스택**을 갖고있다.

### 2.5. 프로그램이 실행된다는 것은?

- 실은 영어로 Thread 이다. 코드로 비유하면 위에서 아래로 하나씩 꿰면서 내려가는 것이다.

    ```java
    public class Operator {
        public static void main(String[] args){
            int sum1 = 1;
            int sum2 = sum1 + 1;
            System.out.println("sum1 = " + sum1);
            System.out.println("sum2 = " + sum2);
        }
    }
    ```

- 이렇듯 프로세스의 코드를 실행하는 흐름을 `스레드(Thread)`라 한다.

### 2.6. 스레드 종류

- `단일 스레드` : 한 프로세스 내의 하나의 스레드만 존재
- `멀티 스레드` : 한 프로세스 내에 여러 스레드가 존재

## 3. 스레드와 스케줄링

### 3.1. 단일 코어 스케줄

- **운영체제(OS)는 내부에 스케줄링 큐**를 가지고 있고, 각각의 **==스레드는 스케줄링 큐에서 대기==**한다.
- 단일코어기 때문에 CPU 코어 1개와 스레드 3개가 있다고 가정하자.
- 스케줄링 큐(Queue)에는 스레드A1, 스레드B1, 스레드B2가 대기한다.

    ![[image 11 4.png|image 11 4.png]]

- CPU 코어는 스레드A1을 꺼내고 실행한다. 이때, 스레드A1 의 프로그램 코드를 수행하고 CPU 를 통한 연산도 일어난다.

    ![[image 12 3.png|image 12 3.png]]

- 그리고 운영체제는 스레드A1을 잠시 멈추고 큐 가장 끝으로 넣은 후, 스레드 B1을 실행한다.

    ![[image 13 3.png|image 13 3.png]]

- 위 과정들을 반복하면서 스레드를 실행한다.

### 3.2. 멀티 코어 스케줄링

- CPU 코어가 2개 이상이라면 한 번에 많은 스레드를 물리적으로 실행하게 된다.
- 스케줄링 큐에 스레드B2, 스레드B1, 스레드A1이 대기하고 있고, CPU 코어가 2개라고 가정하자.

    ![[image 14 3.png|image 14 3.png]]

    ![[image 15 2.png|image 15 2.png]]

    ![[image 16 2.png|image 16 2.png]]

    ![[image 17 2.png|image 17 2.png]]

- 각 CPU 코어에 스레드를 각각 넣고 한 스레드를 실행중 잠시 정지시키고 스케줄링 큐에 남아있는 스레드와 교환해서 다른 스레드를 실행한다.
- 위 과정을 반복한다.

### 3.3. 스케줄링(Scheduling) 이란?

- **CPU에 어떤 프로그램이 얼마만큼 실행될지는 운영체제가 결정**하는데 이것을 `스케줄링(Scheduling)`이라 한다.
- 시간으로 작업을 분할하는게 아닌, CPU를 최대한 활용할 수 있는 다양한 우선순위와 최적화 기법을 사용한다.

## 4. 컨텍스트 스위칭

**==멀티태스킹이 반드시 효율적인 것은 아니다 !==**

### 4.1. 컴퓨터의 멀티 태스킹

- 우선 예시로 CPU 코어가 1개 있다고 가정한다.
- 스레드 A 를 실행하고 있다가 잠시 멈추고 스레드 B를 실행한다. 그리고 스레드 B의 실행이 끝나면 다시 스레드 A로 돌아가야 하는데, 이 때 ==**스레드 A의 코드가 어디까지 수행되었는지 위치를 찾아야 한다.**==
- 그리고 계산하던 변수들의 값을 CPU에 다시 불러들여서 스레드 A를 마저 실행해야 한다.
- **==스레드 A를 멈추는 시점에 CPU에 사용하던 값들을 메모리에 저장해두어야 한다. 그리고 스레드 A를 다시 실행할 때 이 값들을 CPU에 다시 불러온다.==**
- 위와 같은 과정을 `컨텍스트 스위칭(Context Switching)` 이라 한다.
- ==**멀티스레드는 대부분 효율적이지만 컨텍스트 스위치 과정이 필요하므로 항상 효율적인 것은 아니다 !**==

    - 컨텍스트 스위칭 과정에는 약간의 비용이 발생하기 때문이다.
    - 예를 들어 1 ~ 10000 까지 더해야 한다고 가정해보자.
        - `스레드 1` : 1 ~ 5000 까지 더함
        - `스레드 2` : 50001 ~ 10000 까지 더함
        - 마지막에 스레드 1 결과와 스레드 2 결과를 더함.
    - ==**CPU 코어가 2개**==라면 멀티스레드로 병렬 처리하므로 연산을 2배 빠르게 처리할 수 있다.
    - ==**CPU 코어가 1개**==라면 `스레드 1` 을 1 ~ 1000정도까지 연산하고 잠시 멈추고, `스레드 2` 를 5001 ~ 6001 까지 연산하면서 반복한다.
        - 위 경우에는 각 연산을 멈출 때 마다 어디까지 연산헀는지 기억해야하고, 컨텍스트 스위칭 비용까지 생기게 된다. → ==**연산시간** + **컨텍스트 스위칭 시간**==

    위 경우때문에 멀티스레드가 반드시 효율적인것은 아니다 !


### 4.2. CPU 바운드 작업 VS I/O 바운드 작업

- `CPU 바운드 작업(CPU bound tasks)`
    - **==CPU 의 연산 능력을 많이 요구하는 작업==**을 의미한다.
    - 예시: 복잡한 수학 연산, 데이터 분석, 비디오 인코딩, 과학적 시뮬레이션 등등
- `I/O 바운드 작업(I/O bound tasks)`
    - **==디스크, 네트워크, 파일 시스템 등과 같은 입출력(I/O)작업을 많이 요구하는 작업==**을 의미한다.
    - 이러한 작업은 I/O 작업이 완료될 때까지 대기 시간이 많이 발생하며, `유휴(대기) 상태`에 있는 경우가 많다.
    - 예시: DB 쿼리처리, 파일 읽기/쓰기, 네트워크 통신, 사용자 입력 처리 등

### 4.3. 웹 어플리케이션 서버

_==분야마다 다르지만, 실무에서는 CPU 바운드 작업보다 I/O 바운드 작업이 많다.==_

- 예를 들어 백엔드 개발자의 경우 주로 웹 어플리케이션 서버를 개발하는데, 스레드가 1 ~ 10000까지 더하는 CPU의 연산이 필요한 작업보다는 **==사용자의 입력을 기다리거나 DB 호출을 하는 등의 작업이 더 많다.==**
- 즉, 스레드가 CPU 를 많이 사용하지 않는 I/O 바운드 작업이 많다는 뜻이다.
- **==일반적인 Java 웹 애플리케이션 서버의 경우 사용자의 요청 하나를 처리하는데 스레드 1개가 필요==**하다.
- 그래서 사용자의 요청을 하나 처리하는데 스레드는 CPU 를 1% 정도 사용하고, 대부분 DB 에 쿼리를 날리고 대기한다거나 파일 입출력 등을 하기 때문에 CPU를 사용하지 않고 대기한다.
- 이 경우 CPU 코어가 4개 있다고 해서 스레드 숫자고 CPU 코어에 맞춰 4개로 설정하면 안된다. 이 때 4명의 동시 사용자 요청만 처리하고 CPU 는 4% 정도 밖에 사용하지 못한다.
- 즉, ==**실무에서는 성능 테스트를 통해서 최적의 스레드 숫자를 찾는 것이 이상적**==이다.

### 4.4. 이상적인 스레드 갯수 설정

- `CPU 바운드 작업` : **CPU 코어 수 + 1개**
    - CPU 를 거의 100% 사용하는 작업이므로 **==스레드를 CPU 숫자에 최적화==**
- `I/O 바운드 작업` : **CPU 코어 수 보다 많은 스레드를 생성**, CPU를 최대한 사용할 수 있는 숫자까지 스레드 생성
    - **==단, 너무 많은 스레드를 생성하면 컨텍스트 스위칭 비용도 함께 증가==**
